# AIエージェントの業務適用を見据えた生成AIアプリケーション層の検討

---

## 1. 目的

生成AIアプリケーションの構成を **Application層 / AI Assurance層 / Tool層** で分けて考える。
ここでは、特に **Application層** における実装パターン（WF型 / 自律型 / SV型）と、その構成要素（機能ユニット）・制御方式を整理する。

---

## 2. Application層の基本整理：WF型 / SV型 / 自律型 

生成AIを業務に適用する際の代表的な実装パターンとして、以下の3つを置く。これらは「誰が次に何をすべきかを判断するか（統制の主体）」によって区別される。

* **WF型（ワークフロー型）**：定型〜準定型の処理を、あらかじめ定められた手順（プログラム）の連鎖で実現する。
* **SV型（スーパーバイザー型）**：非定型・高リスク領域において、AIによる統制（Supervisor）と、人間による最終監督（Human Supervisor）を前提に、複数の専門ユニットの合議で実現する。
* **自律型（エージェント型）**：目標を与えられ、AI自身が「思考→行動→観察」のループを回して、利用するツールや手順を動的に決定する。

実装・運用の観点では、次のように **実現手段（OSS/基盤）を割り当てる** と整理しやすい。

* **WF型の実装基盤**：Dify（Workflow/Chatflow、GUI、運用UI、Human Input 等）
* **SV型の実装基盤**：LangGraph（StateGraph、ループ/合議、Checkpointer、interrupt 等）
* **自律型の実装基板**: Claude Codeなどのコーディングエージェント(Dockerコンテナ 等)

## 2.1 設計の鉄則：予見性に基づくモデル選択

アーキテクチャを選択する際は、タスクの「予見性（事前にどこまで分解できるか）」を基準に、左側から順に検討する。**不必要に動的なモデルを採用しないこと**が、コスト・精度・制御性の面で重要である。

| 予見性のレベル | 推奨モデル | 制御のイメージ | 決定のタイミング |
| --- | --- | --- | --- |
| **完全にタスク分解可能** | **WF型** | 鉄道（レール走行） | **設計時**（人間が決定） |
| **ある程度の実行計画あり** | **SV型** | 指揮者と奏者 | **実行時**（SVが決定） |
| **計画不能（試行錯誤が必要）** | **自律型** | ジャズのセッション | **実行時**（現場の合議） |

---

## 3. WF型（ワークフロー型）：機能ユニットの連鎖で組む

WF型は「入力（非構造）→処理→出力（構造）」を、比較的固定したステップでつなぐ。
最小単位はチャットUIではなく、**1つの具体的タスクを処理するプロンプト（＝機能ユニット）とデータの流れ** と捉える。

> 実装例：DifyのWorkflowで、抽出/分類/変換/判定などのノードを繋いで表現しやすい。

### 3.1 機能ユニット（例）

* **抽出ユニット**：メールやPDFから、必要な項目だけを抜き出す
* **分類ユニット**：問い合わせをカテゴリ分けする（例：クレーム/見積依頼）
* **変換ユニット**：文章やメモを、後続システムが扱える形式（JSON/CSV/テンプレ文書）に整形する（例：Text-to-SQL）
* **判定ユニット**：規約・社内ルールに照らして Yes/No 判定する（次工程の分岐条件になる）

---

## 4. SV型（スーパーバイザー型）：役割を持つユニットの合議と人間の監督で組む

SV型は「正解が一つではない」「責任の所在が重い」領域において、自律型の「予測不能さ」という弱点を克服し、安全に業務適用するために用いられる。

ここで言う「SV（スーパーバイザー）」には、**AIと人間の2つのレイヤー**が存在する。

1. **AIのSupervisor（統制役）**：各専門AIワーカーの働きを監視し、品質が基準に達するまでやり直し（ループ）を命じる。
2. **人間のSupervisor（最終責任者）**：AIが起案（下書き）した結果に対して、最終的な意思決定を下す。

**SV型の設計指針：**
AIエージェントの一般的な用語として「スーパーバイザー」はAIの司令塔を指すが、業務適用においては**「SV型アーキテクチャを採用する場合は、必ず人間が結果を判断・承認するプロセス（非同期HITL）を組み込むべき」**という原則を置く。

### 4.1 構成ユニット（例）

* **統合ユニット（司令塔）**：計画立案、各ユニットの呼び出し、最終まとめ
* **評価ユニット（監査役 / クリティック）**：案の品質・リスク・抜け漏れを指摘
* **業務ユニット（実行役）**：特定ドメインの専門家として調査・分析・起案する（例：リサーチ、法務、企画）

---

## 5. 自律型（エージェント型）：AI自身の思考ループで組む

WF型のように手順が固定できず、タスク解決のための道筋が探索的になる領域に適用する。
最小単位は「目標」と「利用可能なツール群」であり、AIが自身で計画と実行を繰り返す。

> 実装例：LangChain/LangGraphにおけるReAct（Reason + Act）パターンや、Claude Codeなどのコーディングエージェントなど。

**特徴と得意領域（探索的タスク）**

* **データ分析やリサーチ業務に最適**：例えば「データから傾向を分析する」といったタスクは、AIが「コードを書く→エラーを見る→修正する」という試行錯誤を自律的に行う必要があるため、この型が最も能力を発揮する。
* 未知のエラーや情報不足に対しても、AIが自力でリカバリー（再検索や再計算）を試みる柔軟性を持つ。

**業務適用におけるリスク管理と「段階的自律化」の指針**

* 人間から見て「今なぜそのツールを使っているのか」「いつ終わるのか」が予測しづらいという特性があるため、**データの読み取り（Read-Only）や、隔離された環境（サンドボックス）でのコード実行**にとどめる場合は単独でも非常に強力である。
* 一方で、本番DBの更新や顧客への直接送信など「外界への不可逆な操作（Write/Send）」を伴うタスクにおいては、**「段階的自律化（Progressive Autonomy）」**のアプローチをとるべきである。

1. **導入期（検証〜安定稼働まで）**：自律型を単独で暴走させず、前述の「SV型の配下（ワーカー）」として組み込み、必ず人間の承認（Human-in-the-loop）を挟んで安全性を担保する。
2. **成熟期（安定稼働の確認後）**：稼働実績が積まれ、予期せぬ挙動（暴走）のリスクが排除できた段階で、人間の介入を「例外検知時のみ」に限定したり、限定されたスコープ内での**単独稼働（完全自律化）への移行**を視野に入れる。

## 6. 制御パターン：オーケストレーション型 / 階層型 / Hand-off型

SV型や自律型において、複数のユニット（エージェント）をどう連携させるかには代表的な3パターンがある。

### 6.1 オーケストレーション型（集中管理型）

* **統合ユニット（SV）** が計画を立て、個々のユニットを順次呼び出し、結果を評価する。
* 透明性が高く、人間が途中介入しやすい（重要業務・高い説明責任に向く）。

### 6.2 階層型（Hierarchical）

* 親ユニットが、必要に応じて子ユニット（チーム）を呼び出し、複雑なタスクを分割する。
* 自律性が高く、複雑タスクをスケールさせやすい一方、内部のブラックボックス化を防ぐState設計（判断理由の引き上げ）が重要。

### 6.3 Hand-off型（自律分散・リレー型）

* 中央の統制ユニット（SV）を持たず、各業務ユニットが自身の判断で、次に必要な専門ユニットへ処理（State）を直接引き継ぐパターン。
* 柔軟性が高くスピーディだが、処理がどこで迷子になっているかの追跡が難しく、人間の介入（HITL）を挟むポイントが分散しやすいため、高リスク業務での採用は慎重に行う必要がある。

---

## 7. ハイブリッド型（WF×SV）：静的フロー管理と動的オーケストレーションの組み合わせ

実務では「外側をWF、内側をSV」のように組み合わせた **ハイブリッド型** が実装・運用ともに扱いやすい。

* **外側（業務フロー/運用/UI）＝WF型（例：Dify）**
  * 受付 → 前処理 → SV呼び出し → 承認（非同期HITL）→ 反映、などの業務手順を固定フローで可視化。


* **内側（難所の推論/合議/品質改善）＝SV型（例：LangGraph）**
  * Supervisorが状況に応じてワーカーを選び直すなどの動的制御。



---

## 8. 代表OSSでの実現例（静的フロー / 動的オーケストレーション）

#### 8.1 DifyでのWF型（GUIベースの静的フロー）例

* **典型構成**: 入力ノード → 情報抽出ノード → 分類ノード → テンプレート生成ノード → 出力
* **特徴**: 分岐は明示的でルーティングが固定。非エンジニアでもメンテナンスしやすく、プロンプトの調整や非同期HITL（Human Input）の組み込みがGUI上で容易に行える。

#### 8.2 LangGraphでのWF型（コードベースの静的フロー）例

* **典型構成**: Node A（前処理）→ Node B（API実行）→ Node C（DB保存）
* **特徴**: LLMによる動的ルーティング（`Conditional Edge`）を使わず、固定の経路（`Normal Edge`）でノードを接続する。DifyのようなGUIツールでは表現しきれない複雑なデータ変換、厳密な型定義（Pydantic等）、既存システムとの深い統合、またはGitによるバージョン管理（CI/CD）が必須な業務システムに適用する。

#### 8.3 LangGraphでのSV型（動的オーケストレーション）例

* **典型構成**: Supervisor → Worker（調査）→ Critic（評価）→ Supervisor（合格なら完了/不合格ならループ）
* **特徴**: Supervisorが実行状況に応じて次工程を動的に決定する。中断/再開が必要なら `interrupt` で「状態を持った停止」が可能。
* **拡張性（階層化）**: タスクが巨大な場合は、特定のWorkerを「子Supervisor」とした **Subgraph（階層型）** 構成をとることができる。親SVからは一人の専門家に見えるが、内部ではさらに専門チームが合議する、といったスケールが可能。

#### 8.4 LangGraphでの自律型（合議・Swarm）例

* **典型構成**: Agent A ↔ Agent B ↔ Agent C（共通Stateの共有）
* **特徴**: 特定の固定司令塔（SV）を持たず、現在のStateを見て「次は誰が動くべきか」をエージェント間で自律的にハンドオフ（遷移）する。明確な手順が存在せず、試行錯誤が必要な探索的タスクに向く。

#### 8.5 Claude Code / Codex / Gemini CLI / Cline CLI（高度自律型プログラミングエージェント）等での自律型例

* **典型構成**: SV (目標設定/レビュー) → Coding Agent (自律思考・実行ループ) in Sandbox
* **特徴**:
  * 開発者がロジックを組むReActエージェントとは異なり、ツール側が「ファイル操作、ターミナル実行、テスト、デバッグ」までを自律的に完結させる。
  * **Native MCP Support**: Claude Code等はMCP（Model Context Protocol）を標準サポートしており、我々が定義したTool層の資産（社内DBやAPI）を即座に「道具」として取り込むことが可能。


* **エンタープライズ適用のための防衛線（ガードレール）**:
  * **Tool層での隔離（Sandbox）**: 物理環境を直接操作させず、Dockerコンテナ等の隔離環境内でのみ実行権限を許可する。
  * **AI Assurance層でのコマンド監査**: 実行されるシェルコマンドやファイル変更をリアルタイムで監視し、破壊的操作や機密データの外部流出をネットワークレベルでブロックする。
  * **非同期HITLによる確定操作**: AIが提案した修正（Diff）をApplication層のCheckpointerでフリーズ。人間が承認した後にのみ、本番環境への反映（Git Push等）を許可する。

## 9. 非同期処理の必要性と実装設計

AIエージェント（特に自律型・SV型）を既存のシステムやUI（Dify等）と連携させる際、最大の技術的ハードルとなるのが「処理時間の不確実性」である。これを解決し、システムを堅牢に保つためには、同期的な通信（APIの応答待ち）ではなく、**非同期処理**を前提としたアーキテクチャ設計が必須となる。

### 9.1 エージェント処理の長時間化に伴うタイムアウト対策（ジョブ化とWebhook）

* **課題（タイムアウトのリスク）**:
エージェントによる探索的タスク（ログ解析、反復的なWeb検索など）や、SV配下での合議プロセスは、数分から数十分かかることがある。これを同期的（HTTPリクエストを繋いだままレスポンスを待つ形）に実行すると、呼び出し元（DifyノードやAPIゲートウェイ）でタイムアウトエラーが発生し、システム全体がクラッシュしてしまう。
* **解決策（非同期ジョブパターンの適用）**:
1. **受付即答**: 上位システム（Dify等）からLangGraphへ処理を依頼した際、LangGraph側のAPIは処理の完了を待たず、即座に「受付完了（Thread IDやJob ID）」を返す（HTTP 200 OK）。
2. **バックグラウンド実行**: LangGraphは裏側でエージェントループ（思考と行動）を安全に実行する。
3. **イベント通知による再開（Webhook）**: 処理が完了したタイミングで、LangGraph側から上位システム（DifyのWebhook URL等）へ結果をプッシュ通知し、停止していた上位のワークフローを再開させる。



### 9.2 非同期HITL（Human-in-the-Loop）による状態の永続化

* **課題（人間の反応遅延）**:
高リスク業務における承認プロセスでは、人間が「結果を確認して承認ボタンを押す」までに数時間〜数日かかる場合がある。この間、プロセスをメモリ上で待機させることはリソースの枯渇を招き、サーバーの再起動等でAIの推論文脈（State）が消失するリスクがある。
* **解決策（Checkpointerとinterruptによるフリーズと復元）**:
ハイブリッド型（外側WF・内側SV）では、「State（状態）の主導権と永続化」をLangGraph側に寄せる設計とする。
1. **状態のフリーズ**: 人間の承認が必要な操作（対外送信やシステム更新）の直前で、LangGraphの `interrupt` 機能を発火させる。これにより、**Checkpointer（DB）に現在のメモリ・文脈（State）が丸ごと保存（フリーズ）され、プロセスは安全に終了（待機状態へ移行）**する。
2. **UIでの待機**: WF側（Dify）はその `thread_id` を運搬して、UI上で承認待ち（Human Input）のステータスを表現する（状態の二重管理を避ける）。
3. **状態の復元と再開**: 人間がUI上で承認アクションを行うと、保持していた `thread_id` をキーにLangGraphのStateがデータベースから完全に復元され、安全に後続の処理（更新・送信等の確定操作）が再開される。

## 10. Tool層（MCP等）との関係：リソースアクセスは「手足」

WF型でもSV型でも、DB/API/ファイル等のリソースアクセスは外界に作用する **ツール（手足）** として整理する。
MCP（Model Context Protocol）等をインターフェースとして利用し、AI（Application層）に **どこまでの権限を渡すか**、影響の大きい操作には **人の承認（HITL）** を挟むなどのガードレールを置く。

---

## 11. 評価の二段構え：業務固有（アプリ内）と汎用基盤（AI Assurance層）

生成AIを大規模な組織でスケールさせるために、評価・保証を以下の2段に分ける。

* **AIエージェントによる評価ユニット（業務固有 / Application層内）**：業務の「プロとしての質」を担保する（例：SV型におけるCritic役として、契約ポリシーに沿っているかなどを自律的に評価）。
* **AI Assurance層（汎用・共通基盤 / 独立層）**：企業としての「守り」を担保する（例：PII検知、ハルシネーション抑制、インジェクション防御等のシステム的な統制）。

---

## 12. 責任分界と結論

**責任分界**

* **開発部門**：業務ユニット・評価ユニットの精度（＝業務品質）
* **IT・ガバナンス部門**：AI Assurance層の堅牢性（＝全社安全）
* **ユーザー（人間のSV）**：最終出力を確認し、業務上の最終決定に対する責任

**結論**

* Application層はWF型 / 自律型 / SV型を中心に捉え、適材適所で制御パターン（オーケストレーション/階層/Hand-off）を選択する。
* 高リスク業務には非同期HITLを前提としたハイブリッド型（外側WF・内側SV）が有効である。
