# AIエージェントの業務適用を見据えた生成AIツール層の検討

---

## 1. 目的と定義：AIの「手足」をステートレスに標準化する

生成AIアプリケーションをエンタープライズの業務で安定稼働させる鍵は、不確実な「思考（Application層）」と、確実な「実行（Tool層）」をアーキテクチャレベルで完全に分離することにある。

* **目的**: Application層（WF/SV/自律型エージェント）が外界のシステムとやり取りするための、高信頼・標準化された「業務実行インターフェース（MCPサーバー等）」を定義・整備する。
* **実態**: 社内データ（ファイルサーバー、BOX、DB、Wiki）やSaaS（Zendesk等）にアクセスするための個別のAPI群を、AIが解釈・実行可能な形（Model Context Protocol等）でカプセル化したものの集合体である。
* **大原則（ステートレス）**: Tool層は状態（State）を持たず、呼び出されたら「即座に」結果、または受付票（Job ID）を返す。人間を待つなどの状態管理（永続化）は絶対に行わない。

---

## 2. なぜ「個別のAPI直接連携」ではなく「Tool（MCP）層」なのか？

Tool層をApplication層から独立させる主な狙いは、アプリやAIモデルの変更に耐えうる **“業務実行インターフェースの資産化”** にある。個別のプロンプト内にAPIの叩き方を直書きするアプローチからの脱却を図る。

* **再利用性（Reusability）**: 複数業務のAIエージェントから、同じツール（MCPサーバー）を共通の道具として使い回すことができる。
* **置換容易性（Portability）**: LLMのモデル（GPT, Claude, Gemini等）を将来入れ替えても、標準プロトコルに準拠していればツール接続の仕様や実装は維持できる（アプリ改修の極小化）。
* **ガバナンスの一元化**: ツールごとの権限・監査・制約を共通のプロトコルで管理できる。

---

## 3. Tool層の設計原則：権限・意味・非同期・可逆性

Tool層を「AIに持たせる道具箱」として設計する際、実務でボトルネックとなる「権限の壁」「意味の壁」「時間の壁」を突破するための鉄則を定める。

### 3.1 セキュリティと権限の「プロキシ（代理）」管理（最重要）

「だれがどのデータにアクセス可能か？」の制御は、単なる固定のAPIキー（サービスアカウント）ではなく、**“実行ユーザーの権限コンテキスト”** で行うことを原則とする。

* **User Context Forwarding**: アプリケーションを利用しているユーザーの認証情報（OAuthトークン等）をMCP経由でバックエンドシステムまで貫通（引き継ぎ）させる。
* **越権アクセスの防止**: ツール側で「このユーザーがアクセス可能なデータのみ」をフィルタリングしてAIに返すことで、役員専用情報などの機密データ漏洩をアーキテクチャレベルで防ぐ。

### 3.2 統合認証基盤とデータ基盤・ゲートウェイの積極的採用（実装方針）

Tool（MCP）サーバー内に個別の権限ロジックやアクセス制御を独自実装することは、開発・運用コストの肥大化とセキュリティホールの原因となる。また、**LLMに生データ（物理テーブル等）を直接触らせると、業務コンテキストの欠如によるハルシネーション（誤集計・誤解釈）が高確率で発生する。**
これらを防ぐため、**「権限判定、データマスキング、およびデータの意味付け（セマンティックレイヤー）の提供は、バックエンドのデータ基盤やゲートウェイ製品に委譲する」** ことをアーキテクチャの基本方針とする。
MCPは、受け取ったユーザートークンをそのままバックエンドに横流しする「On-Behalf-Of（代理実行）フロー」の中継役に徹する。

* **統合認証基盤（IdP）の連携**: Entra IDやOkta等を利用し、システム全体でトークンベースの権限継承を標準化する。
* **セマンティックレイヤーを提供するデータ基盤・ゲートウェイの採用（構造化データ）**:
* **【重要】** 単なるデータ統合・権限管理（行/列マスキング）だけでなく、物理DBの複雑な構造を隠蔽し、AIが理解しやすいビジネス用語（指標やエンティティ）で定義された**「セマンティックレイヤー（意味的データモデル）」の構築を必須とする。** AIには原則としてこのセマンティックビューのみを公開し、生テーブルへの直接アクセスは禁止する。
* これを実現するため、社内の既存資産に合わせて以下の技術要素を適材適所でバックエンドに配置し、Tool層（MCP）はそれらを呼び出す構成とする。
* **モダンデータスタックにおけるセマンティック層**（dbt Semantic Layer, LookML 等）
* **データ仮想化 / フェデレーション**（Denodo, Starburst 等）
* **データグラフ / GraphQL**（Hasura, Apollo 等）




* **カスタムインストラクションによる意味付けの補完（API・非構造化データ）**:
* ゲートウェイ側でセマンティックモデルの直接的な構築が困難なSaaS APIやレガシーシステムについては、**データの意味を提供するカスタムインストラクション（MCPのプロンプトやDescription定義など）で補完する**ことで、セマンティックレイヤーの役割を代替する。
* ファイルサーバー等の非構造化データ（RAG用途）については、検索ゲートウェイ（Azure AI Search等）において、タグやビジネスコンテキストなどのメタデータ付与を強く推奨する。


* **ゼロトラストプロキシの活用（通信とネットワークの保護）**:
* Zscaler（ZPA/ZIA）などのZTNA/SWGソリューションをTool層の通信経路に組み込む。
* **対内通信**: MCPから社内システムへのアクセスは、アイデンティティベースで認可された通信のみを許可し、ネットワークレベルでの越権アクセスを防ぐ。
* **対外通信**: MCPから外部WebやSaaSへの通信経路において、DLP（情報漏洩防止）や不正URLへのアクセス遮断機能を効かせ、意図しないデータ持ち出しをインフラ層でブロックする。



### 3.3 タイムアウトを防ぐ「非同期ジョブ」設計

AIが指示した処理（重いログ解析、巨大なファイル検索等）が数分以上かかる場合、同期通信ではApplication層との通信がタイムアウトし、システムがクラッシュする。これを防ぐため、非同期を前提としたインターフェースとする。

* **受付の即答**: 時間のかかるツールは処理の完了を待たず、即座に「Job ID」のみをAIに返す（ステートレス化）。
* **状態の分離**: AI（Application層）はそのJob IDをもとに一時休眠し、Tool層は裏側で処理を完遂させる。

### 3.4 書き込みアクションの「可逆性」と「承認前提の設計」

実業務では「DB更新」「チケット起票」等の **Write（不可逆操作）** が必ず発生する。

* **Read/Writeの厳格な分離**: 参照専用ツール（Read）と更新用ツール（Write）を明確に分ける。
* **Dry Run（事前検証）**: Write系ツールにおいて、実際の更新前に「何が起きるか（差分）」をシミュレーションしてAI（および人間）に提示する機能を持たせる。
* **承認トークンの要求**: 更新用ツールはAIからの単独実行を拒否し、「人間の承認（HITL）とAI Assurance層のチェックを通過したリクエスト」のみを受け付ける仕様とする。

## 4. Tool層の4大機能要件（接続・意味・統制・安全）

Tool（MCP）サーバーおよびその後方に配置するゲートウェイを実装・選定する際に満たすべき機能要件を以下に整理する。

| 分類 | 機能要件 | 概要 |
| --- | --- | --- |
| **接続**<br>

<br>(Connectivity) | **プラグイン化** | MCPによるDB/SaaS/社内APIの統一的な接続インターフェース。 |
|  | **ハイブリッドアクセス** | 非構造化データ（RAG）と構造化データ（SQL/API）への統一アクセス。 |
| **意味**<br>

<br>(Semantics) | **データ抽象化（ビュー）** | 物理的なテーブル構造を隠蔽し、AIが理解できるビジネス用語のセマンティックモデルとしてデータを提供する機能（dbt、データ仮想化等のバックエンド層と連携）。 |
|  | **文脈の提供** | APIの仕様や制約に対し、AI向けの詳細なカスタムインストラクション（メタデータ/Description）を付与・提供できる機能。 |
| **統制**<br>

<br>(Governance) | **Identity-aware Access** | 前述の通り、実行ユーザーの権限をバックエンドまで継承する認可制御。 |
|  | **実行監査ログ出力** | 「どのユーザーの代理で、どのAIが、何のデータを操作したか」をログ基盤へ出力。 |
| **安全**<br>

<br>(Safety) | **Dry Run機能** | 実行前に更新差分を返すシミュレーション機能。 |
|  | **局所的Rate Limiting** | 特定のバックエンドシステムに対するAIからの過剰なリクエスト（DoS状態）をツール側で防ぐ自己防衛。 |

## 5. 他層（AI Assurance層・Application層）との責務分界

アーキテクチャ全体の一貫性を保つため、「Tool層（実行部）」、「AI Assurance層（関所）」、「Application層（脳）」の明確な分離方針を提示する。

### 5.1 分離の基本方針

* **Tool層（手足）**: システムへの「物理的な接続と実行」に責任を持つ。ユーザー権限に基づく確実なデータ取得（Read）と、Dry Run等の安全な実行機構（Write）を提供する。
* **Application層（脳）**: ツールの選定と実行タイミングの決定を行う。**「人間の承認（HITL）を待つ」という状態の永続化はこの層のCheckpointerが担う。**
* **AI Assurance層（免疫）**: アプリ横断でツール実行の通信をインターセプトし、意味論・リスクベースでの判定、ハルシネーション検知、最終的なブロック（同期ゲート）を担う。

### 5.2 機能配置の整理と判断基準

実務上、データ製品ベンダーが提供するAPI側に十分な統制機能が備わっていないことも多い。その場合の現実的な割り切りを含め、以下のように責務を分界する。

| 検討機能 | 配置先 | 理由・設計思想 |
| --- | --- | --- |
| **ユーザー権限継承（AuthZ）** | **Tool層** | 接続先システム固有の権限モデルに依存するため、ツール側（MCPサーバー内）で透過的に処理し、バックエンドのゲートウェイ等と連携して確実な権限制御を行うべき。 |
| **セマンティックレイヤー** | **データ基盤 / ゲートウェイ** | データの意味付けと抽象化はDWHやデータ仮想化などのバックエンド層で集中的に管理する（困難な場合はTool層のカスタムインストラクションで補完）。 |
| **Dry Run / 可逆性実装** | **Tool層** | 対象システムごとに「どう戻すか」「どう差分を出すか」の実装が異なるため、ツール内部でカプセル化する。 |
| **監査ログ（事実の記録）** | **両方** | Tool層は「APIを叩いた確定的な事実」を記録し、AI Assurance層は「AIの思考から実行までのE2Eの文脈（なぜ叩いたか）」を相関IDで記録する。 |
| **HITL（人間の承認待機）** | **Application層** | **【重要】Tool層はステートレスであるべき。** 人間を待つ（状態の永続化）のはLangGraph等の役割であり、Toolは承認済みの結果を受け取って即時実行するのみ。 |
| **Rate Limit / 異常遮断** | **AI Assurance層** | エージェントの暴走検知や全社的なAPIコールの閾値管理は、AI Assurance層で横断的に行うのが理想。（※ツール側やZTNA等にも局所的な防衛線を設ける） |

**結論**:
“ツール単体の健全な実行と権限管理” は **Tool層** （および後方のゲートウェイ群）で担保し、“エージェントの振る舞い統制と承認フロー（HITL）” は **Application層 ＋ AI Assurance層** で担保する。この責務分界により、タイムアウトに強く、拡張性の高い堅牢なシステムが実現する。
