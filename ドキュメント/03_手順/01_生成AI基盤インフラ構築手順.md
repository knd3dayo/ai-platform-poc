## ステップ1：統合監視・プロンプトCMS（Langfuse）の構築

公式リポジトリをクローンして構築します。これがネットワークの起点となります。

### 1-1. リポジトリのクローンと環境設定

```bash
# 公式リポジトリを 1_langfuse としてクローン
git clone https://github.com/langfuse/langfuse.git 1_langfuse
cd 1_langfuse
```

### 1-2. `docker-compose.yml` の編集（ネットワーク設定）

Langfuseの `docker-compose.yml` をエディタで開き、末尾にネットワーク定義を追加（または修正）して、**全サービスが共通ネットワークに参加**するようにします。

```yaml
# 1_langfuse/docker-compose.yml の末尾に追加

networks:
  default:
    name: ai_platform_net
    driver: bridge

```

> [!NOTE]
> これにより、Langfuse内の `langfuse-web` や `postgres` が自動的に `ai_platform_net` に所属し、後続のLiteLLMから名前解決が可能になります。

### 1-3. 起動とAPIキーの取得

1. `docker compose up -d` を実行します。
2. `http://localhost:3000` にアクセスし、新規アカウント作成（初回ユーザーが管理者になります）。
3. プロジェクト設定から **Public Key** と **Secret Key** を生成し、メモします。

## ステップ2：AI Assurance層（LiteLLM Proxy）の構築

### 2-1. プロジェクト初期化

```bash
cd ..
mkdir 2_litellm
cd 2_litellm
uv sync

```

### 2-2. 設定ファイルの作成

**`2_litellm/.env`**

```env
LITELLM_MASTER_KEY=sk-poc-master-key-12345
OPENAI_API_KEY=sk-your-openai-api-key
LANGFUSE_PUBLIC_KEY=pk-lf-... # メモしたキー
LANGFUSE_SECRET_KEY=sk-lf-... # メモしたキー

```

**`2_litellm/docker-compose.yml`**
公式Langfuse構成に合わせ、`LANGFUSE_HOST` はサービス名である **`langfuse-web`** を指定します。

```yaml
services:
  litellm:
    image: ghcr.io/berriai/litellm:main-latest
    environment:
      - LITELLM_MASTER_KEY=${LITELLM_MASTER_KEY}
      - OPENAI_API_KEY=${OPENAI_API_KEY}
      - LANGFUSE_PUBLIC_KEY=${LANGFUSE_PUBLIC_KEY}
      - LANGFUSE_SECRET_KEY=${LANGFUSE_SECRET_KEY}
      - LANGFUSE_HOST=http://langfuse-web:3000 
    volumes:
      - ./config.yaml:/app/config.yaml
      - ./src/ai_platform_litellm:/app/ai_platform_litellm
    ports:
      - "4000:4000"
    command: [ "--config", "/app/config.yaml" ]
    networks:
      - ai_platform_net

networks:
  ai_platform_net:
    external: true

```
### 2-3. テスト

# ZitadelでのPythonテストクライアント用のキー作成手順

まずは、Zitadelのコンソールで **Projects** ➔ **`AI-Platform`** （作成済みのプロジェクト）を開いてください。

### 設定1：Pythonスクリプト用（Machine-to-Machine）

ブラウザを介さず、スクリプトが直接認証するための設定です。

1. **Applications** ➔ **Create** をクリック。
2. **Name**: `mcp-test-client`
3. **Type**: **`API`** を選択。
4. **Auth Method**: **`Basic`** を選択。
5. **情報の保存**:
* 表示される **`Client ID`** と **`Client Secret`** をメモし、Pythonコードの変数にセットします。

---
### LiteLLM（config.yaml）側での受け入れ設定

LiteLLM側では、MCPサーバーへ伝搬可能なHTTPヘッダー名を設定する。

```yaml
mcp_servers:
  my_hello_mcp:
    url: "http://mcp-server:5101/mcp"   # FastMCPの Streamable HTTP のデフォルトパス
    type: "http"                        # LiteLLM側で Streamable HTTP を指定する値
    extra_headers: ["Authorization", "litellm-api-key", "test"]  # ここで指定したヘッダーは、LiteLLMからMCPサーバーに転送される

```

## ステップ4：一連の流れの検証

1. **インフラ起動**: `docker compose up -d` (Zitadel, LiteLLM, MCPサーバー)。
2. **Zitadel設定**: 上記ステップ1の通り、Client ID等の発行。
3. **LiteLLM再起動**: `config.yaml` を読み込ませる。
4. **クライアント実行**: Pythonスクリプトを実行し、Zitadelで認証されたトークンがLiteLLMに受け入れられ、バックエンドの `my-backend-tool` の結果が返ってくれば成功です。
