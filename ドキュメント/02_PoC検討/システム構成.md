
# システム構成

生成AI基盤/AIレディ基盤を実現するためのシステム構成と各コンポーネントの役割。

## 本ドキュメントの見方（2軸）

本ドキュメントでは「層」という言葉を **2つの軸**で使います。

1. **コンポーネント層（配備単位）**：実体（デプロイされる箱）として存在するもの。
2. **横断機能（Cross-cutting concerns）**：複数コンポーネントにまたがって適用される関心事（例：Assurance、Observability）。

※したがって、ここに列挙される見出しは「直列に積まれる1本のスタック」を意味しません。

## コンポーネント層（配備単位）

### フロントエンド層

* **役割**: ユーザーインターフェースの提供と、非同期イベントのリアルタイム待機。

* ReactなどのSPAアプリケーション。
* ログイン機能、業務選択機能、指示入力、結果表示、HITL応答、イベント受信(バックグランド処理完了通知、非同期HITL通知)を持つ。
* ログイン処理ではEntraIDと連携して、OIDC認証を行う。

* **補足**:
  * **State保持**: 発行された `trace_id` をセッション内で保持し、再開（Resume）リクエスト時に必ず含める。
  * **リアルタイム通知**: バックグラウンド処理の完了や非同期HITLの発生を検知するため、**SSE (Server-Sent Events)** または **WebSocket** を通じてBFFからのプッシュ通知を受信する。

### APIゲートウェイ/BFF層

* **役割**: 境界防御、ID生成・マッピング、およびバックエンドへの動的ルーティング。

* フロントエンド層からのリクエストを処理する。
* Azure APIM、FastAPIによるBFFアプリケーションの組み合わせからなる。
* リクエストにBearerトークンが含まれていない場合は、ログイン用エンドポイントによるログイン処理を強制する。
* リクエストにtrace_idが含まれていない場合は、trace_idを新規生成する。
* バックエンドのApplication層のAPIエンドポイントをコンフィグファイルで管理する。

* **補足**:
  * **APIM（境界防御）**: EntraIDトークンのJWT検証、レート制限、およびWAFによる防御。
  * **BFF（IDの源泉）**: W3C `traceparent` に準拠した `trace_id` を生成。**「`trace_id` ⇔ `workflow_run_id` (Dify)」のマッピングテーブル（Redis等）を管理**し、製品間のIDの差異を吸収する。
  * **Payload変換**: 各バックエンドの仕様に合わせて、`query` ➔ `inputs` / `thread_id` への詰め替えを行う。
  * **責務の境界（重要）**: 生成AIの **回答チェック/ガードレール** はBFFに閉じず、後述の **AI Assuranceコンポーネント（例：LiteLLM Proxy / MCPゲートウェイ）** に集約する。

### 非同期連携基盤（Event Bus）

* **役割**: **Pub/Sub（メッセージブローカー）** による疎結合な非同期連携と、再試行・DLQなどの実行基盤を提供する。

* **位置づけ**:
  * Event Busは「Application層の一機能」ではなく、**BFF/Application/（必要ならTool）から利用される共有ミドルウェア**。
  * そのため本ドキュメントでは、レイヤーをまたぐ **横断の基盤コンポーネント**として扱う。

* **主なイベントの流れ（例）**:
  * **Publish**:
    * BFF：外部Webhook（Dify/GitHub/GitLab等）を受け、検証・正規化してPublish（必ず `trace_id` を付与）
    * Application：LangGraphの状態変化（`COMPLETED` / `INTERRUPTED` / `FAILED`）や、非同期ワーカーの実行結果をPublish
  * **Subscribe**:
    * Application：LangGraphワーカーがキューをSubscribeして非同期実行、`interrupt()` 等の状態変化をPublish
    * BFF：通知用イベント（完了通知/HITL発生）を購読し、SSE/WebSocketでフロントへPush

* **運用要件（必須）**:
  * **trace_id伝播**: すべてのメッセージ属性/ペイロードに `trace_id`（W3C traceparent準拠）を含める。
  * **冪等性**: 受信側は **重複配信を前提**に、`idempotency_key` 等で副作用を抑止する。
  * **リトライとDLQ**: 一定回数失敗したメッセージは **DLQ（退避キュー）** に隔離し、消失させない。
  * **順序/再処理ポリシー**: ワークフロー種別ごとに「同一 `trace_id` の順序保証が必要か」を明示する。

### Application層

* **役割**: AIエージェント（思考ロジック）の実行とステート管理。

* 生成AIエージェントを提供する層。
* LangGraphの処理を実行するAPIサーバーまたはDifyワークフローをAPI化したコンポーネント群
* 非同期処理用のWebHook通知処理、非同期HITL用のSuspend&Resume機能を必須とする。
* さらにバックエンドとなるTool層のMCPサーバーを呼び出すことも可能。MCPサーバーへのアクセスはLiteLLM MCPゲートウェイのアクセス制御を強制する。
* MCPサーバーを呼び出す際は、フロントエンドから送信されたBearerトークン、BFF層から渡されたLiteLLM APIキーなどを伝搬する。

* **補足**:
  * **LangGraph**: **AsyncSqliteSaver** を使用した非同期ステート永続化。BFFから渡される `trace_id` を `thread_id` として強制利用する。
  * **Dify**: **Human Inputノード** による中断管理。BFFへのコールバック時に自らの `workflow_run_id` を通知する。
  * **共通規約**: すべてのノードで `trace_id` をログに記録し、思考プロセスを透過的にする。

### Tool層

* **役割**: 実データへのアクセスと業務実行。

* 各種バックエンドリソースへアクセスするためのツール群。
* ユーザーの権限によりアクセス可能なデータ範囲を制御するため、フロントエンドから送信されたBearerトークンによる認証・認可を行う。
  (OIDCに対応していないバックエンドリソースはどうするかは要検討)
* アクセス制御の効率化のため、できるだけバックエンドリソースへのゲートウェイ製品を使用することを前提とする。
* データとともにデータの意味を返すようにする。

* **補足**:
  * **認証の委譲**: OIDC非対応リソースに対しては、Tool層（またはBFF）で **「トークン交換（Token Exchange）」** または秘密情報管理ツール（Azure Key Vault等）を用いた代理ログインを行い、権限を安全に昇格/変換する。
  * **セマンティック出力**: データの値だけでなく、メタデータ（項目の意味、データの新鮮度など）を付与してApplication層へ返す。

### AI Assuranceコンポーネント（配備単位）

* **役割**: LLMアクセスやツール実行の前後に **強制的に挟み込み（Enforcement）** を行うことで、ポリシー適用・遮断・監査を実現する。

* **主な構成例**:
  * **LiteLLM AIゲートウェイ（Proxy）**：LLMへのリクエスト/レスポンスをインターセプトし、ガードレール・監査・遮断を実施する。
  * **LiteLLM MCPゲートウェイ**：MCPツールの実行を集約し、ツールごとのアクセス制御・監査を実施する。

* **設計原則（すっきり分けるための条件）**:
  * Application/Toolから外部（LLM/ツール）へのアクセスは、原則として **ゲートウェイ経由に強制**する（直叩きを禁止する）。

## 横断機能（Cross-cutting concerns）

### AI Assurance（安全性・信頼性のガードレール）

* **役割**: LLMへのリクエスト/レスポンスやツール実行をインターセプトして、即時遮断/事後評価/監査などを実施し、生成AI処理の信頼性を保証する。

* **ポイント**: AI Assuranceは単一コンポーネントに閉じず、複数の **実装ポイント（Policy Enforcement Point; PEP）** を持ち得る。
  * 例：APIM（境界防御・入力検査）、LiteLLM Proxy（同期遮断/監査/遮断）、MCPゲートウェイ（ツールアクセス制御）、Tool実行直前（最終チェック）

* **推奨（責務分離）**:
  * BFFは「入口（認証・trace_id生成・ルーティング）」に寄せ、回答チェック等は **Assuranceコンポーネント（ゲートウェイ）側**に寄せる。

* **補足**:
  * **同期遮断（LiteLLM Proxy）**: ツール実行直前に、Assuranceロジック（ポリシーエンジン）が入力内容を検証し、リスクがある場合は処理を中断させる。
  * **監査ログ**: 誰がどのMCPツールに対し、どんなパラメータでアクセスを試みたかの履歴を `trace_id` と紐付けて記録する。

### 運用/監視（Observability）

* **役割**: 分散トレースの可視化と改善サイクル。

* Langfuseによるトレース保存、監査を実施

* **補足**:
  * **串刺し検索**: **Langfuse** を中心とし、APIMのログ（Azure Monitor）と `trace_id` でリンクさせる。
  * **性能監視**: 各コンポーネント（BFF、LLM、MCP、DB等）の処理時間を可視化し、ボトルネックを特定する。
