# 生成AIアプリケーション基盤：技術課題と対応方針 (Update: 2026-02-23)

## 0. この文書の位置づけ

本書は、生成AIアプリケーション基盤のアーキテクチャ検討における**主要な技術課題（全11件）**と、
その**対応方針（規約・設計原則）**を整理したグランドデザイン・ドキュメントです。

* **目的**：現場の実装負担や実装漏れを減らし、**統一的・透過的な仕組み（Proxy/Wrapper/SDK/基盤設定）**で課題を解決する。
* **前提**：例外ケースは後で専用設計とし、まずは一般的・普遍的なアーキテクチャを優先する。

---

## 1. アーキテクチャ基本原則（Architecture Principles）

* **透過的なインフラ解決**：実装漏れやバグを誘発する手動規約を避け、Proxy / Wrapper / SDK 等のインフラ構造で課題を解決する。
* **関心の分離**：フロントエンド(UI)、受付(Gateway)、頭脳(Supervisor)、信頼性(Assurance)、手足(Worker/MCP)を独立レイヤーで管理する。
* **Single Source of Truth**：
* 指示（Prompt）＝ **Langfuse**
* データの意味（Semantic）＝ **MCP**（ツールのスキーマ/description）
* 状態（State）＝ **LangGraph Checkpointer (SQLite/PostgreSQL)**

## 3. 課題別：対応方針（詳細）

## 課題1：アイデンティティ・チェインの構築（認証：Authentication）

**【✅ 設計確定】** EntraIDを唯一の正会員（IdP）とし、BFFを起点として全レイヤー（Dify / LangGraph / MCP）で「操作しているユーザー」を特定・リレーする。

### **■ トークン伝搬の実装規約**

1. **BFF：アイデンティティの結合とマッピング**
* ユーザーがEntraIDでログインし、BFF（FastAPI）がアクセストークンを取得。
* BFFは、トークン内のグループ情報に基づき、実行対象の **Difyアプリ固定APIキー** を特定する。
* Dify API (`/v1/workflows/run`) 呼び出し時、`inputs` にユーザーのアクセストークン、`user` に識別子を封入して送信する。


2. **Dify：セキュアなパイプライン（運び屋）**
* Difyは受け取った `inputs` を、開始ノードの **`access_token` 変数** へマッピングする。
* HTTPリクエストノードにて `Authorization: Bearer {{access_token}}` を付与し、後続のAPIを呼び出す。


3. **LangGraph API：セッションの最終検証**
* リレーされたトークンを受信し、OAuth2イントロスペクション等で「今、この瞬間にそのユーザーが有効か」を最終検証する。

---

## 課題2：多層防御によるアクセス制御（認可：Authorization）

**【✅ 設計確定】** LiteLLMのネイティブ機能を用いた「機能認可」と、IdPトークンによる「データ認可」の二重施錠を行う。

### **■ 認可のチェックゲート（Multi-Layer Gate）**

| レイヤー | 認可の仕組み | 制御の対象（認可の範囲） |
| --- | --- | --- |
| **BFF / LiteLLM** | **仮想キー (Virtual Keys)** | **アプリ認可**: 特定のシステム・チームがどのモデル、どのDifyアプリを動かして良いか。 |
| **LiteLLM Proxy** | **アクセスグループ (`access_group_ids`) / オブジェクト権限** | **機能認可**: その仮想キーが、特定のMCPサーバー（送金用等）にアクセスして良いか。 |
| **LiteLLM Hook** | **Guardrail (Allowed Tools)** | **実行認可**: LLMが提案したツール名が、キーごとの許可リストに含まれているか。 |
| **Tool層 (MCP)** | **EntraID ユーザー個人トークン** | **データ認可**: ユーザーに許可された範囲のデータ（行・列）のみをバックエンドから取得。 |

### **■ 技術側実装規約**

* **LiteLLMの活用**: `/key/update` エンドポイントを用い、仮想キーに `access_group_ids` を紐付け、システム間の境界を物理的に分離する。
* **データ認可の委譲**: MCPツールは、Difyから伝搬された「個人トークン」を用いてバックエンドを叩く。システム（仮想キー）が許可されていても、ユーザー自身に権限がなければデータ取得を拒否するZero Trustモデルを成立させる。

---
### 課題3：AI Assurance層（生成AI信頼性保証）

**【✅ PoC一部検証済】** 全社規模での安全なスケールのために、LiteLLM Proxyを「関所」としたデュアルゲート（同期＋非同期）を構築する。

* **同期ゲート（リアルタイム・ガードレール）**：
LiteLLMの `CustomLogger.async_pre_call_hook` 等を用いて実装。AIがツールを呼び出した際の特殊なペイロード（`content: null`）でエラーが起きないよう考慮しつつ、PIIマスキングや機密情報の検知を行い、違反時は例外スローでLLMへの通信を強制遮断する。

* **非同期ゲート（継続的品質評価）**：
* **Success (`success_callbacks`)**: レスポンス完了後、Trace ID・プロンプト・回答のセットをメッセージキューへ送信し、バックグラウンドのRAGASワーカーが精度（Faithfulness等）を採点する。

---

### 課題4 & 5：非同期アーキテクチャとHITL（Human-in-the-Loop）状態管理

**【✅ PoC検証済】** 長時間の推論ループによるタイムアウトを防ぎ、セキュアな状態管理と承認プロセスを実現する。

* **イベント駆動・Webhookコールバック（FastAPI）**：
UI（Dify）からのリクエストに対し、FastAPIは即時応答（HTTP 200/202）を返し、処理をバックグラウンドタスクへ逃がす。処理の完了や承認待ちが発生した際は、`event_type` と `metadata` を付与した Payload を Webhook で統合ルーターへ通知し、ブロードキャストストームを防ぐ。

* **LangGraphによる状態の永続化（Checkpointer）**：
AIの頭脳（Supervisor）は LangGraph で構築し、`SqliteSaver` (本番は PostgreSQL) を Checkpointer として利用する。プロセスが終了しても、`thread_id` をキーとして過去の会話履歴やツールの実行計画（State）が完全に復元される。

* **`interrupt_before` による厳格なHITL制御**：
副作用のあるツール（送金等）を実行するノードの手前でグラフの実行を強制停止する。ユーザー（Dify）からの `/api/resume_async` リクエストを受け取るまでツールは絶対に実行されない。


**【✅ 設計確定】** 相関IDとして W3C `trace-id` を採用し、ログと状態を物理的に統合する。

* **ID 統一規約**: 非同期処理の開始時（BFF または Dify レイヤー）に生成された W3C `traceparent` から **`trace-id`** を抽出し、これを LangGraph の **`thread_id`** および Webhook のコールバック ID として利用する。
* **整合性の確保**: `traceparent` が存在しないリクエスト（直接実行等）に対しては、境界レイヤーで新規に `traceparent` を発行し、以降の全プロセスでこれを唯一の識別子として強制伝搬させる。

---
### 課題6：統合オブザーバビリティ（E2Eトレース）

**【✅ 設計確定】** ログを「後から見るもの」から「状態を特定する鍵」へ昇格させる。
複数コンポーネントを跨ぐリクエストを一本の糸で可視化する。

* **W3C Trace Contextの採用**：DifyからLLM、MCP、Backendまで **`traceparent`** ヘッダーを欠落なく伝搬させる。

* **Langfuseへの集約**：LiteLLM Proxyで `traceparent` から抽出した trace-id をLangfuseの `trace_id` として強制マッピングし、ガードレール遮断理由や評価スコアをすべて集約する。

* **Langfuse / Azure Monitor 連携**: 全てのステート（Checkpoint）に `trace_id` を付帯させることで、Langfuse 等のダッシュボードから、その実行時の「変数の中身」や「承認待ち状態」を直接参照可能にする。

### W3C `traceparent` を ID として利用するメリット

1. **「点」と「線」の結合**:
* `uuid4` の場合：データベース上の状態（State）と、Application Insights や Langfuse 上のログ（Trace）を紐付けるために、別途マッピングテーブルや検索が必要になります。
* `trace-id` の場合：ログの ID そのものがデータベースのキーになるため、トラブルシューティング時に「ログから状態を引く」「状態からログを引く」ことが、ツールを跨いでシームレスに行えます。

2. **分散トレーシングの標準準拠**:
* Dify から発行された `traceparent` をバックエンドまで引き継ぐことで、BFF ➔ Dify ➔ LangGraph ➔ MCP という複雑な経路を一本の糸（Trace ID）で可視化できます。

3. **互換性**:
* W3C `trace-id` は 128ビット（32文字の16進数）であり、`uuid4` とデータサイズが同じです。LangGraph の `thread_id`（文字列）としてそのまま格納可能です。

### 実装時の注意点：フォーマットの変換

W3C `traceparent` は `00-4bf92...-00` という形式ですが、LangGraph やデータベースのキーとして使う場合は、真ん中の **`trace-id`（32文字の16進数部分）** だけを抽出して使うのが一般的です。

```python
# 実装イメージ
trace_parent = request.headers.get("traceparent") # 00-adce...-01
trace_id = trace_parent.split("-")[1] # trace-id部分のみ抽出

# LangGraphの実行
config = {"configurable": {"thread_id": trace_id}}
inputs = {"messages": [...]}
app.invoke(inputs, config)

```

* **IDフォーマット規約**:
* W3C Trace Context 準拠の 55文字ヘッダーを採用。
* 中央の 32文字 `trace-id` を抽出し、ステート管理用の `thread_id` として各 API へ引き渡す。


* **BFFミドルウェア責務**:
* `Authorization` ヘッダー欠落時はログインを強制。
* `traceparent` ヘッダー欠落時は新規生成し、`log` および `metadata` への注入を保証する。

この ID 統合により、銀行業務のような「証跡（ログ）」と「結果（データ）」の厳密な一致が求められるシステムにおいて、非常に強力な監査基盤を構築する。

**【✅ 設計確定】** BFFを「IDの源泉（Root of Trust）」とし、製品仕様によるIDの差異をBFFレイヤーで吸収・統合する。

### **■ ID設計の規約（Identity Architecture）**

| ID名称 | 生成・管理場所 | 役割・用途 |
| --- | --- | --- |
| **`trace_id`** | **BFF** | **不変の共通キー（Correlation ID）**<br><br>・W3C `traceparent` から抽出される32文字の16進数。<br><br>・全ログ、Langfuseトレース、通知、およびフロントエンドとの通信に使用。 |
| **`thread_id`** | **BFF (注入)** | **LangGraphのステートキー**<br><br>・BFFが `trace_id` をそのまま `thread_id` として強制指定する。マッピング不要。 |
| **`workflow_run_id`** | **Dify (自動採番)** | **Dify内部の実行管理キー**<br><br>・Difyが自動発行するID。非同期処理の再開（Resume）やWebhookの特定に必須。 |

### **■ 実装規約：IDマッピング・ライフサイクル**

1. **IDの生成と注入（Request時）**
* BFFは `trace_id` を生成し、Dify/LangGraphへの全リクエストに `traceparent` ヘッダーおよびペイロード内の `trace_id` 変数として注入する。


2. **リソースIDの捕捉（Response時）**
* **Dify実行時**: BFFはDify APIからのレスポンスに含まれる `workflow_run_id` を取得し、自らが発行した `trace_id` とのペアを **マッピングテーブル（Redis等）** に保存する。
* **保存期間**: ワークフローが完全に終了（完了/却下）するまで、または一定期間（例: 24時間）保持する。


3. **相関の復元（Webhook / Resume時）**
* **Webhook（Dify ➔ BFF）**: Difyから `workflow_run_id` 付きの通知が届いた際、BFFはマッピングテーブルから対応する `trace_id` を逆引きし、ログの継続性とユーザー通知の宛先を特定する。
* **Resume（UI ➔ BFF）**: フロントエンドから `trace_id` で再開リクエストが届いた際、BFFはマッピングテーブルから `workflow_run_id` を取得し、DifyのResume APIを呼び出す。



### **■ 監視プラットフォームでの統合**

* **Langfuse連携**: Dify/LangGraphから出力される全てのトレースに `trace_id` をタグ付けすることで、製品を跨いだ「一連の思考プロセス」を一つの画面で可視化する。
* **トラブルシューティング**: ユーザーから「エラーが出た」と報告された際、提示された一つの `trace_id` を入力するだけで、BFFのログ、Difyの実行詳細、LangGraphの状態、バックエンドのSQL発行履歴までを串刺しで特定可能にする。

### 課題7：データの意味付け

LLMのハルシネーションや破壊的実行を防ぐための規約。

* **メタデータの2層規約**：原則はスキーマ定義（`description`）による静的意味の付与。実行時の警告等が必要な高リスクツールのみ、JSONエンベロープ（動的）による文脈付与を行う。

---
### 課題8&9：自律型ワーカーのサンドボックスと性能最適化

完全な隔離環境（コンテナ）をオンデマンドで立ち上げつつ、コールドスタートの遅延を隠蔽する。

* **Ephemeral（使い捨て）コンテナ**：リクエストごとにコンテナを起動・機密注入・回収し、即時破棄（Self-Destruct）する。
* **予測的投機実行（Speculative Boot）**：上位エージェントがワーカーの必要性を予見した段階で、非同期でプロビジョニングを開始。
* **セキュア・ウォームプール**：認証情報を持たない「空のコンテナ」をスタンバイさせ、実行直前にトークンをメモリ（tmpfs等）へ注入することで初動を高速化する。

### 課題10：プロンプト統合管理

指示内容の二重管理を防ぎ、無停止でのアップデートを実現する。

* **Langfuse CMS化**：コード内にプロンプトをハードコードせず、LiteLLMの `general_settings` やアプリ層のSDKを利用してプロンプトを動的フェッチする。
* **キャッシュ＋TTL機能**：SDKの標準機能により、サーバー再起動なしで最新の指示を全レイヤーへ配信する。

## 課題11：GatewayとBFFの役割分担（ハイブリッド・アーキテクチャ）

**【✅ 設計確定】** 境界防御（APIM）と業務ロジック（BFF）を分離し、堅牢性と柔軟性を両立させる。

### **■ 構成の定義と分担**

| コンポーネント | 主な責務（Policy / Code） |
| --- | --- |
| **Azure APIM** | **境界防御・共通基盤**: <br><br>・EntraIDトークンのJWT検証<br><br>・WAF / DDoS対策 / レート制限<br><br>・共通 `traceparent` の発行（クライアント未付与時）<br><br>・バックエンド（BFF）の隠蔽と一元監視 |
| **BFF (FastAPI)** | **AI/業務コンテキスト変換**: <br><br>・`config.yml` に基づく動的なバックエンド（Dify/LangGraph）ルーティング<br><br>・リクエストペイロードの変換（`query` ➔ `inputs` / `thread_id` へのマッピング）<br><br>・非同期HITLの状態管理（再開ロジック）<br><br>・OpenAPI (Swagger) スキーマの動的生成 |

### **■ 技術的実装規約**

1. **関所の二重化**: クライアントは必ず APIM を経由する。APIM は検証済みのリクエストのみを BFF に流し、BFF は `agent_gatekeeper` （依存関係）によって最終的なトークンとトレースコンテキストの整合性を保証する。
2. **動的仕様の集約**: バックエンドごとの OpenAPI 仕様の差異は BFF 内の変換レイヤーで吸収する。これにより、Gateway (APIM) の設定は「一律のパス」としてシンプルに保ち、AIエージェントの頻繁な追加・変更は `config.yml` の更新のみで対応可能とする。
3. **観測性の統合**: APIM で発行された `trace-id` を BFF が抽出し、LangGraph の `thread_id` として利用。これにより、Azure Monitor (APIMログ) から Langfuse (AIログ) までを一つの ID で検索可能にする。

---
### 課題12：Write系ツールのDry Run

LLMのハルシネーションや破壊的実行を防ぐための規約。

* **Dry Runの実装回避**：開発者にシミュレーション実装を強制しない。ツール呼び出しのペイロード（AIの意図）をインターセプトして人間に承認させるか、Terraformのように「Plan」と「Apply」のツールを物理的に分離する設計を推奨する。

---
### 課題13：LangGraph Stateのスキーマ進化

コード更新による、フリーズ中のState復元エラーを防ぐ。

* **Version Tagging**：Stateに `_schema_version` を持たせ、構造変更時にインクリメントする。
* **State Migration**：旧バージョンを最新スキーマへ変換するアダプター関数を実装するか、メジャーアップデート時は旧ワーカーを並行稼働（Blue/Green）させて古いスレッドを消化する。

---

## 4. 開発・運用を担保する「ゲート機構」（CI/CD & Runtime）

規約を個人の努力に依存せず、システム的に強制・誘導する仕組み。

1. **ゴールデンパス（自動的に正しい実装へ誘導）**：
* `langfuse.get_prompt()`、`traceparent` 伝播、SSE接続をデフォルトで内包した標準SDK/テンプレート（Scaffold）を提供する。


2. **CI/CDゲート（未対応はデプロイ拒否）**：
