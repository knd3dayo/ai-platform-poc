# 生成AI基盤（AIレディ基盤） 共通基盤アプリの設計方針

## 1. 目的

本ドキュメントは、生成AIアプリケーションを構成する各層（BFF、LiteLLM、Application、Tool）が守るべき共通の規約を定め、エンタープライズ品質の「信頼性」「追跡可能性」「開発効率」を担保することを目的とする。

---

## 2. コア設計原則

* **Traceability (追跡可能性)**: 全ての処理を `trace_id` で串刺しにし、Langfuseおよびログで一気通貫の調査を可能にする。
* **Decoupling (疎結合)**: エージェント（Application層）は自身の状態を外部に「実況（Event）」するだけであり、永続化の責任を持たない。
* **Identity Propagation (認証伝播)**: ユーザーのBearerトークン（JWT）を最下層のDenodoまで安全に運び、ユーザー権限に基づいた動的アクセス制御を徹底する。
* **Resilience (回復性)**: AI特有の不安定さを許容し、リトライやハートビート、キャンセル処理を標準実装する。

---

## 3. 共通データプロトコル（スキーマ）

### 3.1 共通コンテキスト (Context Object)

各層の関数間・コンポーネント間で必ず伝播させる情報の定義。

* `trace_id`: W3C `traceparent` 準拠。
* `user_id` / `bearer_token`: ユーザー識別と認可情報。
* `tenant_id`: 組織/部署単位の識別。

### 3.2 標準イベント・ペイロード

Event Bus（Redis Streams）に流すJSONの標準形式。

```json
{
  "header": {
    "trace_id": "string",
    "event_type": "AGENT_THOUGHT | TOOL_CALL | JOB_STATUS | ERROR",
    "occurred_at": "ISO8601",
    "producer": "component_name"
  },
  "payload": {
    "level": "INFO | DEBUG | WARNING | SUCCESS",
    "message": "AIが現在行っていることのテキスト",
    "metadata": { "tool_name": "string", "duration": "float" },
    "data": { "JSONオブジェクト（任意）" }
  },
  "idempotency_key": "重複排除用キー"
}

```

## 4. 層別設計方針と実装要件

### 4.1 Tool層 (MCP / Denodo)

* **認証の委譲 (Token Exchange)**:
* 渡されたJWTを検証し、Denodoへのクエリ実行時に「そのユーザーとして」アクセスするよう認証を委譲する。

* **セマンティック応答**:
* データの値だけでなく、プロンプトのヒントとなるメタデータ（「この値はZendeskのチケットIDです」等）を常に付与する。

### 4.2 ガバナンス層(LiteLLM Proxy)

* **LiteLLM Hookロジック**:
* **事前Hook**: プロンプト内のPII（個人情報）を検知し、決定論的なアルゴリズムでマスクする。
* **事後Hook**: LLMの回答を検証し、ポリシー違反（ハルシネーション等）があれば `trace_id` を添えてアラートをPublishする。


* **Audit Log**: 誰がどのツールを呼んだかのログを、`trace_id` と共にLangfuseへ送信する。

### 4.3 Application層 (LangGraph / Dify)

* **実況中継の実装**:
* 各ノード（思考ステップ）の開始・終了時に、必ず標準イベント形式でメッセージをPublishする。
* 処理が長時間に及ぶ場合、接続維持（Keep-alive）のためのハートビートを定期発行する。


* **キャンセル・ハンドリング**:
* Event Bus上の `JOB_CANCELED` イベントを定期的にチェック（または割り込み処理）し、検知した場合は `interrupt` してリソースを解放する。

### 4.4 API Gateway / BFF層

* **ID管理**: リクエスト受信時に `trace_id` がなければ新規発行し、レスポンスヘッダに付与する。
* **状態の清書 (State Writer)**: Event BusをSubscribeし、イベントをフロントエンドへ転送（SSE）すると同時に、状態管理DB（Redis JSON）へ履歴を追記（Append）する。
* **セッション管理**: ブラウザ再起動時、Redisから最新のスナップショットを抽出して初期画面を構築する。


---

## 5. エラーハンドリングと再試行

* **指数バックオフ**: 外部API呼び出しには共通のリトライライブラリを適用する。
* **DLQ (Dead Letter Queue)**: Event Busで一定回数失敗したメッセージはDLQに退避させ、`trace_id` で検索可能な状態にする。

---

## 6. 開発ガイドライン（PoC向け）

* **言語**: Python 3.10+ (asyncio必須)
* **ライブラリ**: FastAPI, Redis-py, LangGraph, Pydantic (バリデーション)
* **SDK**: 全てのアプリケーションは、基盤チームが提供する「共通Context伝播ライブラリ」を使用すること。
