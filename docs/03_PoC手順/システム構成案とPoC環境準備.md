# 生成AI基盤（AIレディ基盤） システム構成と実装要件

本ドキュメントでは、エンタープライズ向けの生成AI基盤を実現するためのシステム構成を、アーキテクチャを構成する「7つの層（フロントエンド＋裏側6層）」に分割し、それぞれの層が担う役割と実装すべき具体要件を定義します。

## 0. フロントエンド層

ユーザーとシステムが直接対話するインターフェースであり、非同期で行われるAIの思考プロセスをリアルタイムに可視化します。

* **提供する機能**:
* ユーザー認証（ログイン）と業務UI（チャット、フォーム等）の提供。
* AIの思考プロセス（実況中継）のリアルタイム表示と、HITL（人間による承認）への応答。
* セッション切断・再ロード時のUIの完全復元。
* 実行中ジョブのキャンセル要求。


* **機能提供に必要なコンポーネント**:
* Webブラウザ、または社内ポータル/モバイルアプリ。


* **機能提供に必要な技術**:
* React等のSPA（Single Page Application）フレームワーク。
* OIDC (OpenID Connect) / OAuth 2.0。
* SSE (Server-Sent Events) または WebSocket。


* **機能提供に必要なアプリ実装**:
* EntraID等のIdPと連携したOIDCログイン処理と、Bearerトークンの取得・付与。
* BFFから受け取った `trace_id` のセッション内保持と、APIリクエストへの付与。
* BFFの同期API（GET）を叩いて初期状態（過去の思考履歴含む）を画面に復元する処理。
* BFFとSSEコネクションを確立し、流れてくるイベントで画面（進捗バーやチャット）を動的に更新する処理。



## 1. API Gateway / BFF層

システム全体の入口として境界防御を行い、トラフィックのルーティングと「UI向けのジョブ状態・履歴」の永続化を独占管理します。

* **提供する機能**:
* 境界防御（WAF、レート制限）とトークン検証（認証の強制）。
* 分散トレースの主キーとなる `trace_id` の新規発行と伝播。
* リクエストの性質に応じた動的ルーティング（Fast Track / Slow Track）。
* UI復元のためのジョブ状態および「エージェント思考履歴」の追記・永続化。


* **機能提供に必要なコンポーネント**:
* API Gateway（Azure APIM 等）。
* BFFアプリケーションサーバー。
* IdP（EntraId、Oktaなど）


* **機能提供に必要な技術**:
* FastAPI (Python) または Node.js/Go。
* JWT検証技術。


* **機能提供に必要なアプリ実装**:
* **[APIM]** JWTトークン検証と不正アクセスのブロック。
* **[BFF]** W3C `traceparent` 準拠の `trace_id` 発行と、バックエンドAPI群のエンドポイント管理（コンフィグ）。
* **[BFF]** フロントエンドとバックエンド（Dify等）間のペイロード変換と、製品ID（`workflow_run_id`等）と `trace_id` のマッピング。
* **[BFF]** Event Busから流れてくる細かなイベントを購読し、フロントエンドへSSEでPush配信すると同時に、状態管理DBの対象ドキュメントの配列へ履歴として追記（Append）する処理。



## 2. 非同期連携基盤（Event Bus / 状態管理DB）

処理時間が読めないAIジョブやHITLのメッセージを確実かつ疎結合にリレーし、DBへの高頻度なポーリングを防ぐ巨大な非同期ネットワークです。

* **提供する機能**:
* Pub/Subモデルによるコンポーネント間の疎結合な非同期メッセージング。
* AIが大量に吐き出す細かなイベントを受け止めるショックアブソーバー（緩衝材）。
* 処理失敗時のリトライと、デッドレターキュー（DLQ）への退避。
* 状態管理DB。

* **機能提供に必要なコンポーネント**:
* メッセージブローカー（Event Bus）。
* NoSQL（MongoDB, CosmosDB等）または Redis（KVS）。


* **機能提供に必要な技術**:
* Redis Streams, Apache Kafka, またはクラウドマネージドキュー（Azure Service Bus等）。


* **機能提供に必要なアプリ実装**:
* ※基盤自体はミドルウェアであるため、各層（BFFやApplication）から利用するための**共通SDK/ライブラリ**として以下を実装する。
* すべてのメッセージペイロードへの `trace_id` の必須付与。
* `idempotency_key` を用いた重複排除（冪等性の担保）ロジック。
* 規定回数エラーになったメッセージをDLQへ隔離する再処理ポリシーの設定。



## 3. Application層

AIの「脳」として、プロンプトやエージェント（WF型/SV型/自律型）を用いて推論、タスク分解、オーケストレーションを実行します。

* **提供する機能**:
* 業務ロジックに基づくLLMの呼び出しと、ツールの実行指示。
* 人間の承認（HITL）を待つ際のリソース解放と、一時的な文脈（State）の保存。
* 思考プロセスの外部への報告（実況中継用イベントの送信）。


* **機能提供に必要なコンポーネント**:
* AIエージェント実行環境（APIサーバーまたはコンテナワーカー）。
* エージェント専用の内部記憶DB（Checkpointer）。


* **機能提供に必要な技術**:
* LangGraph, Dify, 自律型エージェント実行環境(YOLOモード用サンドボックス。Docker、gVisor、FireCrackerなど）。
* コーディングエージェント（Claude Code、Codex、Gemini CLI、Cline CLIなど)
* Python/TypeScript。
* SQLite/PostgreSQL（Checkpointer用）。


* **機能提供に必要なアプリ実装**:
* Event Busを購読（Subscribe）し、非同期でジョブを起動する処理。
* **[LangGraph]** `AsyncSqliteSaver` 等を使用したStateの永続化と、HITL要求時の `interrupt()` による処理の中断・再開ロジック。
* ノード遷移やツール実行のたびに `AGENT_THOUGHT` 等のイベントをEvent BusへPublishする処理。
* Event Busから「キャンセル命令」を受信した際に、即座に処理を破棄して安全に終了するロジック。



## 4. AIガバナンス層

Application層から外界へのすべての通信に強制的に介在（Enforcement）し、企業のポリシー適用とセキュリティ監査を担保します。

* **提供する機能**:
* プロンプト送信前のPII（個人情報）マスキングと、入力内容のガードレール検証（同期遮断）。
* LLMアクセスやツール実行の集約と、監査ログの取得。
* 回答内容の非同期モニタリング（ハルシネーション評価）。


* **機能提供に必要なコンポーネント**:
* LiteLLM Proxy（AIゲートウェイ） LLMアクセス用ゲートウェイ。
* LiteLLM Proxy（MCPゲートウェイ）ツールアクセス集約用。
* マスキング用内部処理サーバー。


* **機能提供に必要な技術**:
* LiteLLM。
* Microsoft Presidio（決定論的マスキング）または小規模ローカルLLM。


* **機能提供に必要なアプリ実装**:
* Application層からの直叩きを禁止し、すべてProxyを経由させるルーティング設定。
* 外部LLMへデータが渡る手前での、決定論的アルゴリズム（Presidio等）を用いたPIIの検知・置換処理（※デッドロック回避のため外部LLMは使わない）。
* 誰がどのモデル/ツールにアクセスしたかを `trace_id` と共に記録する監査ログ出力機能。



## 5. Tool層

社内システムやSaaSへのAPIアクセスを抽象化し、ユーザーの権限に基づいた安全なデータ入出力と、AIへのセマンティック（意味論的）なコンテキスト提供を行います。

* **提供する機能**:
* バックエンドリソースへの標準化されたインターフェース（MCP等）の提供。
* ユーザー権限（Bearerトークン）の引き継ぎと、アクセス制御（認可）。
* データの値だけでなく、その「意味」や「制約」のメタデータ付与。


* **機能提供に必要なコンポーネント**:
* MCP (Model Context Protocol) サーバー群。
* バックエンドリソースへのゲートウェイ製品。データ仮想化サーバー（Denodo等）や各種社内SaaS/DB。


* **機能提供に必要な技術**:
* MCP, REST API, GraphQL。
* Token Exchange（OAuth 2.0）。


* **機能提供に必要なアプリ実装**:
* フロントエンド〜BFF〜App層を伝播してきたBearerトークンを検証し、各バックエンドシステムの権限モデルに変換（代理ログイン等）する処理。
* 人間を待つ処理（HITL）や状態の永続化を内部に持たない、**完全ステートレス**なAPIとしての実装。
* エージェントがデータを正しく解釈できるよう、プロンプトスキーマやカスタムインストラクションを含めて応答する実装。



## 6. 運用監視基盤（Observability）

すべての層を `trace_id` で串刺しにし、システムの健全性、コスト、およびAIの回答品質を横断的に監視します。

* **提供する機能**:
* 「誰のジョブがどこで止まっているか（または失敗したか）」の分散トレースの可視化。
* トークン消費量やモデル別コストの集計。
* AIの回答精度やガードレール遮断率の評価（Eval）とアラート。


* **機能提供に必要なコンポーネント**:
* 統合監視ダッシュボード。


* **機能提供に必要な技術**:
* OpenTelemetry。
* Langfuse（LLM特化監視）, Datadog または Azure Monitor。


* **機能提供に必要なアプリ実装**:
* 各層のアプリケーション（BFF, App, ガバナンス等）への OpenTelemetry SDK の組み込み。
* HTTPヘッダ（同期）および Event Busペイロード（非同期）を跨いだ `traceparent` の確実なコンテキスト伝播。
* LiteLLM Proxyのコールバック機能等を利用した、Langfuseへの非同期でのログおよび評価スコア（Shadow Eval結果）の送信処理。

## 8. PoC環境構築ロードマップ（5ステップ）

PoCでは、いきなり全層を組むのではなく、**「データが流れる最小の縦軸」**から構築します。
* PoC環境は検証用Azure環境(生成AIのオプトアウト申請済みのもの)を使用。
* バックエンドリソース(Zendesk、Redmineなど)は、実業務で使用しているもの流用。  
  ただし、ネットワーク的な制約によりPoC環境からアクセス不可の場合は、疑似的なサーバーを構築。
* PoCにて得られた知見を元に社内NW上に同様の環境を構築して、実業務に活用する。

### Step 1: 共通基盤とガバナンスの確立

まずは「実行環境」と「関所」と「土管」を作ります。

1. **Docker環境のセットアップ**: 後述のソフトウェア、アプリケーションの実行環境の構築
2. **Langfuseのセットアップ**: 全ての実験ログがここに入るようにする。
3. **Difyのセットアップ**: 「WF型」のアプリケーション実行環境。
4. **サンドボックス環境のセットアップ**: 「自律型」のアプリケーション実行環境。
5. **LiteLLM Proxyのデプロイ**: 外部LLM（Azure OpenAI等）への接続を集約し、APIキーを隠蔽。
6. **Redisのプロビジョニング**: Event Busと状態管理DBとしての箱を用意。

### Step 2: AI脳と手足の接続

「思考（推論）」と「実行（ツール利用）」を具体的に繋ぎ込みます。AIに丸投げするのではなく、適切な「教育」と「道具」を与えるフェーズです。

#### 1. MCPサーバーの構築（データセマンティクスの確立と権限統制）

* **内容**: データ仮想化製品 **Denodo** をバックエンドに据え、調査対象リソース（Zendesk, Redmine, 各種DB等）を仮想統合したMCPサーバーを構築します。
* **データセマンティクス**: 単なる値の返却ではなく、Denodo上のメタデータ（項目の論理名、意味、リレーション）をAIに提供し、AIがデータの背景を正しく推論できる基盤を作ります。
* **ポイント**: データの値だけでなく、Denodoが持つ**メタデータ（論理名、説明文、リレーション情報）**をAIに渡すことで、AIが初見のデータでも「これは顧客IDであり、ZendeskのIDと紐付く」といった推論を自律的に行えるようにします。

* **ユーザー権限に基づくアクセス制御（RBAC/ABAC）**:
* **ID伝播（Identity Propagation）**: フロントエンドで発行された **Bearerトークン（JWT）** を、BFF、Application層、MCPサーバーを通じてDenodoまで透過的に伝播させます。
* **動的フィルタリング**: Denodo側で「誰がリクエストしているか」を識別し、同一のMCPツール呼び出しであっても、**ユーザーの権限に応じて「行レベル（特定の顧客データのみ）」「列レベル（個人情報のマスキング）」で動的にフィルタリング**された結果のみをAIに返します。
* **一元管理のメリット**: 各バックエンドリソース個別に権限を設定するのではなく、Denodoという「セマンティックレイヤー」で権限を集中管理することで、AIエージェントによるデータの不正な「覗き見」を構造的に防止します。


#### 2. エージェントの自律スキル定義（Claude.md / Agent Skills）

* **内容**: サンドボックス環境で稼働するワーカーに、ログ解析やナレッジ検索の「プロの流儀」を仕込みます。
* **権限の意識**: エージェント自身にも「自分がアクセスできる範囲」と「閲覧できない情報の種類」を `CLAUDE.md` 等で明示的に教育し、権限エラーが発生した際の振る舞い（ユーザーへの承認依頼など）を定義します。
* **ポイント**: 単なる指示だけでなく、**「ログでExceptionを見つけたら、まずその前後の50行を読み、タイムスタンプをキーにWikiを検索せよ」**といった、ベテラン担当者の手順（Standard Operating Procedure）を、`CLAUDE.md` 等の定義ファイルとしてエージェントに注入します。

### 3. LangGraphでのプロトタイプ実装（オーケストレーション）

* **内容**: 自律型エージェントの処理を束ねる**SV（スーパーバイザー）型**の論理フローを実装します。
* **ポイント**: **「状態の永続化（Checkpointer）」**の実装もここで行います。調査が複雑になり、時間がかかる場合でも、途中の思考状態をDBに保存し、後から安全に再開できる基盤を確立します。

#### 4. Difyワークフローの構築（確実な実行部隊のAPI化）

* **内容**: 再現性が求められる定型タスク（ログの特定箇所抽出、定型書式での要約、PII検出、特定DBへのRAGなど）をDifyのワークフローとして構築し、**外部呼び出し可能なAPI**として公開します。
* **オーケストレーションへの組み込み（2つの呼び出し経路）**:
* **BFFからの直接呼び出し（Fast Track）**: 複雑な推論を必要としない定型的な要求に対し、BFFが直接Dify APIを叩くことで、低遅延かつ安定した応答を実現します。
* **LangGraphからの呼び出し（Slow Track / ツール利用）**: LangGraph（スーパーバイザー）が複雑な調査の過程で、「このステップは定型的なデータ加工が必要だ」と判断した際、Difyのワークフローを**一つの「Tool」または「専用ノード」**として呼び出します。


* **導入のメリット**:
* **LLMのゆらぎ抑制**: 長大なプロンプトをLangGraphで管理するのではなく、Difyのビジュアルエディタで「確実に動く手順」として切り出すことで、出力の安定性を高めます。
* **開発効率の向上**: 画面構成や定型プロンプトの調整はDify側で完結するため、LangGraph側のコード（脳のロジック）をシンプルに保つことができます。

### 5. Event Bus連携（実況中継の基盤）

* **内容**: 各コンポーネントの動きを `trace_id` と共にEvent Busへパブリッシュします。
* **ポイント**: 特に、自律型ワーカーが「試行錯誤」している最中の**「失敗ログ（検索して見つからなかった、等）」**も流すようにします。これが後のBFFでの実況中継における「AIの苦労が見える化される」というUX上の信頼感に繋がります。

---

### Step 3: BFFによる「清書」と「通知」の実装

「ユーザーへの見せ方」を整えます。

1. **FastAPIでのBFF構築**:
* Event Busからログを拾ってRedis（状態管理DB）にAppendする。
* フロントエンド向けにSSEエンドポイントを公開する。


2. **キャンセル処理の実装**: BFFからEvent Busへ停止信号を送る仕組みを作る。

### Step 4: フロントエンドの実装

「ユーザー体験」を完成させます。

1. **SSE受信用UIの開発**: 実況中継ログを動的に表示するパーツの実装。
2. **UI復元機能**: リロード時にBFFからRedisの履歴を読み込む処理。
3. **Entra ID連携**: 本番を見据えた認証ログインの組み込み。

### Step 5: シナリオテストと改善

実務データでの検証です。

1. **「データ仮想化サーバー停止」シナリオの流し込み**: 実際にログを投げ、AIの思考プロセスが正しく可視化され、人間がHITLで承認できるかを確認。
2. **ガバナンス層のチューニング**: PIIマスキングが正しく機能し、デッドロックが起きないか確認。

## PoCサーバー構成案（4サーバー集約モデル）

Azure VMを利用する場合、以下のような4台構成に集約するのが「管理しやすく、かつ本番に近い特性」を検証可能と考えれる。

### 1. サーバーA：APIフロント・ゲートウェイ（Entry Node）

**役割**: システムの玄関口。トラフィックの交通整理と認証・認可の初動を担います。

* **同居させるコンポーネント**:
* **BFF (FastAPI)**
* **LiteLLM Proxy** (AI/MCPゲートウェイ)


* **集約の理由**:
* 両者とも軽量なPythonベースのプロキシ的な動作であり、CPU/メモリ消費が比較的少ない。
* 「入口のセキュリティとルーティング」という責務が共通している。


* **推奨スペック**: 汎用的なDシリーズ (例: D2s_v5)

### 2. サーバーB：非同期データ・運用基盤（Backbone Node）

**役割**: システムの「記憶」と「監視」を担う、高負荷・高信頼性が求められる層です。

* **同居させるコンポーネント**:
* **Redis** (Event Bus + 状態管理DB)
* **PostgreSQL** (LangGraph内部記憶/Checkpointer)
* **Langfuse** (Observability)


* **集約の理由**:
* RedisとDBはディスクI/Oやメモリ管理のチューニングが重要であり、一箇所にまとめてリソースを監視しやすい。
* Langfuseも内部でDBを持つため、DB周辺の管理を基盤チームがこのサーバー一台で完結できる。


* **推奨スペック**: メモリ最適化 Eシリーズ (例: E4s_v5)

### 3. サーバーC：AI実行・サンドボックス（Compute Node）

**役割**: AIが「思考」し「コードを実行」する、最も重い処理を行う層です。

* **同居させるコンポーネント**:
* **LangGraph** (AIエージェント実行環境)
* **Dify** (WF型実行環境)
* **Firecracker / gVisor** (サンドボックス)


* **集約の理由**:
* **Nested Virtualization（入れ子状の仮想化）**が必須。Firecrackerを動かすための特殊なVM設定（KVM有効化）をこの一台に集中させる。
* エージェント（脳）と実行環境（サンドボックス）が同じネットワーク内（または同ホスト内）にあることで、コード実行のレイテンシを最小化できる。


* **推奨スペック**: 高性能CPU Dv3/Dv4以上のNested Virt対応サイズ (例: D4s_v3)

---
### サーバーD：データアクセス・ツールノード（Data Access Node）

**役割**: 社内データへの「唯一の窓口」。実データへのアクセスとMCP化を担います。

* **同居させるコンポーネント**:
* **Denodo**（データ仮想化プラットフォーム）
* **MCPサーバー群**（Python/Node.js等で記述されたツール群）


* **集約の理由**:
* **リソース管理**: DenodoはJavaのHeapメモリを大量に消費します。サーバーB（Redis/Postgres）と同居させると、メモリの奪い合いでシステム全体が不安定になるリスクがあります。
* **セキュリティ（隔離）**: このサーバーだけが社内のZendesk、Redmine、DB等と直接通信できるようにネットワーク（NSG）を設定します。Application層（サーバーC）から直接データを触らせないことで、ガバナンスを効かせます。


* **推奨スペック**: メモリ最適化 **Eシリーズ (例: E8s_v5)** * ※Denodoの動作には最低でも16GB〜32GB程度のRAMを割り当てるのが理想的です。

---

## 考慮事項

1. **Firecracker用のVMサイズ選定**:
AzureでFirecracker（KVM）を動かすには、**「v3」以降の特定のインスタンス（DシリーズやEシリーズ）**でなければNested Virtualizationが使えません。これを選び忘れると、サンドボックス構築時に詰みます。
2. **Redisのメモリ保護**:
サーバーBでRedisとPostgres、Langfuseを同居させる場合、Redisがメモリを食いつぶして他のプロセスがOOM Killer（強制終了）されないよう、`maxmemory` 設定を厳格に行う必要があります。
3. **セキュリティグループ (NSG)**:
サーバー間通信（例：BFF ➔ Redis）は発生しますが、外部から直接サーバーBやCを叩く必要はありません。APIM ➔ サーバーA ➔ その他の順で、通信経路をホワイトリスト化してください。

## PoCサーバー構成案（ウルトラスリム:2サーバー集約モデル）

## 1. 全体コンセプト

このシステムアーキテクチャを用いてPoCを行う場合のコストを勘案した場合の構成。
運用保守の負荷を下げるため、**「管理と記憶のサーバー」**と、**「思考とデータのサーバー」**の2つに集約します。
高額な商用製品を避けつつ、Denodoを最小構成（4GB RAM）で同居させることで、データの統制力を維持します。

---

## 2. サーバー構成詳細

### サーバー1：入口・管理・ステート（Entry & State Node）/ 運用基盤

システムの「玄関」であり、ユーザーとの対話ログを「記憶」する役割を担います。

* **役割**: 認証、認可、分散トレースの発行、実況中継（SSE）の配信、履歴の永続化。
* **スペック案**: **Azure D4s_v5 (4 vCPU / 16GB RAM)**
* **同居コンポーネント**:
* **BFF (FastAPI)**: UIロジック、状態管理DBへの書き込み。
* **LiteLLM Proxy**: AI/MCPゲートウェイ、PIIマスキング。
* **Redis**: **Event Bus**（Streams）と **状態管理DB**（JSON）の同居、Langfuseのイベント連携用。
* **PostgreSQL**: LangGraphの内部記憶（Checkpointer）、Langfuseのデータ保存用。
* **Langfuse** (Observability)
* **APIM (Managed)**: 境界防御（※Azureマネージドサービス）。


### サーバー2：思考・データ・実行（Compute & Data Node）

AIが「考え」、Denodoを介して「データを取り」、必要に応じて「ツールを動かす」場所です。

* **役割**: エージェント実行、データ仮想化、外部API連携、コード実行。
* **スペック案**: **Azure D4s_v5 (4 vCPU / 16GB RAM)**
* **同居コンポーネント**:
* **LangGraph / Dify**: AIエージェントおよびワークフロー実行環境。
* **Denodo (VDP Server)**: **最小構成（Heap 4GB設定）**。
* **MCP Servers**: 各種システム（Zendesk等）への接続アダプタ。
* **Docker / gVisor**: コード実行用のサンドボックス環境。



---

## 3. サーバー2 のメモリ配分計画（重要）

Denodoを同居させるため、Dockerの `mem_limit` 設定を用いて厳格にリソースを配分します。

| コンポーネント | 割り当てメモリ | 備考 |
| --- | --- | --- |
| **Denodo (VDP)** | **4.0 GB** | JVMヒープサイズを4GBに固定。 |
| **LangGraph / Dify** | **4.0 GB** | 同時並行ジョブ数に応じて調整。 |
| **Docker (Sandbox)** | **2.0 GB** | コード実行時のみ消費。 |
| **MCP Servers** | **1.0 GB** | 軽量なPython/Nodeプロセス。 |
| **OS / Overhead** | **2.0 GB** | カーネル、監視エージェント等。 |
| **(バッファ/空き)** | **3.0 GB** | 突発的な負荷への備え。 |
| **合計** | **16.0 GB** |  |

## 5. 実装フェーズの優先順

PoC の大枠の進め方は本書の「8. PoC環境構築ロードマップ（5ステップ）」を主とし、実装観点では次のフェーズを目安に進める。

1. Phase 1 (Infrastructure): LiteLLM Proxy と Langfuse のセットアップ、および HTTP ベースの MCP 通信路（OAuth 統合）の確立。
2. Phase 2 (Logic): Dify の 3 分割ワークフローを用いた、読み取り専用ツールの非同期 RAG 実装。
3. Phase 3 (Write/Auth): 非同期 HITL とトークン再注入を用いた、安全な書き込み系ツールの実装。
4. Phase 4 (Autonomy): ラッパーAPIによるサンドボックス制御と、Codex / Claude Code の統合（コールドスタート対策含む）。
