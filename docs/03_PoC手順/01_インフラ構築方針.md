# インフラ構築方針

## 概要
生成AI基盤を構成する各ミドルウェアを提供する。

## 方針
* 各ミドルウェアはdockerコンテナで構築して、dockerのネットワークで接続
* HTTP/HTTPSのインターネット接続はSquidコンテナを経由させる。
* HTTP/HTTPS以外のインターネット接続は拒否するようdockerのネットワーク設定を行う。

## 現状（PoC・infra配下の実装）
* `infra/00-network/` に Squid コンテナを配置している。
   * 以前は `ai_platform_net` の作成を目的としたネットワーク用 compose だったが、現状は「出口（egress）Squid」を意識したネットワーク構成に移行中。
   * `ai_platform_internal`（`internal: true`）と `ai_platform_egress`（通常bridge）を作り、Squid を両ネットワークに接続している。
* 主要ミドルウェアはディレクトリ単位で docker compose を分割している。
   * 移行前の構成: `ai_platform_net`（external）へ接続（例: Postgres/Redis/Langfuse/Dify/Zitadel）
   * 移行中の構成: `ai_platform_internal`（external）へ順次移行中。
   * 移行済み: Postgres（`infra/01-postgresql/`）、Redis（`infra/02-redis/`）、Langfuse（`infra/03-langfuse/`）、Dify（`infra/04-dify/`）、LiteLLM（`infra/05-litellm/`）、Zitadel（`infra/91-zitadel/`）
* Dify は `ssrf_proxy_network`（`internal: true`）と Squid（`ssrf_proxy`）を用意し、一部の外向き通信をプロキシ前提で扱っている。
* `infra/06-denodo/` は docker compose によるミドルウェア提供ではなく、SDK/設定資材（`denodo-ai-sdk`）中心の配置になっている。

#### 現状との乖離（暫定・インフラ）
1. Squid 経由が「用意」されているが、「強制」されていない

   * 方針/期待: HTTP/HTTPS のインターネット接続は Squid コンテナを経由させる。
   * 現状:
      * `infra/00-network/` で `ai_platform_internal`（`internal: true`）+ `ai_platform_egress` を作り、Squid を両方へ接続する構成に移行中。
      * `ai_platform_internal`（`internal: true`）にのみ接続しているコンテナは **直接インターネットへ出られない**ため、外向き通信が必要な場合は Squid（`squid:3128`）に向けた proxy 設定が必須になる。
      * ただし proxy 設定（`HTTP_PROXY`/`HTTPS_PROXY`/`NO_PROXY` 等）の注入はサービス横断では統一されておらず、現状は一部（例: LiteLLM の `infra/05-litellm/.env`、Denodo SDK の `infra/06-denodo/denodo-ai-sdk/env_compose`）に限って定義されている。
      * Dify はアプリ設定として `SSRF_PROXY_HTTP_URL`/`SSRF_PROXY_HTTPS_URL` を `ssrf_proxy` に向けているが、`ssrf_proxy` 自体が `ai_platform_egress` に接続していないため（`ai_platform_internal` 側のみ）、現状の構成のままだと **Dify の外向き通信（例: Marketplace）を成立させにくい**。
      * Zitadel は `ai_platform_internal` に加えて専用ネットワーク `zitadel` にも接続しており、ネットワーク境界を厳格にする場合は `zitadel` 側も `internal: true` とする等の整理が必要になり得る（複数ネットワーク参加時のデフォルトルート/egress経路に注意）。
   * 影響: “HTTP/HTTPS は必ず Squid 経由”という前提での監査・制御（到達先制限/ログ一元化）が成立しにくい。

2. 「HTTP/HTTPS 以外のインターネット接続拒否」を Docker ネットワーク設定だけで担保できていない

   * 方針/期待: HTTP/HTTPS 以外のインターネット接続は拒否する（ネットワーク設定で担保）。
   * 現状:
      * `ai_platform_internal`（`internal: true`）に閉じている限り、HTTP/HTTPS 以外に限らず **そもそもインターネット到達ができない**（= 方針より強い制約になっている）。
      * 一方で、サービスが **internal ではない別ネットワーク** にも接続している場合（例: Zitadel の `zitadel` ネットワーク）、そのネットワークが外向き経路になり得るため、結果として HTTP/HTTPS 以外の到達も含めた egress が成立し得る。
      * コンテナ側/ホスト側の firewall（iptables/nftables）等で “非HTTP を遮断しつつ HTTP/HTTPS は proxy のみ許可” を横断的に担保する仕組みは、リポジトリ上の資材としては未整備/未検証（例: Denodo SDK コンテナは `NET_ADMIN` を付与しているが、遮断ポリシーの定義は compose からは読み取れない）。
   * 影響: HTTP/HTTPS 以外のプロトコル（任意ポート宛）で外部へ到達し得る余地が残り、方針のセキュリティ境界が曖昧になる。

3. Squid のアクセス制御が「許可リスト運用」の意図と整合していない

   * 方針/期待: Squid 経由とする前提で、必要最小限の到達先（許可リスト）に絞る。
   * 現状:
      * `infra/00-network/squid.conf` は許可リスト（例: `.opencode.ai`）を定義しているが、末尾が `http_access allow all` となっており実質的に広い許可になっている。
      * 一方で、Dify の `ssrf_proxy`（`infra/04-dify/**/ssrf_proxy/squid.conf.template`）は allowlist（例: `.marketplace.dify.ai`）+ `http_access deny all` を含み、到達先制限の意図が明確。
   * 影響: “Squid を経由していること”だけでは到達先制限にならず、制御点として弱い。

4. 「各ミドルウェアは docker コンテナで構築」という粒度が揺れている

   * 方針/期待: 生成AI基盤を構成する各ミドルウェアを docker コンテナで提供する。
   * 現状: `infra/06-denodo/` は “ミドルウェア提供” というより SDK/サンプル実行（`denodo-ai-sdk`）中心で、他ミドルウェア（Postgres/Redis 等）と同じ粒度の「基盤ミドルウェア」としては扱いにくい。
   * 影響: “インフラは compose 群で再現できる”という前提が崩れ、環境再現性・運用観点の整理が難しくなる。

5. ネットワークが複線化しており、意図しない egress 経路/到達性が生まれ得る

   * 方針/期待: 各ミドルウェアは同一ネットワーク上で相互に接続できる。
   * 現状: 主要ミドルウェアは `ai_platform_internal`（external）に揃ってきている一方で、Dify の `ssrf_proxy_network` など、目的別に **複数ネットワークを併用**している構成が残っている。
      * この場合、名前解決/到達性の“分断”というより、どのネットワークが egress になっているか（デフォルトルート）や、internal 前提を崩す経路が紛れ込んでいないかの確認が重要になる。
   * 影響: internal 前提の egress 制御や到達先制限が、例外ネットワーク経由で崩れやすく、調査/監査の難易度が上がる。

#### 共有 PostgreSQL の利用方針（Zitadel）
* Zitadel は `infra/91-zitadel/` 内で DB コンテナを内包せず、`infra/01-postgresql/` で起動する PostgreSQL コンテナ（`postgres:5432`）を `ai_platform_internal` 経由で利用する。
* `docker compose up` の実行順は `infra/01-postgresql/` を先、`infra/91-zitadel/` を後とする。compose をまたぐ `depends_on` は使えないため、起動順は運用手順で担保する。
* 初回起動時の DB 作成は Zitadel の `start-from-init` に委ねる前提とし、共有 PostgreSQL 側に専用の init スクリプトを置く場合は別タスクで管理する。

## 対応方針（案・後続検討）
* 最小: 共有 Squid（`infra/00-network/`）のアクセス制御を許可リスト前提に整理し、各ミドルウェアからプロキシ設定（`HTTP_PROXY`/`HTTPS_PROXY`/`NO_PROXY` 等）を注入する。
* 方針準拠を強める: “外向き通信の出口”を Squid（または egress ゲートウェイ）に集約するネットワーク設計（internal ネットワーク＋出口専用サービス）へ寄せる。
* 非HTTP の遮断: Docker ネットワークだけでの担保が難しい場合は、コンテナ側の firewall（iptables）またはホスト側の egress 制御を併用する（PoC段階の暫定は許容範囲を別途定義）。

### 方針準拠を強める（具体案: internal ネットワーク + 出口 Squid）
狙いは「各ミドルウェアは外へ直接出られない」状態をネットワークで作り、HTTP/HTTPS の外向き通信は Squid だけが実施できるようにすること。

#### 1) ネットワークを2枚に分ける
* **内部ネットワーク（internal）**: 各ミドルウェアはここだけに接続する（= 直接インターネットへ出られない）
* **外向きネットワーク（egress）**: Squid だけが接続し、インターネットへ出る

例（`infra/00-network/docker-compose.yml` のイメージ）:

```yaml
services:
   squid:
      image: ubuntu/squid:latest
      volumes:
         - ./squid.conf:/etc/squid/squid.conf:ro
      # NOTE: 社内LANからSquidを叩く必要がないなら ports 公開は不要。
      # ports:
      #   - "3128:3128"
      networks:
         - ai_platform_internal
         - ai_platform_egress

networks:
   ai_platform_internal:
      name: ai_platform_internal
      driver: bridge
      internal: true

   ai_platform_egress:
      name: ai_platform_egress
      driver: bridge
```

#### 2) 各ミドルウェアは internal のみに接続する
各 compose（例: Postgres/Redis/Langfuse/LiteLLM/Zitadel 等）は、`default` ネットワークを `ai_platform_internal` に寄せ、**egress 側には接続しない**。

例（各 compose 側）:

```yaml
networks:
   default:
      name: ai_platform_internal
      external: true
```

ポイント:
* `ports:` の公開（ホスト→コンテナ）は internal ネットワークでも可能なため、UI/API公開のために egress ネットワークへ参加させる必要は基本的にない。
* 「ミドルウェア間の通信」は internal ネットワーク内のサービス名解決で行う。

#### 3) HTTP/HTTPS は Squid へ（アプリ設定 or Proxy環境変数）
internal ネットワーク上の Squid を参照して、各コンテナに以下を注入する（必要なものだけ）。

* `HTTP_PROXY=http://squid:3128`
* `HTTPS_PROXY=http://squid:3128`
* `NO_PROXY=localhost,127.0.0.1,postgres,redis,langfuse-web,clickhouse,minio,litellm,zitadel` （例）

補足:
* Dify のようにアプリがプロキシURLを個別に持つ場合（`SSRF_PROXY_HTTP_URL` 等）は、それを Squid に向ける。
* すべてのミドルウェアが proxy 変数に従うわけではない（アプリ/SDK依存）ため、方針の「強制」は **internal ネットワークで直接外へ出られない**ことが主担保になる。

#### 4) Squid 側で「許可する通信」を絞る
Squid の設定で以下を最低限行う。

* `Safe_ports` を 80/443 に絞る（必要最小限）
* `CONNECT` を 443 のみに絞る
* `http_access deny all` を最後に置く
* 到達先ドメインを allowlist 化する（PoCはまず “必要最小限＋追加は都度”）

※ `infra/00-network/squid.conf` は現状 `http_access allow all` があるため、許可リスト運用にする場合はここを整理する。

#### 5) 運用面の注意（PoCで詰まりやすい点）
* パッケージ取得（`apt/pip/npm`）や、アプリ自身の更新チェック等がある場合、proxy 未設定だと外部通信が失敗する。
   * その場合は「必要なコンテナにだけ proxy 環境変数を付ける」か、「取得元ドメインを allowlist に追加」する。
* `internal` ネットワークへの移行は破壊的になり得るため、切替手順（compose down → ネットワーク作成 → compose up）をPoC手順側に追記するのが望ましい。
