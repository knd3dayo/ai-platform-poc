# アプリケーション層の実現方式

---

## 0. 位置づけ

本書は、[02_AIエージェントの業務適用を見据えた生成AIアプリケーション層の検討.md](../01_アーキテクチャ検討/02_AIエージェントの業務適用を見据えた生成AIアプリケーション層の検討.md) で定義した Application層の責務を、どのような実装方式で成立させるかを整理する文書です。

Application層は生成AIシステムの「脳」として、業務目標の分解、制御フローの委譲、状態遷移、非同期HITL、段階的自律化を担います。本書では、WF型 / SV型 / 自律型の使い分け、Coordinator、状態永続化、ハーネスを中心に実現方式を整理します。

---

## 1. Application層の基本方針

### 1.1 予見性に応じて型を使い分ける

| 型 | 向いている業務 | 主な実装基盤 |
| --- | --- | --- |
| WF型 | 予見可能で固定フロー化しやすい処理 | Dify、固定構成の LangGraph |
| SV型 | 合議、評価、承認、例外判断を伴う処理 | LangGraph |
| 自律型 | 探索的で試行錯誤が必要な処理 | サンドボックス実行基盤 |

**実装原則**

* まず WF型で固定できる範囲を広く取ります。
* 次に、評価や判断が必要な難所だけを SV型へ委譲します。
* 自律型は最後に限定的に導入し、強いハーネスで囲います。

### 1.2 Coordinator を入口に置く

利用者要求を受けた最初のユニットで、次の判断を行います。

* Fast Track と Slow Track のどちらへ流すか
* WF型 / SV型 / 自律型のどれで扱うか
* Read-Only で足りるか、HITL を要求すべきか

Coordinator は業務要求を適切な型へ配分する責務を持ち、すべてを単一のエージェントへ押し込みません。

## 2. 非同期処理を前提にした実装

### 2.1 長時間処理はジョブ化する

AIエージェントは処理時間が読みにくく、人間承認も時間差で返ってくるため、同期応答待ちを前提にしません。

* 上位システムには即時に Job ID または `thread_id` を返す
* 実処理はバックグラウンドで継続する
* 完了、失敗、承認待ち、再開要求は Event Bus や Webhook で通知する

### 2.2 非同期HITLを標準パターンにする

SV型や高リスク処理では、承認待ちを例外ではなく標準フローとして扱います。

* 承認直前で `interrupt` を発火する
* Checkpointer に文脈全体をフリーズする
* 人間入力を受け取った後に `thread_id` をキーに再開する

### 2.3 UI向け状態と AI内部状態を分離する

* **状態管理DB** は UI が参照する現在状態を保持します
* **Checkpointer** は Application層内部の再開用状態を保持します
* **Event Bus** は状態変化の事実を疎結合に伝えます

この 3 つは役割が違うため、単一ストアに混ぜません。

## 3. 型ごとの実装要点

### 3.1 WF型

WF型は、入力検証、分岐条件、失敗時再実行性を明示した固定フローとして実装します。

* 定型の受付、分類、抽出、変換、通知に向きます
* GUI ベースの設計や運用可視化が必要なら Dify を優先します
* `trace_id` と出典メタデータを後続へ渡せる形で保持します

### 3.2 SV型

SV型は、複数ユニットの合議、評価、やり直し、非同期HITL を扱うための状態機械として実装します。

* Supervisor がワーカーへの分担と再試行を制御します
* 業務固有の評価ユニットと、人間の最終承認ポイントを分けて持ちます
* どこで止め、何を保存し、何を再開条件にするかを State 設計として明示します

### 3.3 自律型

自律型は、目標と境界条件のみを与え、思考と行動のループを許容する型です。

* 初期導入は Read-Only または限定権限から始めます
* サンドボックス、反復回数上限、時間上限、予算上限、強制停止を必須にします
* 作業途中の細粒度な HITL よりも、成果物レビューや PR ベースの承認を優先します

---

## 4. ハーネスエンジニアリング

Application層の本質は、AI に自由を与えることではなく、**どの情報を見て、どの状態を保持し、どこまで操作でき、どう止めるかを周辺環境で規定すること**です。

### 4.1 WF型に必要なハーネス

* 入力検証
* 分岐条件の明示
* 監査ログ
* 再実行可能な構造

### 4.2 SV型に必要なハーネス

* Checkpointer による状態永続化
* 非同期HITL
* コンテキスト圧縮
* 合議結果と評価理由の記録

### 4.3 自律型に必要なハーネス

* サンドボックス
* 権限分離
* 反復回数、時間、コストの上限
* 自動テストまたは成果物検証
* Kill Switch

### 4.4 型横断で必要なハーネス

* `trace_id` による共通追跡
* Fast / Slow Track の一貫した分岐
* 共有 State の受け渡し規約
* マルチエージェント時の handoff ポリシー

---

## 5. 補助コンポーネントとの責務分界

### 5.1 North境界との関係

* API Gateway は決定論的防御を担います
* BFF は `trace_id` の確定、Fast / Slow Track、状態管理DB との I/O、通知、Resume 受付を担います
* Application層は内部状態制御と状態変化通知を担います

### 5.2 Tool層との関係

* Application層は、どのツールをいつ使うかを判断します
* Tool層は、権限コンテキストを継承して外界へ実行します
* Tool層内部では人間を待たず、状態も持ちません

### 5.3 AIガバナンス層との関係

* 業務固有の品質評価は Application層が担います
* 企業共通の安全性、根拠性、共通評価、停止判断は AIガバナンス層が担います
* 高リスクな遷移では、Application層の承認待ちと AIガバナンス層の同期検証を組み合わせます

---

## 6. アンチパターン

* すべての要求を自律型へ流し、型の使い分けをしない
* BFF プロセス内に中断中の状態を抱え込む
* UI向け状態、AI内部状態、事実通知を単一ストアに混ぜる
* Tool層で HITL を実装し、Application層の責務を崩す
* `trace_id` が North境界、Event Bus、ワーカー間で分断される
