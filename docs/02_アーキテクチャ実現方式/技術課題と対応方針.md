# 生成AIアプリケーション基盤の技術課題と対応方針

## 0. この文書の位置づけ

本書は、`docs/01_アーキテクチャ検討` 配下の一次文書を基準に、生成AIアプリケーション基盤の主要な技術課題と対応方針を整理する文書です。

ここでの課題は、PoC の進捗メモではなく、**アーキテクチャとして先に定義しておくべき論点**を対象とします。そのため、従来の古い課題番号や PoC ベースのステータス分類は引き継がず、一次文書の責務定義に合わせて再編します。

本書では、課題が **どの層の課題か、または全体の課題か** が一目で分かるよう、カテゴリを明示します。

---

## 1. 課題の分類方針

本書のカテゴリは次の 5 区分です。

| カテゴリ | 意味 |
| --- | --- |
| 全体・横断 | 3 層をまたいで成立する共通課題 |
| Application層 | WF型 / SV型 / 自律型、状態管理、ハーネスに関する課題 |
| Tool層 | MCP、実行インターフェース、意味付け、Read / Write 制御に関する課題 |
| AIガバナンス層 | Guardrails、根拠性、評価、停止判断に関する課題 |
| 周辺実行基盤 | BFF、Event Bus、状態管理DB、Observability など補助コンポーネントの課題 |

**補足**

* API Gateway や BFF は 3 層そのものではなく、責務を成立させる周辺実行基盤として扱います。
* マルチエージェントの East-West 統制のように、単一層へ閉じない論点は「全体・横断」または「AIガバナンス層」を主カテゴリに置き、必要に応じて注記でまたがりを明示します。

課題 ID はカテゴリごとの接頭辞付き連番とし、次の規則で付番します。

* `O-xx`: 全体・横断
* `A-xx`: Application層
* `T-xx`: Tool層
* `G-xx`: AIガバナンス層
* `R-xx`: 周辺実行基盤

検証や実装を管理する際は、必要に応じて親課題の下にサブ課題 ID を切ります。

* サブ課題 ID は `親課題ID-yy` 形式とし、例として `A-03-01`、`G-01-02` のように付番します。
* 親課題はアーキテクチャ論点を表し、サブ課題は「個別に検証完了を判定できる単位」を表します。
* 親課題の検証完了は、必須サブ課題がすべて完了した時点、または不要なサブ課題が明示的に除外された時点で判定します。
* 技術課題検証文書は、親課題全体を扱ってもよいし、`A-03-01` のようなサブ課題単位で個別文書化してもよいものとします。

## 2.1 サブ課題分解方針

各親課題は、原則として次の観点に分解して扱います。

* 構造成立性: その責務分離や境界条件が構成として成立するか。
* 制御成立性: 想定したポリシー、停止条件、権限制御が機能するか。
* 運用成立性: 再開、監査、変更管理、段階導入が運用上破綻しないか。

サブ課題の粒度は、次の条件を満たすように揃えます。

* 1 つのサブ課題に対して、正常系と代表的な異常系の確認条件を定義できる。
* 実装担当または検証担当を割り当てられる。
* 他のサブ課題に依存していても、完了判定そのものは独立に記録できる。

## 3. 課題別の対応方針

### 3.1 全体・横断の課題

#### 課題 O-01: North境界と責務分離

**背景**

一次文書では、3 層アーキテクチャは責務による分離であり、API Gateway や BFF はその責務を成立させる補助コンポーネントとして扱われています。ここを曖昧にすると、入口防御、業務状態管理、意味論的統制が同じ場所に混ざります。

**対応方針**

* API Gateway は JWT 検証、WAF、レート制限などの決定論的防御に限定します。
* BFF は `trace_id` 確定、Fast / Slow Track 分岐、状態管理DB 書き込み主権、通知、Resume 受付を担います。
* AIガバナンス層は自然言語入出力、モデル利用、根拠性、共通評価、停止判断を担います。

**対応する既存検証文書**

* O-01-04: [A-03-01_Docker_Proxy環境によるセキュアなコーディングエージェント利用の検証.md](../11_技術課題検証/A-03-01_Docker_Proxy環境によるセキュアなコーディングエージェント利用の検証.md)

#### 課題 O-02: `trace_id` と Trajectory の一気通貫追跡

**背景**

AI システムの監査対象は個別ログではなく、思考、判断、ツール実行、承認待ち、再開から成る Trajectory です。

**対応方針**

* BFF で `trace_id` を確定します。
* Event Bus、Application層、Tool層、AIガバナンス層、Observability 基盤へ一貫して伝播します。
* `trace_id` を、再開や通知、評価結果検索に使う横断キーとして扱います。

**対応する既存検証文書**

* O-02-03: [A-01-02_スーパーバイザーのツール選択とMCP結果判断の検証.md](../11_技術課題検証/A-01-02_スーパーバイザーのツール選択とMCP結果判断の検証.md)

#### 課題 O-03: 認証と権限コンテキスト継承

**背景**

AI が固定の高権限で動く構成は、越権アクセスや監査不能を招きます。

**対応方針**

* 実行ユーザーの認証コンテキストを end-to-end で継承します。
* AIガバナンス層は Tool 利用の上限制御を行い、Tool層は本人権限に基づく最終認可を受けます。
* 共有 API キー中心ではなく、ユーザーコンテキスト中心の設計に寄せます。

#### 課題 O-04: マルチエージェントの East-West 統制

**背景**

単一エージェントの入口検査だけでは、agent-to-agent handoff、共有 State、再帰的合議による事故は防げません。

**対応方針**

* handoff 前に、委譲先権限、操作種別、予算、共有 State の完全性を検査します。
* 共有 State には State Scrubbing を適用し、不要な文脈や機密情報を削減します。
* セッション単位で反復回数と予算を監視し、再帰的ブロッキングを抑止します。

### 3.2 Application層の課題

#### 課題 A-01: WF型 / SV型 / 自律型の使い分け

**背景**

すべてを単一のエージェント構成で処理しようとすると、制御不能と過剰設計の両方を招きます。

**対応方針**

* 予見性の高い工程は WF型で固定します。
* 合議、評価、承認が必要な難所は SV型へ寄せます。
* 探索的工程だけを自律型へ委譲します。
* Coordinator により入口で型を選びます。

**対応する既存検証文書**

* A-01-02: [T-01-01_コーディングエージェントのMCPサーバー化検証.md](../11_技術課題検証/T-01-01_コーディングエージェントのMCPサーバー化検証.md)
* A-01-02: [A-01-02_スーパーバイザーのツール選択とMCP結果判断の検証.md](../11_技術課題検証/A-01-02_スーパーバイザーのツール選択とMCP結果判断の検証.md)

#### 課題 A-02: 非同期HITLと状態永続化

**背景**

人間承認はリアルタイムで返るとは限らず、同期 API の中で待機させる設計は破綻します。

**対応方針**

* 承認直前で `interrupt` を発火し、Checkpointer へ状態を保存します。
* Resume は `thread_id` をキーに実行します。
* UI 向け状態は状態管理DBに分離し、AI 内部状態と混同しません。

#### 課題 A-03: ハーネスエンジニアリングと段階的自律化

**背景**

自律型を安全に使うには、プロンプトよりも周辺環境の設計が重要です。

**対応方針**

* WF型には入力検証と再実行性、SV型には状態永続化と非同期HITL、自律型にはサンドボックスと停止条件を必須要件として持たせます。
* 段階的自律化を前提に、Read-Only、限定権限、成果物レビューから始めます。

**対応する既存検証文書**

* A-03-01: [A-03-01_Docker_Proxy環境によるセキュアなコーディングエージェント利用の検証.md](../11_技術課題検証/A-03-01_Docker_Proxy環境によるセキュアなコーディングエージェント利用の検証.md)

### 3.3 Tool層の課題

#### 課題 T-01: MCP による標準化された実行インターフェース

**背景**

個別 API の直結は、アプリごとのラッパー乱立と再利用性低下を招きます。

**対応方針**

* Tool 公開は MCP 等の標準プロトコルを基本とします。
* Application層はツール利用判断に集中し、実行仕様の詳細は Tool層へ閉じ込めます。

**対応する既存検証文書**

* T-01-01, T-01-03: [T-01-01_コーディングエージェントのMCPサーバー化検証.md](../11_技術課題検証/T-01-01_コーディングエージェントのMCPサーバー化検証.md)
* T-01-02: [A-01-02_スーパーバイザーのツール選択とMCP結果判断の検証.md](../11_技術課題検証/A-01-02_スーパーバイザーのツール選択とMCP結果判断の検証.md)

#### 課題 T-02: セマンティックレイヤーによる意味付け

**背景**

生データや生 API を直接 AI に見せると、解釈ミスやハルシネーションを引き起こします。

**対応方針**

* データ基盤側で意味付けしたビューや API を利用します。
* Tool 定義の schema や description に業務用語、制約、出典メタデータを含めます。
* 物理スキーマではなく、ビジネス概念で接続します。

#### 課題 T-03: ステートレス実行と非同期ジョブ化

**背景**

Tool層で状態や承認待ちを持つと、Application層との責務分界が崩れます。

**対応方針**

* Tool層は即時結果または Job ID を返します。
* 待機、再試行、承認待ち、通知は Application層で扱います。
* Tool層内部での HITL や業務状態永続化は禁止します。

#### 課題 T-04: Read / Write 分離と Dry Run

**背景**

更新系ツールを参照系と同列に扱うと、不可逆な副作用を安全に制御できません。

**対応方針**

* Read ツールと Write ツールを分離して公開します。
* 更新系ツールには Dry Run を標準装備し、差分と影響範囲を返します。
* 本実行は人間承認と AIガバナンス層の最終ゲートを前提にします。

### 3.4 AIガバナンス層の課題

#### 課題 G-01: 入出力 Guardrails とモデル利用統制

**背景**

自然言語の脅威やモデル利用の逸脱は、決定論的な入口防御だけでは扱えません。

**対応方針**

* 入出力 Guardrails を企業共通機構として持ちます。
* モデル利用経路、Tool 利用上限、予算を一元統制します。
* Fast Track と Slow Track で統制強度を切り替えます。

**対応する既存検証文書**

* G-01-01, G-01-02, G-01-03: [G-01-01_入出力Guardrailsとモデル利用統制の検証.md](../11_技術課題検証/G-01-01_入出力Guardrailsとモデル利用統制の検証.md)
* G-01-02: [A-03-01_Docker_Proxy環境によるセキュアなコーディングエージェント利用の検証.md](../11_技術課題検証/A-03-01_Docker_Proxy環境によるセキュアなコーディングエージェント利用の検証.md)

#### 課題 G-02: グラウンディング、サイテーション、共通評価

**背景**

説明責任を満たすには、AI に引用を出させるだけでは足りず、根拠付けと検証をシステムに組み込む必要があります。

**対応方針**

* 根拠データ取得と出典付与を生成プロセスに組み込みます。
* 同期検証で根拠不足のまま次ステップへ進ませないようにします。
* 事後評価で意味妥当性やポリシー適合性を継続的に点検します。

#### 課題 G-03: Risk-Adaptive 監督と Kill Switch

**背景**

自律型や長時間処理を人間が常時監視する前提はスケールしません。

**対応方針**

* 共通評価基盤や LLM-as-a-Judge を活用し、低スコア時のみ HITL へ接続します。
* 危険兆候時には Kill Switch や予算遮断で停止できるようにします。
* Judge は前段機構であり、最終責任は人間または企業ポリシー側に残します。

### 3.5 周辺実行基盤の課題

#### 課題 R-01: Event Bus、状態管理DB、Checkpointer、Observability の分離

**背景**

AI システムでは、状態、イベント、監査、再開が混ざりやすく、ひとつのストアへ押し込むと責務が崩れます。

**対応方針**

* Event Bus は状態変化の事実通知を担います。
* 状態管理DB は UI が参照する現在状態を担います。
* Checkpointer は Application層内部の再開用状態を担います。
* Observability 基盤は `trace_id` を軸に Trajectory を再構成します。

## 5. 補足論点

### 5.1 補足論点 T-05: Write系ツールのDry Run

LLMのハルシネーションや破壊的実行を防ぐための規約。

* **Dry Runの実装回避**：開発者にシミュレーション実装を強制しない。ツール呼び出しのペイロード（AIの意図）をインターセプトして人間に承認させるか、Terraformのように「Plan」と「Apply」のツールを物理的に分離する設計を推奨する。


### 5.2 補足論点 A-04: LangGraph Stateのスキーマ進化

コード更新による、フリーズ中のState復元エラーを防ぐ。

* **Version Tagging**：Stateに `_schema_version` を持たせ、構造変更時にインクリメントする。
* **State Migration**：旧バージョンを最新スキーマへ変換するアダプター関数を実装するか、メジャーアップデート時は旧ワーカーを並行稼働（Blue/Green）させて古いスレッドを消化する。

## 6. 開発・運用を担保する「ゲート機構」（CI/CD & Runtime）

規約を個人の努力に依存せず、システム的に強制・誘導する仕組み。

1. **ゴールデンパス（自動的に正しい実装へ誘導）**：
* `langfuse.get_prompt()`、`traceparent` 伝播、SSE接続をデフォルトで内包した標準SDK/テンプレート（Scaffold）を提供する。


2. **CI/CDゲート（未対応はデプロイ拒否）**：
* **静的解析**：禁止APIの直叩きや、プロンプトのハードコードを検知してFailさせる。
* **構造検査**：Pydanticの `description` 必須化、StateのJSONシリアライズ可能性、`_schema_version` の存在をテストで検査。


3. **実行時ゲート（最後の砦）**：
* `traceparent` が無いリクエストの拒否/強制採番、およびLiteLLM Proxyを経由しない不正なLLM通信のネットワーク遮断（Egress制御）。
