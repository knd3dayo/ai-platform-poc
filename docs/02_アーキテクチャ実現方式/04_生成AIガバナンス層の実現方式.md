# AIガバナンス層の実現方式

---

## 0. 位置づけ

本書は、[04_AIエージェントの業務適用を見据えた生成AIガバナンス層の検討.md](../01_アーキテクチャ検討/04_AIエージェントの業務適用を見据えた生成AIガバナンス層の検討.md) で定義した Why / What / Conceptual Architecture を、どのような実装方式で成立させるかを整理する文書です。

[01_アーキテクチャ検討](../01_アーキテクチャ検討) 配下の文書が背景、要求、責務分界を扱うのに対し、本書は AI ガバナンス層を成立させる構成要素、通信経路、責務分界、運用上の注意点を扱います。特に、検討文書 5.1 から 5.6 で整理した統制機構を、そのまま実装方式へ落とし込むことを主眼とします。

本書で採用する主要用語は、参考文献および公式ドキュメントに合わせて次の通りとします。

* **NeMo Guardrails**: NVIDIA NeMo Guardrails の略称として用いる。
* **LiteLLM Proxy**: LiteLLM Proxy をモデル利用の統制点として用いる。
* **Langfuse**: Observability / Evaluation platform として用いる。
* **Ragas**: 非同期の事後評価を担う評価フレームワークとして用いる。
* **traceparent / trace_id**: W3C Trace Context のヘッダ名を `traceparent`、内部相関 ID を `trace_id` として使い分ける。

## 1. 実装全体方針

### 1.1 北境界の責務分離

North 境界は単一のゲートウェイで閉じず、次の 3 段で責務分離します。

1. **API Gateway**: JWT 検証、WAF、レート制限、TLS 終端などの決定論的防御を担当する。
2. **BFF**: `traceparent` の正規化、`trace_id` の確定、Fast Track / Slow Track の振り分け、状態管理 DB との I/O、通知制御を担当する。
3. **AI ガバナンス層**: NeMo Guardrails、LiteLLM Proxy、Langfuse を中心に、自然言語入出力、モデル利用、Tool 上限制御、評価、追跡可能性を扱う。

この分担により、外縁の防御、業務状態のオーケストレーション、意味論的統制を独立して進化させられます。特に、BFF で確定した `trace_id` を AI ガバナンス層、Application 層、Tool 層、評価基盤へ一貫して伝播させることが前提になります。

### 1.2 AI ガバナンス層の基本構成

AI ガバナンス層は、単一製品へ責務を押し込まず、用途別のコンポーネントを統制点として接続する構成を基本とします。

* **NeMo Guardrails**: 入出力 Guardrails を実施する。
* **LiteLLM Proxy**: モデル利用経路、モデル選択、予算統制、Tool 上限制御、ログ起点を担う。
* **Langfuse**: トレース集約、評価結果格納、横断検索を担う。
* **Ragas**: 非同期の事後評価を担う。
* **Application 層 / Tool 層**: 業務ロジック、共有 State 整形、利用者本人権限の最終判定を担う。

### 1.3 基本リクエスト経路

5.1 入出力 Guardrails と 5.2 モデル選択・利用経路・予算統制の基本経路は次の通りです。

```text
人間または AI エージェント
	-> API Gateway
	-> BFF
	-> NeMo Guardrails
	-> LiteLLM Proxy
	-> LLM
```

AI ガバナンス層の中では、NeMo Guardrails が自然言語入出力の意味論的検査を担当し、LiteLLM Proxy がモデル経路、利用ポリシー、予算、ログ連携を担当します。North 境界以前の認証やレート制限は API Gateway と BFF が担い、NeMo Guardrails や LiteLLM Proxy に過剰な責務を持たせません。

## 2. 5.1 入出力 Guardrails の実現方式

### 2.1 主担当コンポーネント

入出力 Guardrails は、AI ガバナンス層の中では **NeMo Guardrails を主担当** とし、必要に応じて LiteLLM Proxy の Hook で補完します。

* **NeMo Guardrails**: 入力 rails、dialog rails、output rails、execution rails を用いて、プロンプトインジェクション兆候、許容トピック、PII、禁則表現、出力形式逸脱を検査する。
* **LiteLLM Proxy**: `async_pre_call_hook` などの仕組みで、PoC Phase 1 相当の軽量な遮断、メタデータ付与、ログ送出を補完する。

PoC では LiteLLM Proxy の Hook による軽量実装を先行し、本番では NeMo Guardrails を前段に配置して、より明示的な Guardrails 定義へ寄せていく二段構えを基本方針とします。

### 2.2 実装パターン

* **入力 Guardrails**: ユーザー入力、外部コンテンツ、agent-to-agent handoff 前のメッセージを NeMo Guardrails で検査する。
* **出力 Guardrails**: LLM 応答を NeMo Guardrails で検査し、必要に応じてマスキング、拒否、定型文への置換を行う。
* **補完的な軽量ガード**: LiteLLM Proxy の Hook で、PoC で確認済みの NG ワード、機密キーワード、メタデータ付与を実施する。

### 2.3 実装上の注意

NeMo Guardrails は Guardrails の主実装候補ですが、これだけで間接プロンプトインジェクションや外部送信を完全に防ぐものではありません。Egress 制御、サンドボックス、Tool 経路固定と組み合わせて初めて多層防御になります。

## 3. 5.2 モデル選択・利用経路・予算統制の実現方式

### 3.1 主担当コンポーネント

モデル選択・利用経路・予算統制は、**LiteLLM Proxy を主担当** とします。公式ドキュメント上も LiteLLM Proxy は LLM Gateway として、モデル抽象化、認証、Spend Tracking、Budgets を提供するため、この責務に最も適しています。

### 3.2 実装範囲

* **モデル選択**: 業務またはエージェント種別ごとに `model_list` と仮想キーを定義し、利用可能なモデルを固定する。
* **利用経路の固定**: すべての LLM 呼び出しを LiteLLM Proxy 経由へ統一し、アプリケーションからベンダー API を直接呼ばせない。
* **予算統制**: 仮想キー単位、セッション単位、`trace_id` 単位でトークン消費量とコストを記録し、上限超過時は遮断または HITL へ分岐させる。
* **ログ取得**: 成功・失敗コールバックを Langfuse と接続し、後段の評価基盤へ引き渡す。

### 3.3 実装イメージ

北境界全体では API Gateway と BFF が前段に存在しますが、モデル利用の統制経路は次のように整理します。

```text
人間または AI エージェント
	-> API Gateway
	-> BFF
	-> NeMo Guardrails
	-> LiteLLM Proxy
	-> 許可された LLM
```

この構成により、アプリケーションはどのベンダー LLM を使うかを直接知らず、LiteLLM Proxy の定義だけで差し替えや停止が可能になります。

### 3.4 予算超過時の扱い

予算統制は単なるコスト管理ではなく、カスケード障害や Denial of Wallet の抑止機構でもあります。上限超過時の基本動作は次の優先順位とします。

1. LiteLLM Proxy で追加呼び出しを遮断する。
2. BFF または Application 層へ予算超過イベントを返す。
3. 必要に応じて HITL または Kill Switch へ接続する。

具体的なしきい値や逆シーケンスは業務ごとに調整が必要ですが、上限判定そのものは LiteLLM Proxy 側へ寄せる方針を取ります。

## 4. 5.3 Tool 利用の上限制御の実現方式

### 4.1 基本方針

Tool 利用の上限制御は、**LiteLLM Proxy を AI ガバナンス層の上位統制点** とし、Tool 層が利用者本人権限を最終判定する二層構造とします。

* **AI ガバナンス層の責務**: どの業務、どのエージェント、どの仮想キーが、どの Tool 群・操作種別を使えるかという上限制御を定義する。
* **Tool 層の責務**: 利用者本人のトークンやバックエンド権限を用いて、実データや実操作に対する最終認可を行う。

### 4.2 実装パターン

* **LiteLLM Proxy**: `access_group_ids`、仮想キー、Allowed Tools、Hook により Tool 呼び出し可能範囲を制御する。
* **MCP / Tool 層**: Entra ID などの個人トークンで、バックエンド資源に対する最終認可を行う。
* **Dry Run / Plan-Apply 分離**: Write 系 Tool は Tool 層側で Dry Run を提供し、AI ガバナンス層は承認要否と停止条件を決める。

### 4.3 呼称の扱い

本書では LiteLLM Proxy を主語とし、別名としての AI エージェント Proxy や MCP Proxy は採用しません。理由は、現時点でそれらが製品名でもアーキテクチャ上の確定用語でもなく、責務が LiteLLM Proxy と MCP にまたがって曖昧になりやすいためです。

## 5. 5.4 グラウンディングとサイテーションの実現方式

### 5.1 前提

グラウンディングとサイテーションは、回答文面の装飾ではなく、説明責任を成立させるための実装要件です。Application 層が回答生成時に根拠データと出典メタデータを扱い、AI ガバナンス層が同期検証と事後評価を担う分担とします。

### 5.2 5.4.1 同期検証

同期検証の実現方式は、**現時点では要検討** とします。特に、Google Cloud の Vertex AI のグラウンディング機能以外で、どこまで製品機能として実現できるかは未確定です。

現時点の候補は次の 2 系統です。

* **候補 A: Vertex AI のグラウンディング機能を利用する方式**
	Vertex AI Search によるグラウンディング、Vertex AI RAG Engine によるグラウンディングなどを利用し、根拠とサイテーションを生成時点で付与した上で同期的に検証する。
* **候補 B: セマンティックレイヤーまたは Application 層で根拠付与し、軽量な同期検証ノードを別実装する方式**
	根拠データの取得と出典付与は Application 層またはセマンティックレイヤーで実施し、サイテーション有無、必須メタデータ、Tool 実行前提を Governance Node で決定論的に確認する。

現段階では、PoC および将来のベンダー差し替えを考慮し、特定製品へ断定せず、次の抽象責務だけを確定事項として置きます。

* Tool 実行前に、最低限の根拠有無を同期的に確認する。
* サイテーションや出典メタデータが欠落している場合は次ステップへ進ませない。
* 高レイテンシな意味評価は同期検証へ押し込まず、事後評価へ逃がす。

### 5.3 5.4.2 事後評価

事後評価は、**Ragas またはセマンティックレイヤー製品を利用した非同期評価** を基本方針とします。ここは同期検証より具体化しやすく、既存の PoC 方針とも整合します。

実装パターンは次の通りです。

```text
LiteLLM Proxy success_callbacks / failure_callbacks
	-> メッセージキュー
	-> Async Worker
	-> Ragas またはセマンティックレイヤー製品
	-> Langfuse
```

* **Ragas**: Faithfulness、Answer Accuracy、Response Groundedness、Tool Call Accuracy などを評価候補とする。
* **セマンティックレイヤー製品**: 根拠妥当性や意味整合性の評価を製品側で補える場合は併用候補とする。
* **Langfuse**: スコア、判定理由、`trace_id` を紐付けて保存する。

低スコア時は、アラート、HITL、ポリシー見直し、必要に応じた Kill Switch へ接続します。ただし、Ragas や Judge 系の評価は最終権威ではなく、運用判断へ接続する前段機構として扱います。

## 6. 5.5 エージェント間 Guardrails と State Scrubbing の実現方式

### 6.1 現時点の整理

エージェント間 Guardrails と State Scrubbing の実現方式は、**現時点では要検討** とします。ただし、未決なのは実装位置であり、統制観点そのものは明確です。

### 6.2 暫定アーキテクチャ

現時点の暫定方針は、agent-to-agent handoff の直前に **LiteLLM Proxy 相当の統制点または Governance Node** を配置し、同期検証と同種のチェックを通す構成です。

```text
Agent A
	-> handoff 前検査
	-> Governance Node または LiteLLM Proxy 相当の統制点
	-> Agent B
```

この統制点では、少なくとも次を確認します。

* 委譲先エージェントへ渡してよい権限コンテキストか。
* 共有 State に不要な機密情報、汚染された文脈、外部由来の危険な指示が残っていないか。
* セッション全体の予算や反復回数上限を超過していないか。
* handoff 理由と次アクションが整合しているか。

### 6.3 現時点での結論

したがって、現時点の表現としては **LiteLLM Proxy を中心にしつつ、同期検証ノードを handoff 前へ拡張する方式が暫定候補** と整理するのが妥当です。LiteLLM Proxy 単体で State Scrubbing を完結させるとまでは断定せず、Application 層側の State 整形と協調する前提を明記します。

## 7. 5.6 trace_id 中心の Observability の実現方式

### 7.1 基本方針

`trace_id` 中心の Observability は、個別コンポーネントごとの努力へ依存するのではなく、**BFF 起点の標準化された伝搬契約** として実装します。

* **BFF**: `traceparent` を受け取りまたは新規発行し、内部で使う `trace_id` を確定する。
* **AI ガバナンス層**: NeMo Guardrails、LiteLLM Proxy、評価基盤へ共通ヘッダとメタデータで伝搬する。
* **Application 層 / Tool 層**: `trace_id` を内部状態、ログ、コールバック、非同期イベントに紐付ける。
* **Langfuse**: すべてのトレースと評価結果の集約先とする。

### 7.2 推奨する伝搬契約

* **外部境界**: `traceparent` を正本とする。
* **内部処理**: `traceparent` から抽出した `trace_id` を共通相関 ID とする。
* **同期呼び出し**: HTTP ヘッダまたは SDK メタデータで伝搬する。
* **非同期呼び出し**: Event Bus のメッセージ属性、Webhook payload、状態管理 DB に `trace_id` を明示的に保持する。

これにより、「いろんなところで頑張って実装する」のではなく、「境界ごとに守るべき伝搬契約を固定する」設計へ寄せられます。

### 7.3 Langfuse を中心にした集約

Langfuse は公式に OpenTelemetry ベースの Observability を提供しており、LLM 呼び出しだけでなく非 LLM 呼び出し、セッション、エージェントグラフも表現できます。本構成では次を集約対象とします。

* BFF で確定した `trace_id`
* NeMo Guardrails による遮断や書き換えの結果
* LiteLLM Proxy によるモデル呼び出し、コスト、予算、Tool 呼び出しの記録
* Application 層の状態遷移、HITL、Pause / Resume の記録
* Tool 層および事後評価結果

### 7.4 注意点

* `trace_id` は防御そのものではなく、追跡・監査・改善の基盤である。
* Event Bus、DLQ、Webhook、Resume 経路で `trace_id` が分断されないようにする。
* 製品固有 ID は廃止するのではなく、`trace_id` と関連付けて保持する。

## 8. 運用管理とインシデント対応

### 8.1 管理者 UI に最低限必要な機能

* `trace_id` / `user_id` による横断検索
* HITL 滞留ダッシュボード
* DLQ 管理画面
* コストアラートと Kill Switch 接続

### 8.2 インシデント対応

* **Kill Switch**: 特定モデル、特定 Tool、特定業務の即時停止
* **フォレンジック**: `trace_id` による入力から評価結果までの一気通貫追跡
* **Policy as Code**: 承認、回帰テスト、ロールバックを含むポリシー変更管理

### 8.3 アンチパターン

* `trace_id` が BFF、Event Bus、ワーカー間で分断される
* HITL 待ちをログだけで追跡する
* Kill Switch や予算上限がない
* 評価を PoC で止めてしまう
* 管理者へ生ログだけを見せ、横断トレース UI を用意しない

## 9. PoC to Production

### 9.1 構築ロードマップ

* **Phase 1: Foundation**: LiteLLM Proxy によるモデル集約、Langfuse による完全ログ取得、軽量 Hook による最低限の Guardrails
* **Phase 2: Guardrails & Grounding**: NeMo Guardrails、同期検証ノード、Ragas、セマンティックレイヤー連携の拡張
* **Phase 3: Operations**: HITL、品質劣化アラート、Policy as Code、Kill Switch の運用定着

### 9.2 将来の変化への備え

プロキシや Guardrails 製品は実装本体ではなく制御点として維持し、将来より良い製品が出た場合でもアプリケーション側を大きく改修せずに切り替えられるようにします。特に 5.4.1 と 5.5 は未確定要素が残るため、製品名ではなく責務境界と伝搬契約を先に固定する方針を取ります。
