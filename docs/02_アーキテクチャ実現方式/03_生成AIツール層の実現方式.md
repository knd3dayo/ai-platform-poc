# ツール層の実現方式

---

## 0. 位置づけ

本書は、[03_AIエージェントの業務適用を見据えた生成AIツール層の検討.md](../01_アーキテクチャ検討/03_AIエージェントの業務適用を見据えた生成AIツール層の検討.md) で定義した Tool層の責務を、どのような実装方式で成立させるかを整理する文書です。

Tool層は Application層の不確実な推論と、外界への確実な実行を分離するための標準化された実行インターフェースです。本書では、MCP、権限コンテキスト継承、セマンティックレイヤー、非同期ジョブ、Read / Write 分離、Dry Run を中心に実現方式を整理します。

---

## 1. Tool層の基本方針

### 1.1 MCP を基本形とする

Tool層は、アプリケーションごとに API 呼び出し仕様を埋め込むのではなく、MCP 等の標準プロトコルを基本形とします。

* ツール公開の方法を標準化できる
* モデルや Application層を差し替えても再利用できる
* 権限、監査、制約を集約しやすい

### 1.2 Tool層はステートレスに保つ

Tool層の責務は実行であり、状態管理ではありません。

* 数秒で返せる参照系ツールは同期実行する
* 長時間処理は Job ID を返して裏で継続する
* HITL や業務状態の永続化は Application層へ委譲する

### 1.3 生 API や生データを直接 AI に見せない

* 物理テーブルや複雑な API 仕様はそのまま見せません
* セマンティックレイヤーやメタデータで意味付けしたインターフェースを返します
* 出典や権限属性も返却データに含めます

---

## 2. 権限コンテキスト継承

### 2.1 実行ユーザーの権限で動かす

Tool層では、固定 API キーを共有して AI に広い権限を与えるのではなく、実行ユーザーの認証コンテキストをバックエンドまで透過的に継承することを原則とします。

* OIDC / OAuth2 トークンのフォワーディングを基本とする
* Tool Proxy でヘッダ正規化やトークン変換を行う
* 最終的なデータ認可はバックエンドまたはデータ基盤へ委譲する

### 2.2 上位統制と最終認可を分離する

* AIガバナンス層は、どの Tool 群・操作種別を使えるかという上限制御を担います
* Tool層は、利用者本人がそのデータや操作にアクセス可能かの最終認可を担います

この二層構造により、業務ポリシーとデータ権限モデルを混同しません。

---

## 3. セマンティックレイヤーと意味付け

### 3.1 セマンティックレイヤーを前提にする

Tool層は、物理スキーマや社内固有 API をそのまま AI に公開するのではなく、ビジネス用語で意味付けされたビューや API を利用します。

* データ基盤側での抽象化を優先する
* Tool 定義の `description` や schema に業務用語と制約を書く
* 出典メタデータを結果に添付する

### 3.2 Tool が返すべき最小情報

* 主データ
* 出典メタデータ
* 権限属性
* 操作種別
* Dry Run か本実行かの区別

---

## 4. 非同期ジョブと実行パターン

### 4.1 長時間処理は Job ID で返す

重い検索、ファイル変換、外部システム連携は同期完了を目指しません。

* Tool層は即座に Job ID を返す
* Application層が進行監督、再試行、通知、承認待ちを扱う
* Tool層は受け付けたジョブの実行と結果返却に集中する

### 4.2 Read / Write を分離する

同じバックエンド向けでも、参照系と更新系は別ツールとして公開します。

* **Read ツール** は取得、検索、参照、要約用データ収集
* **Write ツール** は登録、更新、送信、削除、起票

これにより、上位層が危険度を明示的に扱えるようになります。

### 4.3 Dry Run を標準機能にする

更新系ツールでは、本実行の前に Dry Run を実行できる形を標準とします。

* 何が変わるか
* 対象件数はいくつか
* 差分や影響範囲は何か

これらを返し、人間承認や AIガバナンス層の最終ゲートへ渡せるようにします。

---

## 5. 監査と追跡可能性

### 5.1 `trace_id` を必ず通す

Tool層は、BFF で確定した `trace_id` を必ず受け取り、バックエンド呼び出しと監査ログへ引き継ぎます。

### 5.2 監査ログの最小要件

* `trace_id`
* `tool_name`
* `user_id` または代理元識別子
* `target_resource`
* `operation_type`
* `decision`
* `latency_ms`

### 5.3 監査の考え方

追跡対象は「どのツールを呼んだか」だけでは足りません。誰の代理で、どの根拠に基づき、Read / Write / Dry Run のどれを行ったかまで残します。

---

## 6. 他層との責務分界

### 6.1 Application層との境界

* Application層は、どのツールを使うか、どこで承認待ちに入るか、どこで再試行するかを決めます
* Tool層は、その判断を受けて外界への実行をステートレスに行います

### 6.2 AIガバナンス層との境界

* AIガバナンス層は、Tool 利用の上限、入出力統制、共通評価、停止判断を担います
* Tool層は、利用者本人の最終認可、Dry Run、対象システム固有の実行制約を担います

### 6.3 North境界との境界

* BFF は `trace_id` と認証コンテキストを受け渡します
* Tool層はそれを消費し、加工せずに下流へ伝播させます

---

## 7. アンチパターン

* Tool層で人間承認待ちや会話状態を持つ
* 固定 API キーだけで広い権限を持たせる
* 生 SQL、生テーブル、生 API 仕様を直接 AI に公開する
* Read と Write を同一ツールに混在させる
* Dry Run を持たないまま更新系ツールを本番利用する
