# ai-chat-util チーム向け スーパーバイザー検証依頼メモ

## 背景

ai-platform-poc 側では、スーパーバイザーが通常 MCP ツールとコーディングエージェントを適切に使い分け、必要に応じて追加照会や Human In the Loop へ進める構成を目指している。

この観点を整理した検証用ドキュメントとして、以下を作成した。

- `/home/user/source/repos/ai-platform-poc/docs/11_検証/03_スーパーバイザーのツール選択とMCP結果判断の検証.md`

## 依頼したい確認事項

1. LangChain の既存 tool calling だけで、利用可能ツール一覧から安定して適切なツール選択ができるか
2. 既存機能だけで不足する場合、ツール選択専用の追加プロンプトや構造化出力が必要か
3. MCP ツールの結果が不十分な場合に、追加照会へ進むか、ユーザー確認へ進むかを安定して切り分けられるか
4. HITL が必要なケースで、現行の ai-chat-util 実装としてどのレベルまで扱えるか
5. 上記判断過程をログや state から追跡できるか

## こちらで想定している観点

### ツール選択

- ツールの名称、説明、引数情報から適切な委譲先を選べること
- 単純な設定確認は通常ツールへ、複数ステップ調査は coding-agent へ分岐できること

### MCP 結果判断

- ツール結果だけで回答可能かを判断できること
- 情報不足時に、追加の MCP 呼び出しで埋めるべき不足と、ユーザー確認が必要な不足を切り分けられること

### HITL

- 曖昧さや業務判断が残る場合に、確認事項を整理してユーザーへ返せること
- 将来的に非同期 HITL を採る場合、interrupt / resume や state 管理の拡張余地があること

## 相談したい境界

以下に該当する場合は、ai-chat-util チーム側の対応が必要と考えている。

1. 既存 tool calling だけではツール選択の再現率が不足する
2. 結果評価と追加照会の分岐を prompt だけでは安定化できない
3. HITL の導入条件や中断再開の制御が、ai-chat-util 側の state 管理に依存する
4. ツール選択や結果判断の根拠を追えるログが不足している

## 共有するとよい情報

- 入力プロンプト
- 利用可能ツール一覧
- 実際に選ばれたツールと期待したツール
- 各ツールの返却結果
- スーパーバイザー最終回答
- 追加照会または HITL が必要と判断した理由

## 補足

正常系の MCP サーバー連携検証は別途完了しているため、今回の主眼は「接続できるか」ではなく「どう選び、どう判断するか」にある。
