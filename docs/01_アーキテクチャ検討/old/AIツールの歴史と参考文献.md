## 1. 生成AIツール層技術の進化の歴史（2023年〜2026年現在）

AIが外界のシステムを操作する「Tool層」のアーキテクチャは、プロンプトへの直書きから始まり、データ抽象化への苦悩を経て、最終的に独立した標準プロトコル（MCP）へと切り離される劇的な進化を遂げた。

**第1期：LLMの「手」の獲得とReActパラダイム（2023年初頭）**
初期のLLMは学習時点の知識しか持たず、ハルシネーションが頻発した。これを打破したのが、[**『ReAct』**](https://arxiv.org/abs/2210.03629)（2023）という画期的な手法である。思考とツール実行を繰り返すことで外部ツールの利用を可能にしたが、当時はプロンプト内にツールの関数定義を直接書き込むような、アプリとツールが密結合した脆い実装であった。


**第2期：複雑なタスクへの挑戦と「計画と実行」の分離（2023年末〜2024年）**
ReActのような「都度考えながら動く」手法はタスクが複雑化すると無限ループに陥りやすいことが判明し、[**『Plan-and-Solve』**](https://arxiv.org/abs/2305.04091)（2023）の手法が確立された。最初に全体計画（Plan）を立ててから個別のタスク（Tool実行）に移るアプローチであり、「脳（Application層）」と「手足（Tool層）」を論理的に分離する土壌が生まれた。

**第3期：生アクセスの限界と「抽象化（セマンティックレイヤー）」の要求（2024年〜2025年）**
LLMに「生のSQL」や「複雑なSaaS API」を直接ツールとして渡すアプローチが破綻し始めた時期である。AIは物理的なテーブル構造や社内固有のビジネスロジックを理解できず、誤ったクエリを連発した（※後述のBIRD論文やGorilla論文で実証）。
この反省から、AIに生データを触らせるのではなく、データ基盤側でビジネス用語に翻訳した [**「セマンティックレイヤー」**](https://docs.getdbt.com/docs/use-dbt-semantic-layer) を構築し、Toolはその「抽象化・意味づけされたビュー」のみを提供するというインターフェースの高度化が必須要件となった。


**第4期：プロトコルの標準化（MCP）と「文脈・実行」の統合（2025年末〜2026年）**
各企業が独自の抽象化APIを乱造する課題を解決するため、Anthropicらが主導した [**『Model Context Protocol (MCP)』**](https://modelcontextprotocol.io/) が業界標準として定着した。MCPは単なる「Tools（関数実行）」にとどまらず、第3期で求められた「Resources（意味づけされたデータコンテキスト）」をプロトコルレベルで統合した。これによりAIエージェントとデータソースは完全に分離され、ステートレス実行とゼロトラスト（ZTNA）による権限継承を内包する堅牢なインフラへと昇華した。


## 7. 歴史的背景と本システム・アーキテクチャの関係

第1章の進化史は、当システムのTool層が単なる「APIの寄せ集め」ではなく、最新の学術的・産業的パラダイムを反映した**「標準化されたステートレスな実行基盤」**であることを強く裏付けている。当アーキテクチャは以下の通り、公的標準および学術的要件をシステム構造として直接的に充足している。

**① セマンティックレイヤーの分離・委譲（第3期の教訓）**
LLMに直接生データを触らせた際のハルシネーションを防ぐため、「データの意味付けと抽象化」の責務をTool層の内部から切り離し、後方のデータ基盤（セマンティックレイヤー）へ委譲するアーキテクチャ（第4.2項・第6章）を標準としている。
* **準拠する要件・参考文献**: 『Can LLM Already Serve as A Database Interface?』, 『The Semantic Layer』

**② MCPの採用による置換容易性と再利用性の確保（第4期の体現）**
密結合から脱却し、MCPというオープン標準プロトコルを採用した。これにより、AIモデルの進化に依存せず、ツール資産を全社横断で使い回す「業務実行インターフェースの資産化（第3章）」を実現している。
* **準拠する要件・参考文献**: 『Model Context Protocol Specification』

**③ Plan & Solve 思想に基づく「ステートレス化」とHITLの排除（第2期の教訓）**
不確実な「状態管理（Application層）」と確実な「実行（Tool層）」は分離すべきである。本アーキテクチャは、Tool層に人間の承認待ちを持たせず、即座にJob IDを返すステートレス設計（第4.3項・第6章）を徹底することでタイムアウト障害を防いでいる。
* **準拠する要件・参考文献**: 『Plan-and-Solve Prompting』

**④ ゼロトラストに基づく権限プロキシ（第4期のセキュリティ要件）**
Tool（MCP）サーバー内に独自の固定APIキーを持たせず、ユーザーの認証トークンをバックエンドへそのまま貫通させる「User Context Forwarding（第4.1項）」を採用している。
* **準拠する要件・参考文献**: NIST 『SP 800-207: Zero Trust Architecture』

## 8. ツール層アーキテクチャに関する参考文献一覧

### I. AIのエージェント化と実行分離（Tool Use）の基礎理論
**1. ReAct: Synergizing Reasoning and Acting in Language Models (arXiv:2210.03629)**
* **概要**: LLMの推論とツール実行を統合する画期的な論文。外部情報を動的に取得するエージェントの基礎を築いた。
* **アーキテクチャとの関係**: Application層（脳）とTool層（手足）が対話しながらタスクを進めるという、本アーキテクチャ全体の動作インターフェースの裏付けとなる。

**2. Plan-and-Solve Prompting: Improving Zero-Shot Chain-of-Thought Reasoning (arXiv:2305.04091)**
* **概要**: 事前に全体計画（Plan）を立て、その後で実行（Solve）に移る手法の有効性を実証した論文。
* **アーキテクチャとの関係**: 本システムの「人間の承認待ち（HITL）や計画の保持はApplication層が行い、Tool層はただ指示されたタスクをステートレスに即時実行する」という明確な責務分界の理論的根拠。

### II. 生アクセスの限界と抽象化要件（第3期の根拠）
**3. Can LLM Already Serve as A Database Interface? A BIg Bench for Text-to-SQLs (arXiv:2305.03111)**
* **概要**: 学術用のDBでは高精度を出せるLLMが、複雑な実ビジネスのDB（BIRDベンチマーク）に生アクセスすると精度が40%台に急落することを証明した論文。
* **アーキテクチャとの関係**: Tool層から直接生DBを叩かせるアプローチを棄却し、間にセマンティックレイヤーを挟む本設計の強力な根拠となる。

**4. Gorilla: Large Language Model Connected with Massive APIs (arXiv:2305.15334)**
* **概要**: LLMに複雑で大量のAPIを直接操作させると、引数や仕様をでっち上げる「APIハルシネーション」が高確率で発生することを実証した論文。
* **アーキテクチャとの関係**: プロンプト内にAPIの仕様を直接書き込むアプローチの限界を示すものであり、安全な実行のためにMCPのような抽象化されたツールサーバーが必須であることの証明。

### III. インターフェースの抽象化と標準化プロトコル
**5. Model Context Protocol (MCP) Specification**
* **リンク**: [https://modelcontextprotocol.io/](https://modelcontextprotocol.io/)
* **概要**: Anthropicが主導して公開した、AIエージェントとデータソース/ツールを標準化された手法で接続するためのオープン標準プロトコル。
* **アーキテクチャとの関係**: Tool層が個別のAPI連携によるスパゲッティ化を避け、MCPサーバーとして独立・標準化されていることの最大の技術的根拠。

**6. The Semantic Layer: The Interface for LLMs to the Enterprise (dbt Labs Docs)**
* **リンク**: [https://docs.getdbt.com/docs/use-dbt-semantic-layer](https://docs.getdbt.com/docs/use-dbt-semantic-layer)
* **概要**: 複雑な物理データベース構造をビジネス用語に抽象化して提供するデータ基盤のベストプラクティス。
* **アーキテクチャとの関係**: 第4.2項の根拠。Tool層が「意味付けされたビュー（Resources）」のみを提供する設計の正当性を担保する。

### IV. エンタープライズ・セキュリティと権限統制
**7. NIST SP 800-207: Zero Trust Architecture**
* **リンク**: [https://csrc.nist.gov/publications/detail/sp/800-207/final](https://csrc.nist.gov/publications/detail/sp/800-207/final)
* **概要**: 米国国立標準技術研究所（NIST）が定めるゼロトラスト・アーキテクチャの標準定義。「ネットワークの境界ではなく、アイデンティティ（ユーザー権限）に基づくアクセス制御」を大原則とする。
* **アーキテクチャとの関係**: MCPサーバーがIdPを介してユーザー個人の認証トークンを中継し、バックエンドに権限判定を委ねるというセキュアなプロキシ設計のコンプライアンス要件。
