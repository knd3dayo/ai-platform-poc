# AIエージェントの業務適用を見据えた生成AIアプリケーション層の進化と設計背景

*(※ 本資料は、生成AI基盤における「脳（推論と計画）」を担うApplication層について、歴史的背景を踏まえた実装パターン、制御方式、および関連レイヤーとの境界を定義したものである。具体的な実現方式は別紙にて扱う。)*

---

## 1. AIエージェント技術の進化の歴史（2023年〜現在）

生成AIを業務システムに組み込むためのアーキテクチャは、単一の巨大プロンプトへの依存から始まり、役割分担（マルチエージェント）と環境整備（ハーネス）の重要性に気づくという劇的な進化を遂げた。

### 第1期：LLMの「脳」としての発見と基本構造の定義（2023年）
初期の大規模言語モデル（LLM）は「一度テキストを出力して終わり」という単発処理しかできなかった。この限界を突破したのが、OpenAIのLilian Weng氏によるブログ（[*LLM Powered Autonomous Agents*](https://lilianweng.github.io/posts/2023-06-23-agent/), 2023）や代表的サーベイ論文（[*A Survey on LLM based Autonomous Agents*](https://arxiv.org/abs/2308.11432), 2023）である。

これらは、LLMを「推論エンジン（脳）」とし、そこに**「Planning（計画）」「Memory（記憶）」「Action（行動）」**を結合すべきだと定義した。これが全エージェントの「基本解剖図」となった。

### 第2期：複雑性の克服と運用基盤の誕生（2023年末〜2024年）
**【潮流A：複雑性の克服と「マルチエージェント・パターン」の体系化】**
1つの巨大なプロンプトにすべてを任せる手法は精度低下を招き、制御不能になる問題が深刻化した。これに対し、エージェントを専門化して連携させる手法が普及した。Microsoftの [*AutoGen*](https://arxiv.org/abs/2308.08155)（2023）が合議手法を確立し、[*Agent Design Pattern Catalogue*](https://arxiv.org/abs/2405.10467)（2024）によって「ルーティング」や「オーケストレーター・ワーカー」など18の設計パターンが理論的に体系化された。


**【潮流B：実運用の壁と「スキャフォールディング（足場組み）」の誕生】**
自律型エージェントの実運用において、「文脈の忘却」や「無限ループ」といった破綻が露呈した。この時期、LLMを取り囲むシステム側の周辺構造**「Agent Scaffolding（エージェントの足場組み）」**が不可欠であるという認識が広がる。[*MemGPT*](https://arxiv.org/abs/2310.08560)（2023）によるOS的メモリ管理の提案や、LangChain等の同期的限界を突破するため、**状態（State）をDBに永続化し安全に再開できる Checkpointer を備えた LangGraph** が誕生した。

### 第3期：実運用化を支える「2つの潮流」（2025年〜2026年）
エージェントを本番業務に投入するために、以下の2つの概念が並列・相互補完的に発展した。

* **【潮流A：ハイブリッド構成の定着（パラダイムシフト）】**
  「すべてをAIに任せるのは非効率」という教訓から、コスト・速度・確実性を両立する[*Autonoma*](https://arxiv.org/abs/2603.19270)（2026）的な階層型アプローチ、すなわち**「定型処理は WF型、判断や合議を伴う難所は SV型、探索的タスクは自律型へ分けるハイブリッド構成」**がビジネスの最適解となった。
* **【潮流B：ハーネスエンジニアリングの確立（実装技術）】**
  
  スキャフォールディングを産業レベルで深化させた概念として、**「Harness Engineering」**が確立された。Mitchell Hashimoto氏の [*My AI Adoption Journey*](https://mitchellh.com/writing/my-ai-adoption-journey)（2025-2026）や、OpenAI、Anthropicのレポートが相次いで発表された。プロンプトの工夫ではなく、**実行のサンドボックス化や状態の永続化、自動テスト環境などの「環境（ハーネス）」を作り込むこと**で、長時間タスクの完遂を保証する技術が確立された。

## 2. 歴史的背景と本システム・アーキテクチャの関係

第1章の歴史的進化（LLM単体からマルチエージェントへ、そしてハーネスの確立へ）は、当システムがなぜ「ハイブリッド構成」と「強固な周辺環境（ハーネス）」をアーキテクチャの根幹に据えたのかという必然性を完全に裏付けている。

**① 「Application層」と「Tool層」の分離（第1期の教訓）**
推論（脳）と行動（手足）の分離原則に基づきレイヤーを分割し、Application層が勝手に外界を操作するリスクを抑えている（第7.1項）。

**② ハイブリッド構成の導入：潮流Aの体現（第3期の教訓）**
「すべてをAIに任せるのは非効率」という教訓から、制御フローの委譲度に応じて Application層を3つの型（WF/SV/自律型）に分類（第4.2項）し、Coordinatorを用いてこれらを適材適所で組み合わせる階層的アプローチを採用している。

**③ ハーネスへの「委譲（Offload）」：潮流Bの体現（第3期の教訓）**
SV型・自律型を本番稼働させるため、ハーネスエンジニアリング（第6章）をインフラレベルで組み込んでいる。記憶管理やコンテキスト圧縮を自作するのではなく、LangGraphの Checkpointer（状態永続化・非同期HITL）や、DeepAgents等の上位ライブラリが標準で備える仕組みへ意図的に責務を委譲することで、安定性と開発効率を両立している。

## 3. 目的と定義：Application層の責務

前述の歴史的経緯（完全自律一辺倒の限界と、ハイブリッド＆ハーネスの定着）を踏まえ、本システムのApplication層の設計方針を定義する。

* **目的**: 生成AIアプリケーションの構成を Application層 / AIガバナンス層 / Tool層で分けた上で、脳となる Application層 における実装パターン、制御方式、関連レイヤーとの境界を整理する。
* **基本方針**: 制御フローをすべてAIに委ねるのではなく、予見可能な工程は人間が固定し、変動の大きい工程だけをAIに部分委譲し、探索的タスクに限って高い自律性を許容する（WF / SV / 自律の3分類）。また、これらを業務で安全に運用するための「ハーネスエンジニアリング」を一体の設計原則として組み込む。

*(※ 非同期 HITL、長時間実行、Pause/Resume 等の具体的な実現方式は `../02_アーキテクチャ実現方式/04_アプリケーション層の実現方式.md` を正本とする。)*

## 4. Application 層の基本概念と 3 つの型

### 4.1 そもそも「AI エージェント」とは何か？
従来の生成AIが「テキストを生成して終わる」ものであったのに対し、AIエージェントは**「目標（Goal）を与えられ、システムや環境に自ら働きかけてタスクを完遂する主体」**である。業界標準に従い、本書では以下の3要素を持つシステムとして定義する。
1. **推論と計画**: 基盤モデルを「脳」とし、目標をタスク分解し自己反省を通じて次行動を決定する能力。
2. **記憶**: 過去の履歴や業務知識を参照する能力。
3. **行動とツール利用**: API等を通じて外界の情報を取得・操作する能力（※実際の外界接続はTool層へ委譲）。

### 4.2 制御フローの委譲度に基づく 3 つの分類

Anthropicが提唱する連続的なスペクトラムの考え方に沿い、「制御フロー（次に何をするか）を誰が握るか」という軸で以下の3つの型に集約する。

* **WF型（ワークフロー型） / 主な基盤：Dify, LangGraph**
  人間があらかじめ定めた制御フローで実現する。GUIベースの協業を重視する場合は **Dify**、シームレスな状態共有や厳密なバージョン管理を重視する場合は制御フローをハードコードした **LangGraph** を採用する。必要なハーネスは軽量（入力検証、分岐明示、監査ログ等）である。
* **SV型（スーパーバイザー型） / 主な基盤：LangGraph（独自実装）, DeepAgents 等**
  人間が大枠の状態遷移（承認フロー等）を設計しつつ、タスク分担や合議をAIに部分委譲する。厳密なObservability（トレースID伝播等）が求められる場合は**生のLangGraphで独自ハーネスを実装**し、開発効率を優先する場合は仮想FS等を標準搭載した**DeepAgents等を活用**（固定TODO化）する。状態永続化や非同期HITLが必須ハーネスとなる。
* **自律型（エージェント型） / 主な基盤：LangGraph（独自実装）, DeepAgents, Claude Code 等**
  目標と境界条件のみを与え、AI自身に思考・行動のループを完全委譲する。SV型と同じ基盤を用いつつ、動的計画やサブエージェント生成をフルオート化する。権限隔離、サンドボックス、停止条件などの最も強固なハーネスが大前提となる。

**【ハイブリッド構成による実務適用】**
実務においては、受付・前処理・反映などの大枠はWFで固定し、難易度の高い推論・合議工程のみをSVで動的に扱う「階層的ハイブリッド型」が扱いやすい。要求の入口に**「Coordinator（入口ユニット）」**を配置し、「どこまで固定フローで扱い、どこから動的制御へ委譲するか」をルーティングする。

### 4.3 設計の鉄則: 予見性に基づくモデル選択
不必要に動的なモデルを採用しないことが、コスト、精度、制御性の面で重要である。予見性が高いほどWF型、予見性が下がるほどSV型・自律型を検討する。

| 予見性のレベル | 制御主体 | 決定タイミング | 推奨型 | 必要なハーネス要素 | 典型基盤 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **完全にタスク分解可能** | 人間 | 設計時 | **WF型** | 入力検証、分岐条件明示、監査ログ | Dify, 固定のLangGraph |
| **ある程度の実行計画あり** | 人間＋AI | 状態遷移は設計時、分担は実行時 | **SV型** | 状態永続化、非同期HITL、コンテキスト圧縮 | LangGraph, DeepAgents |
| **計画不能で試行錯誤必須** | AI | 実行時 | **自律型** | サンドボックス、権限隔離、停止条件、自動テスト | サンドボックス実行基盤 |

## 5. 各型の概要

### 5.1 WF型: 機能ユニットの連鎖で組む
入力・処理・出力を比較的固定したステップでつなぐ方式。Anthropicが示す Prompt Chaining や Routing 等のパターンを包含する。分岐を含んでも経路自体は人間が定義する。
* **機能ユニットの例**: 抽出ユニット、分類ユニット、変換ユニット、判定ユニット。

### 5.2 SV型: 役割を持つユニットの合議と人間の監督で組む

正解が一つではなく責任が重い領域で、自律型の予測不能さを抑えつつ安全に適用する方式。Orchestrator-Workers や Evaluator-Optimizer パターンに近い。AIのSupervisor（タスク分担・再試行）と、人間のSupervisor（承認・停止判断）を分けて捉える。
* **構成ユニットの例**: 統合ユニット、評価ユニット、業務ユニット（※意味のある節目に非同期HITLを組み込む）。

### 5.3 自律型: AI自身の思考ループで組む
手順が固定できず探索的に解決する必要がある領域に適用する。未知のツール呼び出しをTry & Errorで進めるため、不可逆操作を伴う領域では、Read-Onlyやサンドボックス環境から始める「段階的自律化」を採用する。

## 6. 各型とハイブリッド構成を支える「ハーネス」

ハーネスとは、エージェントがどの情報を見て、どの状態を保持し、どこまで操作でき、どう検証されるかを規定する周辺環境の総体（状態永続化、サンドボックス、コンテキスト圧縮、非同期HITL等）である。AIが誤るたびに人間が補正するのではなく、**環境側の改善によって誤りを再発しにくい構造へ変えていくこと（ハーネスエンジニアリング）**が本稼働の鍵となる。

### 6.1 WF型に必要なハーネス
目的はAIに自由度を与えることではなく、人間が設計した経路を安定して再現し追跡性を確保すること。入力検証、分岐条件の明示、監査ログ、失敗時の再実行性が中心となる。

### 6.2 SV型に必要なハーネス
制御フローの一部を委譲するため、**状態永続化、Pause / Resume、非同期HITL、コンテキスト圧縮**が必須となる。Supervisorが合議と再試行を繰り返すため、途中状態を安全に保持できなければ業務運用に耐えない。

### 6.3 自律型に必要なハーネス

最も高い柔軟性を持つ反面、最大の逸脱リスクを持つため、**権限分離、ファイルシステムやネットワークの隔離、反復回数/時間/コストの上限、強制停止条件、自動テスト**といった最も強固な安全帯（サンドボックス）を構築する。

### 6.4 ハイブリッドマルチエージェント全体を支えるハーネス
単一セッションに依存しない状態引継ぎ、共通トレースID、承認ポイントの一貫性など、型をまたぐ横断要件を担う。特にCoordinatorによる適切なルーティングと、長時間タスクにおける構造化された状態（State）の受け渡しが接着層として機能する。

## 7. 関連レイヤーとの境界と参照先

### 7.1 Tool 層との境界
DB、API、ファイル等の外界アクセスはTool層の責務であり、Application層はそれらをステートレスなツールとして扱う。Application層は Reasoning / Planning / 業務状態制御 を担い、Tool層は Action / External Access を担う。

### 7.2 評価の責務分界
Application層内の評価ユニットは「業務フローに閉じた品質保証」を担う。企業共通の安全性や根拠性の評価、低スコア時の強制停止判断はAIガバナンス層の責務とする（※詳細はガバナンス層定義書 6.3章を参照）。

### 7.3 組織的な責任分界
Application層の文脈では、「開発部門」が業務ユニットと業務固有評価の精度を担保する。IT/ガバナンス部門の監視統制や、ユーザーの最終決裁責任についてはAIガバナンス層側で包括的に定義する。
