# AIエージェントの業務適用を見据えた生成AIアプリケーション層の検討

---

## 1. 目的

生成AIアプリケーションの構成を Application 層 / AI ガバナンス層 / Tool 層で分けて考える。本書では、特に Application 層における実装パターン、制御方式、関連レイヤーとの境界を整理する。

本書で扱う WF 型 / SV 型 / 自律型の 3 分類は、単なる実装パターンの紹介ではない。2023 年以降の AI エージェント技術の発展において、単一の巨大なプロンプトや完全自律一辺倒の構成では、複雑な業務に対して精度、コスト、制御性の面で限界があることが明らかになった。その結果、定型処理は WF、判断や合議を伴う難所は SV、探索的タスクは自律型へ分けるマルチエージェント構成と、それを安全に動かすハーネスエンジニアリングが実運用の前提として定着した。本書は、この歴史的経緯を踏まえて本システムの Application 層における採用方針を整理する。

非同期 HITL、長時間実行、Pause / Resume、UI 復元、イベント設計などの実現方式は、[../02_アーキテクチャ実現方式/04_アプリケーション層の実現方式.md](../02_アーキテクチャ実現方式/04_アプリケーション層の実現方式.md) を正本として扱う。

## 2. Application 層の基本概念と 3 つの型（WF / SV / 自律）

生成AIを業務システムに組み込むにあたり、まずは「AIエージェント」の定義と、それを実装するためのアーキテクチャの分類を定義する。

ここでの 3 分類は、理論上あり得る型を並べたものではなく、業界の進化の中で収束してきた実務上の採用方針である。すなわち、制御フローをすべて AI に委ねるのではなく、予見可能な工程は人間が固定し、変動の大きい工程だけを AI に部分委譲し、探索的タスクに限って高い自律性を許容する。また、SV 型や自律型を業務で安全に運用するには、状態永続化、非同期 HITL、サンドボックス、停止条件といったハーネスが不可欠である。本章では、この 2 つを一体の設計原則として整理する。

### 2.1 そもそも「AI エージェント」とは何か？

従来の生成AI（チャットボット等）が「ユーザーの指示に対してテキストを生成して終わる」ものであったのに対し、AIエージェントは**「目標（Goal）を与えられ、システムや環境に自ら働きかけてタスクを完遂する主体」**である。

業界標準（OpenAIやNVIDIA等の定義）に従い、本書ではAIエージェントを以下の3要素を持つシステムとして定義する。


1. **推論と計画（Reasoning & Planning）**: 基盤モデル（FM/LLM）を「脳」として扱い、目標をタスクに分解し、自己反省（Reflection）を通じて次の行動を決定する能力。
2. **記憶（Memory）**: 過去の履歴や業務知識（ベクトルDB等）を参照する能力。
3. **行動とツール利用（Execution & Tool Use）**: APIや社内システムを通じて外界の情報を取得し、操作（Action）する能力。

### 2.2 制御フローの委譲度に基づく 3 つの分類

学術研究（『Agent Design Pattern Catalogue』等）では、基盤モデルをベースとしたエージェントのアーキテクチャとして18種類の設計パターンが提唱されている。
本書では、これら多様なパターンをエンタープライズ業務に適用するため、**「制御フロー（次に何をするか）を誰が握るか」**という軸で整理し、以下の代表的な3つの型に集約する。


これは、Anthropicが提唱する「Workflow（開発者が経路を事前定義する）」から「Agent（LLM自身が経路を動的決定する）」に至る連続的なスペクトラムの考え方とも完全に整合する。それぞれの定義と本システムにおける実装基盤の選定方針は以下の通りである。

* **WF型（ワークフロー型） / 主な基盤：Dify, LangGraph**
  人間があらかじめ定めた制御フローで、LLMやツールを連鎖させて実現する。予測可能な定型処理を安全かつ低コストに実行することに特化している。
  GUIベースで業務部門との協業を重視する場合は **Dify** 等が適する一方で、エンジニアリングの観点から「SV型・自律型とのシームレスな状態（State）の共有」や「コードベースでの厳密なバージョン管理」を重視する場合は、制御フローを決定論的に固定（ハードコード）した **LangGraph** をWF型の基盤として採用するアプローチも極めて有効である。必要なハーネスは比較的軽量であり、入力検証、分岐条件の明示、監査ログなど、人間が決めた経路を確実に再現するための足場が中心となる。

* **SV型（スーパーバイザー型） / 主な基盤：LangGraph（独自実装）, DeepAgents 等**
  人間が大枠の状態遷移（タスク分割や承認フロー）を設計しつつ、その中でのタスク分担、合議、再試行などの制御をAIに部分委譲する。
  実装にあたっては、トレースIDの伝播（Observability）や独自ログなどエンタープライズ特有の厳密な制御が求められる場合に**生のLangGraphをベースとして独自にハーネスを実装する選択肢**と、仮想ファイルシステムや記憶管理等のハーネスが標準搭載された**DeepAgents等の上位ライブラリを活用し開発効率を優先する選択肢**を要件に応じて使い分ける。ここでAIの動的計画機能を意図的に制限（固定TODO化）して運用することで、堅牢なSV型を効率的に実現する。SV型では、LangGraph の Checkpointer に代表される状態永続化、Pause / Resume、非同期 HITL、動的要約やメッセージ切り詰めによるコンテキスト圧縮が、業務運用を成立させるハーネス要件となる。

* **自律型（エージェント型） / 主な基盤：LangGraph（独自実装）, DeepAgents, Claude Code 等**
  目標と境界条件のみを与え、AI自身が思考・行動・観察のループで手順を動的に決定するループを完全委譲する。SV型と同様の実装基盤（LangGraphやDeepAgents等）を用いつつ、TODOの動的計画やサブエージェント生成機能を解放（フルオート化）して利用する。予期せぬ挙動に備え、権限を隔離した強固なハーネス環境（サンドボックス等）での実行を大前提とする。必要なハーネスは最も重く、権限分離、ファイルシステムやネットワークの隔離、反復回数・時間・コスト上限、監査ログ、自動テスト、強制停止条件まで含めて設計する必要がある。

**【ハイブリッド構成による実務適用】**
実務においては、単独の型だけで完結させるよりも、外側をWF、内側をSVとする「階層的ハイブリッド型」が扱いやすい。受付、前処理、承認、反映などの業務の大枠（手順）はWFで固定し、難易度の高い推論、合議、品質改善の工程のみをSVで動的に扱う構成である。
さらに、受付時の意図解釈、優先度判定、どのフローへ乗せるかのルーティング判断を担う**「Coordinator（入口ユニット）」**を配置することで、WFとSVの責務分界点をより明確に設計できる。Coordinator は補助的な飾りではなく、要求の入口で「どこまでを固定フローで扱い、どこからを動的制御へ委譲するか」を判断する実務上の重要な責務を持つ。

基盤との対応づけも、この整理に沿って理解できる。Difyは人間が制御経路を事前定義するWF型に向き、LangGraphは固定フローにも使えるが、特に状態遷移を保ちながらAIに部分委譲するSV型において真価を発揮する。自律型エージェントは目標ベースの探索的タスクに必須となるが、サンドボックス、ガードレール、停止条件、監査ログといったセーフティネットの構築が前提となる。これは、歴史的に「完全自律」だけでは本番運用に耐えず、ハイブリッド構成とハーネスエンジニアリングの両輪が必要になったことを反映している。これらの整理は、AnthropicのWorkflow/Agentの区分、およびOpenAIやNVIDIAが示すエージェントコンポーネントの根本定義と矛盾なく統合されている。


### 2.3 設計の鉄則: 予見性に基づくモデル選択

不必要に動的なモデルを採用しないことが、コスト、精度、制御性の面で重要である。Anthropic の整理と同様に、まずは単純な構成から始め、必要な場合にのみ複雑性を上げる。予見性が高いほど WF 型、予見性が下がるほど SV 型や自律型を検討し、あわせて制御フローをどこまで AI に委譲するかを判断する。

| 予見性のレベル | 制御フロー主体 | 決定のタイミング | 推奨モデル | 必要なハーネス要素 | 典型基盤 |
| --- | --- | --- | --- | --- | --- |
| 完全にタスク分解可能 | 人間 | 設計時 | WF 型 | 入力検証、分岐条件の明示、監査ログ | Dify、固定フローの LangGraph |
| ある程度の実行計画あり | 人間 + AI Supervisor | 状態遷移は設計時、分担と再試行は実行時 | SV 型 | 状態永続化、非同期 HITL、Pause / Resume、コンテキスト圧縮 | LangGraph、必要に応じて DeepAgents |
| 計画不能で試行錯誤が必要 | AI | 実行時 | 自律型 | サンドボックス、権限隔離、停止条件、監査ログ、自動テスト | サンドボックス型実行基盤、必要に応じて自律型ライブラリ |

## 3. 各型の概要
### 3.1. WF 型: 機能ユニットの連鎖で組む

WF 型は、入力、処理、出力を比較的固定したステップでつなぐ方式である。最小単位はチャット UI ではなく、1 つの具体的タスクを処理する機能ユニットとデータの流れで捉える。

Anthropic が示す prompt chaining、routing、parallelization などのパターンは、この WF 型に包含される。分岐や並列実行を含んでも、制御経路そのものは人間が事前に定義するため、GUI や DAG で経路を明示しやすい Dify や、コードで固定フローを記述する LangGraph が適する。

#### 3.1.1 機能ユニットの例

* 抽出ユニット
* 分類ユニット
* 変換ユニット
* 判定ユニット

---

### 3.2. SV 型: 役割を持つユニットの合議と人間の監督で組む

SV 型は、正解が一つではなく責任が重い領域で、自律型の予測不能さを抑えつつ安全に業務適用するための方式である。


Anthropic が示す orchestrator-workers や evaluator-optimizer パターンに近いのが、この SV 型である。AI の Supervisor と、人間の Supervisor を分けて捉える点が重要であり、前者はタスク分担、評価、再試行（ループ）を担い、後者は承認、停止判断、責任ある最終決裁を担う。

業務適用では、人間承認を伴う非同期 HITL（Human-in-the-loop）を組み込むことを原則とする。ただし承認を全ステップに挟むのではなく、不可逆操作、対外送信、低確信度、ポリシー抵触のおそれがある場面など、意味のある節目に組み込む。

### 3.2.1 構成ユニットの例

* 統合ユニット
* 評価ユニット
* 業務ユニット

---

### 3.3. 自律型: AI 自身の思考ループで組む

自律型は、手順が固定できず探索的に解決する必要がある領域に適用する。目標と利用可能なツール群を与え、AI が計画と実行を繰り返す。

前述の OpenAI が示す planning / memory / tool use、NVIDIA が示す reasoning / planning / execution の根本定義に最も忠実な型であり、Application 層（脳）が目標分解、自己反省（Reflection）、次行動の決定を担い、Tool 層（手足）が外界への実行を担う。

業務適用では、Read-Only やサンドボックス内の実行から始め、不可逆操作を伴う領域では段階的自律化を採用する。自律型は未知のツール呼び出しやコード実行を Try & Error で進めるため、論理的なフレームワークだけで閉じず、権限、ファイルシステム、ネットワークを隔離したサンドボックス、ならびに反復回数、時間、コスト、書き込み範囲を制限する停止条件を前提にする。

## 4. 各型とハイブリッド構成を支えるハーネス

AIエージェントの業務適用では、基盤モデルそのものの性能だけでなく、エージェントがどの情報を見て、どの状態を保持し、どこまで操作でき、どのように検証されるかを規定する周辺構造が成果を大きく左右する。本書では、この周辺構造をハーネスと呼ぶ。ハーネスは単なる補助機能ではなく、WF型、SV型、自律型、およびそれらを組み合わせたハイブリッドマルチエージェント構成を、業務で安全かつ継続的に運用するための基盤である。

近年の議論では、ハーネスエンジニアリングとは、エージェントが誤るたびに人間がその場で補正し続けるのではなく、環境側の改善によって同じ誤りを再発しにくい構造へ変えていくことだと整理されている。OpenAI や Anthropic が示すように、長時間タスクや複雑なタスクでは、ツール、状態管理、観測性、評価、停止条件、知識の記録といった環境設計が、単体モデルの能力と同じくらい重要になる。

### 4.1 ハーネスとは何か

本書におけるハーネスとは、エージェントの実行環境、状態管理、検証手段、統制機構を含む設計上の総体である。具体的には、状態永続化、非同期HITL、コンテキスト圧縮、サンドボックス、権限隔離、停止条件、監査ログ、評価の仕組み、知識の記録と参照経路などが含まれる。

重要なのは、ハーネスが自由度を奪うための制約ではなく、エージェントが正しい作業を継続できるようにする足場である点である。特に業務適用では、モデルにすべてを期待するのではなく、環境側で再現性、監督可能性、安全性を補うことが前提となる。

### 4.2 WF型に必要なハーネス

WF型では、制御フロー自体は人間が事前に定義するため、必要なハーネスは比較的軽量である。中心となるのは、入力検証、分岐条件の明示、監査ログ、失敗時の再実行可能性である。

ここでの目的は、AI に大きな自由度を与えることではなく、人間が設計した経路を安定して再現し、運用上の追跡性を確保することにある。そのため WF 型では、複雑な自律性よりも、経路の明示性、再現性、保守容易性を支えるハーネスが重視される。

### 4.3 SV型に必要なハーネス

SV型では、制御フローの一部をAIに委譲するため、WF型より強いハーネスが必要になる。特に重要なのは、状態永続化、Pause / Resume、非同期HITL、コンテキスト圧縮、評価と再試行のループを支える仕組みである。

SV型では、Supervisor がタスク分担、合議、評価、再試行を繰り返すため、途中状態を安全に保持できなければ業務運用に耐えない。また、生成役が自身の成果を甘く評価しやすいという性質があるため、生成と評価を役割として分離し、外部から品質を点検できる構成が有効である。SV型のハーネスは、部分委譲された動的制御を、監督可能で再開可能な形に保つ役割を持つ。

### 4.4 自律型に必要なハーネス

自律型では、目標と境界条件のみを与え、AIが探索的に手順を決定する。そのため、ハーネスは最も強く設計しなければならない。必要となるのは、権限分離、ファイルシステムやネットワークの隔離、反復回数や時間やコストの上限、監査ログ、強制停止条件、自動テストや検証機構である。

自律型は最も高い柔軟性を持つ一方で、最も大きな逸脱リスクも持つ。したがって、自由度を高めるほど、環境側の安全帯も厚くする必要がある。業務適用においては、まず Read-Only や限定権限で開始し、十分な監査と停止条件を整えたうえで段階的に自律性を拡張する考え方が妥当である。

### 4.5 ハイブリッドマルチエージェント全体を支えるハーネス

実務で多く採用されるのは、WF型、SV型、自律型を単独で使い切る構成ではなく、それぞれを適材適所で組み合わせるハイブリッドマルチエージェント構成である。この場合のハーネスは、各型ごとの局所要件に加えて、型をまたぐ全体要件を担う必要がある。

具体的には、要求受付時の意図解釈とルーティング、状態の受け渡し、共通トレースID、承認ポイントの一貫性、監査可能性、評価結果の引継ぎなどである。特に Coordinator は、要求の入口で「どこまでを固定フローで扱い、どこからを動的制御へ委譲するか」を判断する重要な役割を持つ。ハイブリッド構成におけるハーネスは、単体エージェントの安全装置ではなく、複数の型が連携しながら業務を完遂するための接着層でもある。

また、長時間タスクでは、単一セッションに依存しない状態引継ぎも重要になる。コンテキスト圧縮だけでなく、必要に応じて状態を構造化して次の作業単位へ受け渡すことは、長時間稼働における有効な設計原則である。

### 4.6 本書と実現方式文書の役割分担

本書では、ハーネスを各型とハイブリッド構成を成立させるための設計原則として扱う。状態永続化をどの基盤で実現するか、どのようなイベント設計で Pause / Resume を実現するか、どの監視基盤やテスト基盤を用いるかといった実装方式の詳細は、[../02_アーキテクチャ実現方式/04_アプリケーション層の実現方式.md](../02_アーキテクチャ実現方式/04_アプリケーション層の実現方式.md) を正本とする。

ここで重要なのは、ハーネスが実装上の便利機能ではなく、Application層の採用方針そのものを支える設計責務であると理解することである。

### 5.1 Tool 層との境界

DB、API、ファイル等の外界アクセスは Tool 層の責務であり、Application はそれらをツールとして扱う。Application 層は reasoning / planning / 業務状態制御を担い、Tool 層は action / external access を担う。影響の大きい操作には HITL や AI ガバナンス層のゲートを挟む。

### 5.2 評価の責務分界

Application 層内の評価ユニットは、業務フローに閉じた品質保証を担う。企業共通の守りとしての共通評価基盤、低スコア時の HITL、停止判断は AI ガバナンス層の責務であり、正本は [02_AIエージェントの業務適用を見据えた生成AIガバナンス層の検討.md](../01_アーキテクチャ検討/02_AIエージェントの業務適用を見据えた生成AIガバナンス層の検討.md) の 3.5 を参照する。

### 5.3 組織的な責任分界

Application 層の文脈では、開発部門が業務ユニットと業務固有評価の精度を担う。IT / ガバナンス部門、ユーザーを含む全体の責任分界は AI ガバナンス層側で定義し、正本は [02_AIエージェントの業務適用を見据えた生成AIガバナンス層の検討.md](../01_アーキテクチャ検討/02_AIエージェントの業務適用を見据えた生成AIガバナンス層の検討.md) の 3.5 を参照する。
