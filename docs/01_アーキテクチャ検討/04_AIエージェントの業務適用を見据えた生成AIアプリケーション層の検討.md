# AIエージェントの業務適用を見据えた生成AIアプリケーション層の検討

---

## 1. 目的

生成AIアプリケーションの構成を Application 層 / AI ガバナンス層 / Tool 層で分けて考える。本書では、特に Application 層における実装パターン、制御方式、責任分界を整理する。

非同期 HITL、長時間実行、Pause / Resume、UI 復元、イベント設計などの実現方式は、[../02_アーキテクチャ実現方式/04_アプリケーション層の実現方式.md](../02_アーキテクチャ実現方式/04_アプリケーション層の実現方式.md) を正本として扱う。

---

## 2. Application 層の基本整理: WF 型 / SV 型 / 自律型

生成AIを業務に適用する際の代表的な実装パターンとして、次の 3 つを置く。

* WF 型: あらかじめ定めた手順の連鎖で実現する
* SV 型: AI による統制と人間による最終監督を前提に合議で実現する
* 自律型: AI 自身が思考、行動、観察のループを回して手順を動的に決定する

実装基盤の典型例は次のとおりである。

はい、おっしゃる通りです！LangChain社（LangGraphの開発元）が公開した**「DeepAgents」**は、間違いなく**典型的かつ最先端の「FMベースのエージェント」**です。

LangChain社は、LangGraphのエージェントランタイムを土台としつつ、より高度で複雑なタスクをこなすための機能群をパッケージ化したライブラリとして `deepagents` をリリースしました。

これをご作成中のアーキテクチャ資料（WF / SV / 自律型の3分類）に照らし合わせると、**DeepAgentsは「自律型（エージェント型）」の実装基盤として完璧にフィットします。**

DeepAgentsがどのような点でFMベースのエージェント（自律型）であり、アーキテクチャ上でどう位置付けられるのかを整理します。

---

### 1. なぜDeepAgentsは「FMベースのエージェント」なのか？

前述の論文（Agent Design Pattern Catalogue等）で定義されているFMベースのエージェントの要件は、「LLMを脳として使い、**計画（Planning）、記憶（Memory）、行動（Action）**のモジュールを持つこと」でした。
DeepAgentsは、まさにこの学術的なモジュール構成をOSSとして直接的に実装しています。

* **計画モジュール（Planning Tool）**:
    * 組み込みの `write_todos` ツールなどを持ち、AI自身が巨大な目標を細かなTODOリストに分解し、進捗を自己追跡しながらタスクを進めます。
* **記憶モジュール（File System Tools / Long-term Memory）**:
    * `ls`, `read_file`, `write_file` といった仮想ファイルシステム操作のツール群を持ち、LLMのコンテキストウィンドウ（短期記憶の限界）を超えた膨大なログや過去資料を「外部記憶」として読み書きします。
* **行動・委譲モジュール（Subagent Tool）**:
    * 必要に応じて特定のタスクに特化した「サブエージェント」を動的に生成し、処理を並列化・委譲します。

### 2. 3分類（WF / SV / 自律型）における位置づけ

* **WF型の実装基盤**: Dify（Workflow/Chatflow、GUI、運用UI 等）
* **SV型の実装基盤**: LangGraph（StateGraph、ループ/合議、CheckpointerによるHITL 等）
* **自律型の実装基盤**:
    * **独立環境型**: Claude Code などのコーディングエージェント（Dockerコンテナ等のサンドボックス環境）
    * **ライブラリ型**: **DeepAgents**（LangGraphを土台に、TODO計画・仮想ファイルシステム・サブエージェント動的生成等の機能を備えた、長期タスク向けの自律エージェント基盤）

---


### 2.1 設計の鉄則: 予見性に基づくモデル選択

不必要に動的なモデルを採用しないことが、コスト、精度、制御性の面で重要である。予見性が高いほど WF 型、予見性が低いほど SV 型や自律型を検討する。

| 予見性のレベル | 推奨モデル | 決定のタイミング |
| --- | --- | --- |
| 完全にタスク分解可能 | WF 型 | 設計時 |
| ある程度の実行計画あり | SV 型 | 実行時 |
| 計画不能で試行錯誤が必要 | 自律型 | 実行時 |

---

## 3. WF 型: 機能ユニットの連鎖で組む

WF 型は、入力、処理、出力を比較的固定したステップでつなぐ方式である。最小単位はチャット UI ではなく、1 つの具体的タスクを処理する機能ユニットとデータの流れで捉える。

### 3.1 機能ユニットの例

* 抽出ユニット
* 分類ユニット
* 変換ユニット
* 判定ユニット

---

## 4. SV 型: 役割を持つユニットの合議と人間の監督で組む

SV 型は、正解が一つではなく責任が重い領域で、自律型の予測不能さを抑えつつ安全に業務適用するための方式である。

AI の Supervisor と、人間の Supervisor を分けて捉える点が重要である。業務適用では、人間承認を伴う非同期 HITL を組み込むことを原則とする。

### 4.1 構成ユニットの例

* 統合ユニット
* 評価ユニット
* 業務ユニット

---

## 5. 自律型: AI 自身の思考ループで組む

自律型は、手順が固定できず探索的に解決する必要がある領域に適用する。目標と利用可能なツール群を与え、AI が計画と実行を繰り返す。

業務適用では、Read-Only やサンドボックス内の実行から始め、不可逆操作を伴う領域では段階的自律化を採用する。

---

## 6. 制御パターン: オーケストレーション型 / 階層型 / Hand-off 型

複数ユニットの連携には、代表的に次の 3 パターンがある。

### 6.1 オーケストレーション型

統合ユニットが計画を立て、個々のユニットを順次呼び出し、結果を評価する。透明性が高く、人間が途中介入しやすい。

### 6.2 階層型

親ユニットが必要に応じて子ユニットを呼び出し、複雑なタスクを分割する。自律性が高いが、内部ブラックボックス化を防ぐ状態設計が重要である。

### 6.3 Hand-off 型

中央の統制ユニットを持たず、各業務ユニットが次の専門ユニットへ状態を引き継ぐ。柔軟性は高いが、追跡性と HITL の組み込みは難しくなる。

---

## 7. ハイブリッド型: WF × SV の組み合わせ

実務では、外側を WF、内側を SV とするハイブリッド型が扱いやすい。

* 外側の WF は、受付、前処理、承認、反映などの業務手順を固定フローで可視化する
* 内側の SV は、難所の推論、合議、品質改善などの動的制御を担う

---

## 8. 代表 OSS での実現例

* Dify は GUI ベースでの WF 型に向く
* LangGraph はコードベースの WF 型や SV 型に向く
* 自律型エージェントは探索的タスクに向くが、サンドボックスとガードレールが前提になる

# AIガバナンス層の実現方式

---

## 0. 位置づけ

本書は、[02_AIエージェントの業務適用を見据えた生成AIガバナンス層の検討.md](../01_アーキテクチャ検討/02_AIエージェントの業務適用を見据えた生成AIガバナンス層の検討.md) で定義した Why / What / Conceptual Architecture を、どのような実装方式で成立させるかを整理する文書です。

[01_アーキテクチャ検討](../01_アーキテクチャ検討) 配下の文書が背景、要求、責務分界を扱うのに対し、本書は Guardrails、Proxy、隔離、観測、評価、運用、PoC から本番への展開方法を扱います。

## 1. Guardrailsの実現方式

### 1.1 実装方針

ガバナンスレイヤーの具体実装は、既存の OSS や SaaS を Python 製の軽量プロキシで接続し、統制点としてまとめる構成が現実的です。全体像としては LiteLLM、NeMo Guardrails、Langfuse などを FastAPI ベースのプロキシで統合します。

### 1.2 AI Request ガバナンス Gateway

既存の API / HTTP Gateway は統合エンドポイントとして維持し、その直下に生成AI向けの HTTP 内容検査・統制を担う AI Request ガバナンス Gateway を配置します。

* **役割**: OIDC 等の認証、レート制限と連携しつつ、PII、禁則表現、インジェクション兆候などの入力 / 出力フィルタを適用する
* **実装パターン**: Kong の AI Proxy、Nginx + Lua、または Python サイドカーによる本文検査
* **適用対象**: チャット入力、外部連携 Webhook、MCP を含む north 境界の HTTP リクエスト

---

## 2. アクセス制御とProxy

### 2.1 Proxyを統制点として置く理由

生成AIで問題になるのは、入力の意図、出力の妥当性、ツール実行の正当性といった意味論の領域です。この領域は製品単体で E2E に説明責任まで含めて統制しにくいため、統制点としての Proxy をアーキテクチャとして確保します。

### 2.2 役割分担

* **API Gateway**: JWT 検証、WAF、レート制限、TLS 終端などの決定論的入口防御
* **LiteLLM / Model Proxy**: モデル抽象化、モデル切り替え、予算制御、ログ取得
* **Tool Proxy**: 権限チェック、Read-Only 制御、根拠データや出典の付与
* **ガバナンス Proxy**: 3方向（User / Data / Model）のインターセプト、統制ロジックの実行、ログと評価の起点

### 2.3 ガバナンス Proxy の実装イメージ

FastAPI ベースの軽量 Web サーバーとして実装し、LiteLLM、NeMo Guardrails、Langfuse を内部で利用します。Kubernetes ではサイドカーとしてデプロイし、アプリケーションからの外部通信を強制的に通過させます。

---

## 3. サンドボックスと実行隔離

### 3.1 サイドカー構成と隔離

アプリケーション（WF / SV）とガバナンス Proxy を同一 Pod 内のサイドカーとして配置し、ネットワークポリシーで直接外部へ出られない構成を取ります。

* **Network Policy**: アプリコンテナから外部への直接通信を遮断する
* **Localhost Binding**: アプリは localhost 上の Proxy を経由しないと外部へ出られない
* **Best of Breed**: LiteLLM、NeMo Guardrails、Ragas、Langfuse などを用途別に組み合わせる

---

## 4. ObservabilityとTraceability

### 4.1 目的

生成AIエージェントは、停止と再開、不確実な品質、複数コンポーネントを跨ぐ実行という特性を持つため、止められる・追える・説明できる監視基盤が必要です。

### 4.2 観測対象

* **可用性と性能**: API 成功率、p95 / p99 レイテンシ、Event Bus バックログ、DLQ 発生数
* **正しさと安全性**: ガードレール遮断率、Ragas / LLM-as-a-Judge のスコア、HITL の滞留時間
* **コストと資源**: テナント別トークン消費量、モデル別コスト、1 trace_id あたりの最大コスト

### 4.3 BFFを north 境界の制御点として置く理由

BFF は単なる HTTP 中継ではなく、入口正規化、Fast / Slow Track の経路制御、状態管理 DB の正本管理、外部境界の隠蔽を担う制御点です。これにより API Gateway は決定論的防御に専念でき、AIガバナンス層は意味論的統制と観測に集中できます。

### 4.4 trace_id中心設計

BFF が最終的に確定する trace_id を全レイヤーへ伝播させ、同期通信、非同期通信、永続化層の共通キーとします。thread_id や workflow_run_id などの製品固有 ID は trace_id と紐付けて管理します。

### 4.5 状態管理DBとトレースDBの分離

* **状態管理DB**: ユーザー向け UI の現在状態、思考履歴、Cancel / Resume 状態、ID マッピング
* **トレースDB**: プロンプト、トークン、ツール呼び出し順序、評価スコア、判定理由
* **書き込み主権**: 状態管理 DB は BFF、詳細履歴は Langfuse などの観測基盤へ送る

### 4.6 同期 / 非同期境界と通知

* **Fast Track**: 同期応答と SSE ストリーミング
* **Slow Track**: 202 Accepted で Event Bus へ委譲
* **Session Recovery**: 状態管理DBから最新スナップショットを返す
* **Cancel / Resume**: BFF が受け付け、状態管理 DB と Event Bus の双方へ反映する

## 5. Evaluationと運用監視

### 5.1 Evaluationを運用に組み込む理由

モデルの劣化やポリシーの不整合は、測らなければ検知できません。評価は PoC のための一時的作業ではなく、本番監視の一部として継続実行します。

### 5.2 Async Worker による評価基盤

評価処理はレスポンス後に非同期で実行します。Proxy がキューへ評価要求を投入し、Celery + Redis 等のワーカーが Ragas を実行し、Langfuse へスコアを書き戻します。低スコア時はアラートや HITL、ポリシー更新の判断へ接続します。

### 5.3 評価モード

* **Shadow Eval**: 本番影響なしの影評価
* **Async Monitor**: 応答後評価による継続監視
* **Synchronous Gate**: 高リスク操作での応答前ブロック / HITL 分岐

### 5.4 Semantic Layer

Grounding と権限制御を支える意味論データアクセス層として、GraphQL や FastAPI ラッパー API を用います。返すのは生データではなく、出典メタデータや access_level を含む意味付きデータです。

---

## 6. 運用管理とインシデント対応

### 6.1 管理者UIに最低限必要な機能

* trace_id / user_id による横断検索
* HITL 滞留ダッシュボード
* DLQ 管理画面
* コストアラートとキルスイッチ接続

### 6.2 インシデント対応

* **Kill Switch**: 特定モデル、特定ツール、特定業務の即時停止
* **フォレンジック**: trace_id による入力から評価結果までの一気通貫追跡
* **Policy as Code**: 承認、回帰テスト、ロールバックを含むポリシー変更管理

### 6.3 アンチパターン

* trace_id が BFF、Event Bus、ワーカー間で分断される
* HITL 待ちをログだけで追跡する
* Kill Switch や予算上限がない
* 評価を PoC で止めてしまう
* 管理者へ生ログだけを見せ、横断トレース UI を用意しない

---

## 7. PoC to Production

### 7.1 構築ロードマップ

* **Phase 1: Foundation**: LiteLLM によるモデル集約、Langfuse による完全ログ取得、サイドカー構成の確立
* **Phase 2: Logic & Grounding**: Ragas、Semantic Layer、出典メタデータ整備
* **Phase 3: Operations**: HITL、品質劣化アラート、ポリシー運用

### 7.2 将来の変化への備え

プロキシは実装そのものではなく制御点として維持し、将来より良い製品が出た場合でもアプリケーション側を改修せずに切り替えられるようにします。不要な機能はパススルーできるようにし、監査ログ収集点としての価値を残します。

## 1. 非同期処理を前提にした実装設計

AI エージェントは処理時間が読みにくく、人間承認も数時間から数日単位で遅延し得るため、同期 API 応答待ちではなく非同期ジョブ前提で設計します。

### 1.1 長時間実行のジョブ化

* 上位システムからの依頼に対し、即座に Thread ID や Job ID を返す
* 実処理はバックグラウンドで行う
* 完了時は Webhook や Event Bus を通じて上位フローを再開する

### 1.2 非同期HITLによる状態永続化

* 承認直前で LangGraph の interrupt を発火する
* Checkpointer に文脈全体をフリーズする
* 人間承認後、thread_id をキーに状態を復元して処理再開する

---

## 2. Event Bus / 状態管理DB / Checkpointer の分離

### 2.1 三者分離の原則

* **Event Bus**: 状態変化イベントを疎結合に伝える
* **状態管理DB**: UI が参照する現在状態のスナップショットを保持する
* **Checkpointer**: Application 層内部の文脈をフリーズ / 復元する

この 3 者を分けることで、推論の状態とユーザーに見せる状態を混同せずに済みます。

### 2.2 CQRSパターン

* Event Bus は「状態が変わった」という事実を伝える
* 状態管理DB は `RUNNING` や `PAUSED_FOR_HITL` などの現在状態を返す
* Checkpointer は interrupt 後の再開を可能にする内部状態ストアとして使う

### 2.3 Event Bus を挟む理由

* DB ポーリング地獄の回避
* ワーカー起動の疎結合化
* 思考ログやツール実行イベントの緩衝

---

## 3. 代表ユースケース

### 3.1 長時間実行のジョブ化

BFF が `JOB_REQUESTED` を publish し、Application のワーカーが subscribe して処理し、完了時に `JOB_COMPLETED` や `JOB_FAILED` を publish します。

### 3.2 思考プロセスのリアルタイム可視化と UI 復元

ワーカーが `AGENT_THOUGHT` を publish し、BFF が SSE でフロントへ流しつつ、状態管理DBへスナップショットを保持します。再接続時は状態管理DBから UI を復元します。

### 3.3 非同期HITL（Pause / Resume）

Application 側で `HITL_REQUIRED` を publish し、BFF が状態管理DB更新と通知を行います。人間入力を受けた BFF は `HITL_RESUMED` を publish して Application を再開します。

### 3.4 外部イベント駆動

GitHub / GitLab などの Webhook を BFF が受け、`PR_CREATED` や `PR_COMMENTED` のイベントとして publish し、Application のエージェントジョブを起動します。

---

## 4. north境界との責務分界

### 4.1 役割分担

* **API Gateway**: 決定論的な入口防御
* **BFF**: trace_id の確定、状態管理DB I/O、Fast / Slow Track、通知、Cancel / Resume
* **Application**: interrupt、Checkpointer、内部状態制御、状態変化の通知

### 4.2 書き込み主権

状態管理DBへの書き込み主権は BFF が持ち、Application やワーカーは Event Bus 経由で状態変化を通知します。これにより密結合を防ぎます。

---

## 5. イベント設計の最低限の標準

### 5.1 共通属性

* `trace_id`
* `event_type`
* `occurred_at`
* `producer`
* `idempotency_key`

### 5.2 ペイロード設計

HITL_REQUIRED などのイベントでは、状態、thread_id、人間が判断するタスク概要などを data に含めます。

### 5.3 運用上の注意点

* 冪等性を前提に Subscriber を実装する
* リトライと DLQ を実装する
* イベントには署名や mTLS を用い、PII は参照キー化する
* Redis Streams 等の PoC 構成でも trace_id と idempotency_key は最初から入れる

---

## 6. アンチパターン

* Event Bus を検索用途に使い、状態復元をイベントソーシングに頼る
* エージェントから状態管理DBを直接更新する
* BFF プロセス内に状態を保持する
* trace_id が north 境界、Event Bus、ワーカー間で分断される
* DLQ を実装せず、失敗イベントを消失させる

## 7. 関連文書と責任分界

### 7.1 Tool層との関係

DB、API、ファイル等の外界アクセスは Tool 層の責務です。Application はそれらをツールとして扱い、権限の大きい操作には HITL や AI ガバナンス層のゲートを挟みます。

### 7.2 評価の二段構え

* **Application 層内の評価ユニット**: 業務固有の品質保証
* **AI ガバナンス層の共通評価基盤**: 企業共通の守りと評価

### 7.3 責任分界

* **開発部門**: 業務ユニット・評価ユニットの精度
* **IT / ガバナンス部門**: 共通基盤としての堅牢性
* **ユーザー**: 最終出力と業務上の最終判断
