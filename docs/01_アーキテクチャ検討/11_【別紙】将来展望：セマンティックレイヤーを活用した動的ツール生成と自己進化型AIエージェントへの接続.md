# 【別紙】将来展望：セマンティックレイヤーを活用した動的ツール生成と自己進化型AIエージェントへの接続

*(※ 本資料は、Tool層アーキテクチャ本編を補完する別紙である。現在の標準技術（MCP等）は静的なツール定義を前提としているが、将来的にAIエージェントが自律的にツールを生成・拡張する「自己進化型」のパラダイムへ移行する可能性を見据え、その学術的背景とシステム拡張の方向性を整理したものである。)*

---

## 1. 動的ツール生成の歴史的背景と現状のギャップ

### 「ツールを自作する」という概念の萌芽（2023年）
LLMが既存のツールを使うだけでなく、「自らツールを作り出し、それを利用する」という概念自体は決して新しいものではない。2023年5月の段階で、[『Large Language Models as Tool Makers (LATM)』(https://arxiv.org/abs/2305.17126)](https://arxiv.org/abs/2305.17126) や [『CREATOR』(https://arxiv.org/abs/2305.14318)](https://arxiv.org/abs/2305.14318) などの記念碑的論文により、「AIを『ツールを作るメタAI』と『ツールを使う軽量AI』に分割し、未知のタスクに対する解決コードをその場で生成してキャッシュする」というアプローチの有効性がすでに実証されていた。

### 現状の標準プロトコル（MCP）の限界：静的パラダイム（2026年現在）
しかし、2026年現在のエンタープライズにおける事実上の標準プロトコルである MCP（Model Context Protocol）は、この動的なパラダイムを完全には取り込めていない。

現在のMCPサーバーは、開発者（人間）が事前に `description` や利用可能な関数のスキーマを静的に記述し、LLMは起動時にそれを読み込んで「Tool User（利用者）」として振る舞う設計にとどまっている。この静的アプローチが採用されている最大の理由は、現時点におけるシステムの「予測可能性」「セキュリティ（権限境界の固定化）」および「監査の容易性」を優先しているためである。プロトコルレベルで動的生成が今後どう標準化されるか（あるいは別の枠組みへ移行するか）は、いまだ業界内で確定していない。

---

## 2. メタデータとセマンティックレイヤーが拓く「動的ツール生成」の現実味

プロトコルが静的であっても、周辺のエンタープライズ・データ基盤の進化により、実運用における「ツールの動的生成」は単なる夢物語ではなくなりつつある。

AI（Tool Maker）が未知のタスクに対して動的にツール（コードやSQL）を生成しようとした際、最大の障壁となるのは「物理テーブルの複雑さ」である。生データベースの解読困難なテーブル名や複雑なJOIN条件をAIにゼロから解読させると、高確率でハルシネーションを起こす。この問題を解決し、動的ツール生成を安全に実現する大本命のアプローチが、**データカタログおよびセマンティックレイヤー基盤の活用**である。

### 2.1 エンタープライズ・データ基盤が果たす中核的役割
これら製品群は、AIが自律的にツールを生成するための「機械可読なナレッジグラフ（高品質な設計書）」として機能する。動的ツール生成の成否は、AIのコーディング能力以上に、これらの基盤が提供するメタデータの質に完全に依存する。

* **Unity Catalog (Databricks等) ── データの信頼性とリネージの提供**
  企業内のデータに対する統合アクセス権限と、データの血統（Lineage）を管理するカタログ。AIはツールを生成する前に、このカタログのAPIを叩き、「対象タスクに使える『Gold層（信頼済み）』のテーブルはどれか」を検索し、承認済みのスキーマ定義を獲得する。
* **dbt Semantic Layer ── 指標と計算ロジックの提供**
  「売上」や「アクティブユーザー数」などのビジネス指標を物理構造から切り離して定義する基盤。AIは生のSQLを組み立てるのではなく、dbtのメタデータを読み解き、「dbtの指標APIに適切なパラメータを渡してデータを取得する関数（ツール）」を動的に生成する。
* **Denodo ── データ仮想化と論理インターフェースの提供**
  多様なデータソースを一つの論理的なデータレイヤーとして仮想化して見せるプラットフォーム。AIはDenodoのデータディクショナリを解釈することで、裏側がSAPかSalesforceかを一切気にすることなく、標準化された仮想ビューを叩くためのAPIクライアント（ツール）を動的に生成できる。

### 2.2 カタログ駆動の「動的ツール生成」パイプライン

これら基盤を組み合わせた将来のAIエージェントの処理フローは、以下の通りとなる。

1. **検索 (Discovery)**: 高機能なメタAI（Tool Maker）が、Unity CatalogやDenodoのメタデータAPIを叩き、ユーザーの要求に関連する承認済みデータモデルを検索する。
2. **解釈 (Interpretation)**: ヒットしたデータセットについて、dbt Semantic Layer等から具体的な計算定義（必要なディメンション等）を読み取る。
3. **生成 (Creation)**: 読み取った仕様に基づき、「目的のデータを抽出・集計するPythonスクリプト（または一時的なエフェメラルMCPサーバー）」をオンザフライで自動生成する。
4. **検証と実行 (Governance & Execution)**: 生成されたコードをサンドボックス内でDry Run検証した上で、実行エージェント（Tool User）が使用してユーザーに回答を返す。

## 3. 動的AIエージェント（自己進化型）への接続

Tool層が静的な関数の集合体から、動的に生成・拡張される環境へと移行することは、上位の Application 層にいるAIエージェントの性質を根本から変容させる。それは「固定された手足で作業するAI」から、**「環境に合わせて自らの手足を進化させる自己進化型エージェント」**へのパラダイムシフトである。

* **経験からのツール自律生成**:
  最新の [『Evolving from Tool User to Creator via Training-Free Experience Reuse in Multimodal Reasoning (UCT)』(https://arxiv.org/abs/2602.01983)](https://arxiv.org/abs/2602.01983) の研究が示す通り、エージェントは「一度手動で苦労して解いた複雑なタスクの経験」を抽出し、それを再利用可能なプログラム（ツール）として自律的にパッケージ化し始める。
* **未知の環境への適応**:
  [『DynaSaur: Large Language Agents Beyond Predefined Actions』(https://arxiv.org/abs/2411.01747)](https://arxiv.org/abs/2411.01747) のアプローチのように、事前に定義されたAPI群が存在しないシステムに直面しても、エージェント自身がドキュメントを読み解き、Python等でそのシステムを操作するためのカスタムツールを動的にコーディングして実行する。
* **進化の評価（ベンチマークの誕生）**:
  さらに、[『Tool-Genesis』(https://arxiv.org/abs/2603.05578)](https://arxiv.org/abs/2603.05578) のような自己進化型エージェント専用のベンチマークが登場したことは、「ツールの自律的生成・拡張能力」が、今後のAIにとって例外的な機能ではなく、標準的に求められる基礎能力となりつつあることを意味している。

## 4. 本編アーキテクチャの価値：「動的生成」を受け入れるための器

未来のAIが自律的にツールを記述し、配備する時代（LLMs as Tool Makers）が到来した際、最も危険なのは「AIがどこで何のコードを実行し、誰の権限でアクセスしているか分からない」という状態に陥ることである。

本編で定義した Tool層アーキテクチャが、**「ステートレスな実行の徹底」「IdP連携によるユーザー権限継承（User Context Forwarding）」「サンドボックス化」** そして **「MCPという標準インターフェースへの準拠（カプセル化）」** を定めたことは、この未来に対する最大の布石となる。

将来、AIが動的にツールを生成したとしても、それがこの「標準化されたTool層（隔離されたコンテナとMCPのインターフェース）」の中に配備される限り、AIガバナンス層は既存の統制ルール（権限監査、予算管理、Egress遮断）をそのまま適用し、暴走を封じ込めることができる。すなわち、現在の静的な標準化は、未来の動的な自己進化を安全に受け入れるための「不可欠な器（インフラ）」なのである。

## 5. 動的ツール生成と自己進化に関する参考文献一覧

本別紙の基礎となる、Tool Creation（ツールの自作）と自己進化型エージェントに関する主要な学術論文を以下に示す。

### I. ツール生成の基礎パラダイム（2023年）
**1. Large Language Models as Tool Makers (LATM) (arXiv:2305.17126)**
* **リンク**: [https://arxiv.org/abs/2305.17126](https://arxiv.org/abs/2305.17126)
* **概要**: AIを「ツールを作るメタAI」と「ツールを使う軽量AI」に分割し、未知の問題に対して再利用可能な関数（ツール）を動的生成・キャッシュして解決する概念を実証した画期的論文。

**2. CREATOR: Disentangling Abstract and Concrete Reasonings of Large Language Models through Tool Creation (arXiv:2305.14318)**
* **リンク**: [https://arxiv.org/abs/2305.14318](https://arxiv.org/abs/2305.14318)
* **概要**: 既存のAPIに頼るのではなく、LLMにドキュメントとコード生成能力を与え、「自分専用の抽象的なツール」を創り出させることでAIの推論能力が劇的に向上することを証明した論文。

### II. 動的ルーティングと自己進化型エージェントへの発展（2024年〜2026年）
**3. DynaSaur: Large Language Agents Beyond Predefined Actions (arXiv:2411.01747)**
* **リンク**: [https://arxiv.org/abs/2411.01747](https://arxiv.org/abs/2411.01747)
* **概要**: 事前定義された固定API（静的ツール）に依存せず、エージェント自身が一般的なプログラミング言語を用いて、必要なアクションを動的に生成・結合・実行するフレームワーク。

**4. AutoTool: Dynamic Tool Selection and Integration for Agentic Reasoning (arXiv:2512.13278)**
* **リンク**: [https://arxiv.org/abs/2512.13278](https://arxiv.org/abs/2512.13278)
* **概要**: 推論の軌跡の途中で、必要になった瞬間に巨大なツールプールの中から動的にツール（およびそのDescription）を選び出し、統合する手法。コンテキスト限界を突破する動的構成の根拠。

**5. Evolving from Tool User to Creator via Training-Free Experience Reuse in Multimodal Reasoning (arXiv:2602.01983)**
* **リンク**: [https://arxiv.org/abs/2602.01983](https://arxiv.org/abs/2602.01983)
* **概要**: 事前の再学習を行わず、エージェントが推論過程の「経験（試行錯誤）」を抽出し、その場で再利用可能なツールコードとして自動生成するフレームワーク（UCT）。エージェントがリアルタイムで進化する証明。

**6. Tool-Genesis: A Task-Driven Tool Creation Benchmark for Self-Evolving Language Agent (arXiv:2603.05578)**
* **リンク**: [https://arxiv.org/abs/2603.05578](https://arxiv.org/abs/2603.05578)
* **概要**: AIエージェントが「いかに適切に自分用のツールを作成・進化させることができるか」を評価するベンチマーク。「Tool Creation」がエージェントの基礎能力として標準化されつつあることを示す。
