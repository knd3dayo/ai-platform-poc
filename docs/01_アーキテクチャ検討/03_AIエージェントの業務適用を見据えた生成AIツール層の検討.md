# AIエージェントの業務適用を見据えた生成AIツール層の進化と設計背景

*(※ 本資料は、Application層（脳）から独立し、外界との安全な接続を担う「Tool層（手足）」の設計妥当性と、その具体的なアーキテクチャ方針を定義したものである。)*

---
## 1. 生成AIツール層技術の進化の歴史（2023年〜2026年現在）

AIが外界のシステムを操作する「Tool層」のアーキテクチャは、プロンプトへの直書きから始まり、データ抽象化への苦悩を経て、最終的に独立した標準プロトコル（MCP）へと切り離される劇的な進化を遂げた。

このTool層の進化は、単なるAPI連携の歴史ではない。それはAIの推論を助けるための**『インターフェースの構造化（ACIの確立）』**、AIの認知限界を補う**『抽象化の外部化（セマンティックレイヤー）』**、そしてシステムを安定稼働させるための**『状態と実行の分離（ステートレス化）』**という3つの巨大な潮流によって形作られている。

### 第1期：LLMの「手」の獲得と構造化への模索（2022年末〜2023年初頭）
初期のLLMは学習時点の知識しか持たず、ハルシネーションが頻発した。これを打破したのが、思考とツール実行を繰り返すことで外部ツールの利用を可能にした[『ReAct: Synergizing Reasoning and Acting in Language Models』(https://arxiv.org/abs/2210.03629)](https://arxiv.org/abs/2210.03629)（2022末〜2023）という画期的な手法である。

しかし、当初は自然言語のテキストから正規表現でコマンドを抽出するような脆い実装であった。この構造的な欠陥に対し、[『Toolformer: Language Models Can Teach Themselves to Use Tools』(https://arxiv.org/abs/2302.04761)](https://arxiv.org/abs/2302.04761)（2023）は、LLMにAPI呼び出しの構文を直接学習させることで、モデル自らが自発的かつ正確に外部APIを呼び出せることを証明した。ここから、プロンプトでの指示ではなく「構造化されたツール呼び出し」へとインターフェースが進化を始めた。

### 第2期：複雑なタスクへの挑戦と「状態と実行の分離」（2023年中頃〜末）
ReActのような「都度考えながら動く」手法は、タスクが複雑化すると無限ループやタイムアウトでクラッシュしやすいことが判明した。これを解決するため、[『Plan-and-Solve Prompting: Improving Zero-Shot Chain-of-Thought Reasoning』(https://arxiv.org/abs/2305.04091)](https://arxiv.org/abs/2305.04091)（2023）が確立され、最初に全体計画（Plan）を立ててから個別のタスク実行（Solve）に移るアプローチが主流となった。これが「脳（Application層）」の不確実な状態管理と、「手足（Tool層）」の確実な実行を論理的に分離する（ステートレス化の）土壌となった。

同時に、[『HuggingGPT: Solving AI Tasks with ChatGPT and its Friends in Hugging Face』(https://arxiv.org/abs/2303.13746)](https://arxiv.org/abs/2303.13746)（2023）は、AI自身が数千の専門ツールの機能説明（Description）を読み取って動的にルーティングするアーキテクチャを提唱し、ツール利用における「詳細な文脈メタデータの提供」の重要性を浮き彫りにした。

### 第3期：生アクセスの限界と「認知負荷の外部化・ACIの確立」（2023年末〜2024年中頃）
LLMに「生のSQL」や「複雑なSaaS API」を直接ツールとして渡すアプローチが破綻し始めた時期である。[『Can LLM Already Serve as A Database Interface? A BIg Bench for Text-to-SQLs』(https://arxiv.org/abs/2305.03111)](https://arxiv.org/abs/2305.03111) や [『Gorilla: Large Language Model Connected with Massive APIs』(https://arxiv.org/abs/2305.15334)](https://arxiv.org/abs/2305.15334) などの論文により、AIは物理的なテーブル構造や社内固有のビジネスロジックを理解できず、APIハルシネーション（引数のでっち上げ等）を高確率で起こすことが実証された。

この反省から、AIに生データを触らせるのではなく、データ基盤側でビジネス用語に翻訳した[『The Semantic Layer』(https://docs.getdbt.com/docs/use-dbt-semantic-layer/dbt-sl)](https://docs.getdbt.com/docs/use-dbt-semantic-layer/dbt-sl)を構築し、AIの認知負荷（推論の負担）を外部へ肩代わりさせるアプローチが必須となった。

さらに、[『SWE-agent: Agent-Computer Interfaces Enable Automated Software Engineering』(https://arxiv.org/abs/2405.15793)](https://arxiv.org/abs/2405.15793)（2024）は、人間向けの複雑なインターフェースをAIにそのまま使わせるのではなく、AI専用に設計された簡潔なコマンド群**「ACI（Agent-Computer Interface）」**を提供することで、エージェントのタスク成功率が劇的に向上することを証明した。これにより、「Tool層は対象システムをAI向けに抽象化・整形して提供するプロキシである」という責務が確定した。

### 第4期：プロトコルの標準化（MCPの誕生）と「文脈・実行」の統合（2024年末〜2026年現在）
各企業が独自のACIや抽象化APIを乱造する課題を解決するため、2024年11月にAnthropicらが主導して発表した[『Model Context Protocol (MCP) Specification』(https://modelcontextprotocol.io/)](https://modelcontextprotocol.io/) が、翌2025年にかけて業界主要ベンダーに次々と採用され、業界標準として定着した。

MCPは単なる「Tools（関数実行）」にとどまらず、第3期で求められた「Resources（意味づけされたセマンティックな文脈）」をプロトコルレベルで統合した究極のACIである。これにより、AIエージェントとデータソースは完全に分離され、ステートレス実行とゼロトラスト（ZTNA）による権限継承を内包する堅牢な「エンタープライズ業務実行インフラ」へと昇華した。

## 2. 歴史的背景と本システム・アーキテクチャの関係

第1章の進化史は、当システムのTool層が単なる「APIの寄せ集め」ではなく、最新の学術的・産業的パラダイムを反映した**「標準化されたステートレスな実行基盤」**であることを強く裏付けている。当アーキテクチャは以下の通り、公的標準および学術的要件をシステム構造として直接的に充足している。

**① セマンティックレイヤーの分離・委譲（第3期の教訓）**
LLMに直接生データを触らせた際のハルシネーションを防ぐため、「データの意味付けと抽象化」の責務をTool層の内部から切り離し、後方のデータ基盤（セマンティックレイヤー）へ委譲するアーキテクチャ（第5.2項・第7章）を標準としている。
* **準拠する要件・参考文献**: 『Can LLM Already Serve as A Database Interface?』, 『The Semantic Layer』

**② MCPの採用による置換容易性と再利用性の確保（第4期の体現）**
密結合から脱却し、MCPというオープン標準プロトコルを採用した。これにより、AIモデルの進化に依存せず、ツール資産を全社横断で使い回す「業務実行インターフェースの資産化（第4章）」を実現している。
* **準拠する要件・参考文献**: 『Model Context Protocol Specification』

**③ Plan & Solve 思想に基づく「ステートレス化」とHITLの排除（第2期の教訓）**
不確実な「状態管理（Application層）」と確実な「実行（Tool層）」は分離すべきである。本アーキテクチャは、Tool層に人間の承認待ちを持たせず、即座にJob IDを返すステートレス設計（第5.3項・第7章）を徹底することでタイムアウト障害を防いでいる。
* **準拠する要件・参考文献**: 『Plan-and-Solve Prompting』

**④ ゼロトラストに基づく権限プロキシ（第4期のセキュリティ要件）**
Tool（MCP）サーバー内に独自の固定APIキーを持たせず、ユーザーの認証トークンをバックエンドへそのまま貫通させる「User Context Forwarding（第5.1項）」を採用している。
* **準拠する要件・参考文献**: NIST 『SP 800-207: Zero Trust Architecture』

## 3. 目的と定義：AIの「手足」をステートレスに標準化する

前述の進化史を踏まえ、本システムにおけるTool層の役割を以下に定義する。AIアプリケーションをエンタープライズ業務で安定稼働させる鍵は、不確実な「思考（Application層）」と、確実な「実行（Tool層）」をアーキテクチャレベルで完全に分離することにある。

* **目的**: Application層（WF/SV/自律型エージェント）が外界のシステムとやり取りするための、高信頼・標準化された「業務実行インターフェース（MCPサーバー等）」を定義・整備する。
* **実態**: 社内データ（ファイルサーバー、BOX、DB、Wiki）やSaaS（Zendesk等）にアクセスするための個別のAPI群を、AIが解釈・実行可能なプロトコル（Model Context Protocol等）でカプセル化したものの集合体である。
* **大原則（ステートレス）**: Tool層は状態（State）を持たず、呼び出されたら「即座に」結果、または受付票（Job ID）を返す。人間を待つなどの状態管理（永続化）は絶対に行わない。

---

## 4. なぜ「個別のAPI直接連携」ではなく「Tool層」なのか？

Tool層をApplication層から独立させる主な狙いは、アプリやAIモデルの変更に耐えうる **“業務実行インターフェースの資産化”** にある。個別のプロンプト内にAPIの叩き方を直書きする第1期のアプローチからの脱却を図る。

* **再利用性（Reusability）**: 複数業務のAIエージェントから、同じツールを共通の道具として使い回すことができる。
* **置換容易性（Portability）**: LLMのモデルを将来入れ替えても、標準プロトコルに準拠していればツール接続の仕様や実装は維持できる。
* **ガバナンスの一元化**: ツールごとの権限・監査・制約を共通のプロトコルで一元管理できる。

## 5. Tool層の設計原則：権限・意味・非同期・可逆性

Tool層を「AIに持たせる道具箱」として設計する際、実務でボトルネックとなる「権限の壁」「意味の壁」「時間の壁」を突破するための鉄則を定める。

### 5.1 セキュリティと権限の「プロキシ（代理）」管理（最重要）
「だれがどのデータにアクセス可能か？」の制御は、単なる固定のAPIキーではなく、**“実行ユーザーの権限コンテキスト”** で行うことを原則とする。
* **User Context Forwarding**: アプリケーションを利用しているユーザーの認証情報（OAuthトークン等）をMCP経由でバックエンドシステムまで貫通（引き継ぎ）させる。
* **越権アクセスの防止**: ツール側で「このユーザーがアクセス可能なデータのみ」をフィルタリングしてAIに返す。

### 5.2 統合認証基盤とデータ基盤の積極的採用（セマンティックレイヤー）
LLMに生データを直接触らせると、業務コンテキストの欠如によるハルシネーション（誤集計・誤解釈）が高確率で発生する。これを防ぐため、**「権限判定や、データの意味付け（セマンティックレイヤー）の提供は、バックエンドのデータ基盤やゲートウェイに委譲する」** ことを基本方針とする。
* **統合認証基盤（IdP）の連携**: Entra IDやOkta等を利用し、システム全体でトークンベースの権限継承を標準化する。
* **セマンティックレイヤー（意味的データモデル）の必須化**: 物理DBの複雑な構造を隠蔽し、AIが理解しやすいビジネス用語で定義されたビューのみを公開する。
* **ZTNA（ゼロトラストプロキシ）の活用**: MCPから社内システムへのアクセスは、アイデンティティベースで認可された通信のみを許可する。

### 5.3 タイムアウトを防ぐ「非同期ジョブ」設計
AIが指示した重い処理において、同期通信ではApplication層がタイムアウトしクラッシュする。
* **受付の即答**: 時間のかかるツールは処理完了を待たず、即座に「Job ID」のみをAIに返す（ステートレス化）。
* **状態の分離**: AI（Application層）はJob IDをもとに一時休眠し、Tool層は裏側で処理を完遂させる。

### 5.4 書き込みアクションの「可逆性」と「承認前提の設計」
実業務では「DB更新」「チケット起票」等の **Write（不可逆操作）** が必ず発生する。
* **Read / Write の厳格な分離**: 参照専用ツールと更新用ツールを明確に分ける。
* **Dry Run（事前検証）**: Write系ツールにおいて、実際の更新前に「何が起きるか（差分）」をシミュレーションしてAI（および人間）に提示する。
* **承認トークンの要求**: 更新用ツールは、人間の承認（HITL）とAIガバナンス層のチェックを通過したリクエストのみを受け付ける仕様とする。

## 6. Tool層の4大機能要件（接続・意味・統制・安全）

| 分類 | 機能要件 | 概要 |
| :--- | :--- | :--- |
| **接続** (Connectivity) | **プラグイン化** | MCPによるDB/SaaS/社内APIの統一的な接続インターフェース。 |
| | **ハイブリッドアクセス** | 非構造化データ（RAG）と構造化データへの統一アクセス。 |
| **意味** (Semantics) | **データ抽象化（ビュー）** | 物理テーブルを隠蔽し、ビジネス用語のセマンティックモデルを提供。 |
| | **文脈の提供** | AI向けの詳細なカスタムインストラクション（Description等）の提供。 |
| **統制** (Governance) | **Identity-aware Access** | 実行ユーザーの権限をバックエンドまで継承する認可制御。 |
| | **実行監査ログ出力** | 「誰の代理で、どのAIが、何のデータを操作したか」の証跡記録。 |
| **安全** (Safety) | **Dry Run機能** | 実行前に更新差分を返すシミュレーション機能。 |
| | **局所的Rate Limiting** | 特定のバックエンドに対する過剰なリクエストを防ぐ自己防衛。 |

## 7. 他層（AIガバナンス層・Application層）との責務分界

アーキテクチャ全体の一貫性を保つため、「Tool層（手足）」「Application層（脳）」「AIガバナンス層（免疫・関所）」の明確な分離方針を提示する。

| 検討機能 | 配置先 | 理由・設計思想 |
| :--- | :--- | :--- |
| **ユーザー権限継承** | **Tool層** | システム固有の権限モデルに依存するため、ツール側で透過的に処理する。 |
| **セマンティックレイヤー** | **データ基盤等** | データの意味付けと抽象化はDWHやデータ仮想化等のバックエンド層へ委譲する。 |
| **Dry Runの実装** | **Tool層** | 対象システムごとに「どう戻すか」の実装が異なるため、ツール内部でカプセル化する。 |
| **HITL（人間の承認待機）** | **Application層** | **Tool層はステートレスであるべき。** 人間を待つ（状態の永続化）のはLangGraph等の役割。 |
| **Rate Limit / 異常遮断** | **AIガバナンス層** | エージェントの暴走検知や全社的なAPIコールの閾値管理は横断的に行う。 |

## 8. ツール層アーキテクチャに関する参考文献一覧

本アーキテクチャの妥当性と、Tool層に求められる「構造化」「ステートレス」「抽象化（ACI）」「ゼロトラスト」の設計根拠となる主要な学術論文および標準仕様を示す。

### I. インターフェースの構造化と「状態・実行の分離」（潮流A・Cの根拠）

**1. ReAct: Synergizing Reasoning and Acting in Language Models (arXiv:2210.03629)**
* **リンク**: [https://arxiv.org/abs/2210.03629](https://arxiv.org/abs/2210.03629)
* **概要**: LLMの推論（思考）とツール実行（行動）を統合する画期的な論文。外部情報を動的に取得するエージェントの基礎を築いた。
* **アーキテクチャとの関係**: Application層（脳）とTool層（手足）が対話しながらタスクを進めるという、本システム全体の動作パラダイムの根源的な裏付けとなる。

**2. Toolformer: Language Models Can Teach Themselves to Use Tools (arXiv:2302.04761)**
* **リンク**: [https://arxiv.org/abs/2302.04761](https://arxiv.org/abs/2302.04761)
* **概要**: LLMにAPI呼び出しの構文を学習させることで、モデル自らが自発的かつ正確に外部ツールを呼び出せることを証明した論文。
* **アーキテクチャとの関係**: 自然言語のテキスト抽出ではなく、「構造化された関数呼び出し（Function Calling）」を前提としたAPIインターフェースを整備する設計の原点となる。

**3. Plan-and-Solve Prompting: Improving Zero-Shot Chain-of-Thought Reasoning (arXiv:2305.04091)**
* **リンク**: [https://arxiv.org/abs/2305.04091](https://arxiv.org/abs/2305.04091)
* **概要**: 事前に全体計画（Plan）を立て、その後で実行（Solve）に移る手法の有効性を実証した論文。
* **アーキテクチャとの関係**: 本システムの「人間の承認待ち（HITL）や計画の保持はApplication層が行い、Tool層はただ指示されたタスクをステートレスに即時実行する」という明確な責務分界の理論的根拠。

### II. 認知負荷の外部化と「ACI」の確立（潮流Bの根拠）

**4. HuggingGPT: Solving AI Tasks with ChatGPT and its Friends in Hugging Face (arXiv:2303.13746)**
* **リンク**: [https://arxiv.org/abs/2303.13746](https://arxiv.org/abs/2303.13746)
* **概要**: LLMが数千の専門ツールの機能説明（Description）を読み取り、タスクに応じて最適なツールを動的にルーティングするアーキテクチャ。
* **アーキテクチャとの関係**: ツールを利用する際、「ツールの機能や意味を説明するメタデータ」がいかに重要かを示しており、Tool層がAI向けの詳細な文脈を提供しなければならないことの証明となる。

**5. Can LLM Already Serve as A Database Interface? A BIg Bench for Text-to-SQLs (arXiv:2305.03111)**
* **リンク**: [https://arxiv.org/abs/2305.03111](https://arxiv.org/abs/2305.03111)
* **概要**: 学術用のDBでは高精度を出せるLLMが、複雑な実ビジネスのDBに生アクセスすると精度が急落することを証明した論文。
* **アーキテクチャとの関係**: Tool層から直接生DBを叩かせるアプローチを棄却し、間にセマンティックレイヤーを挟む本設計の強力な根拠となる。

**6. Gorilla: Large Language Model Connected with Massive APIs (arXiv:2305.15334)**
* **リンク**: [https://arxiv.org/abs/2305.15334](https://arxiv.org/abs/2305.15334)
* **概要**: LLMに複雑で大量のAPIを直接操作させると、引数や仕様をでっち上げる「APIハルシネーション」が高確率で発生することを実証した論文。
* **アーキテクチャとの関係**: プロンプト内にAPIの仕様を直接書き込むアプローチの限界を示すものであり、抽象化されたツールサーバーが必須であることの証明。

**7. SWE-agent: Agent-Computer Interfaces Enable Automated Software Engineering (arXiv:2405.15793)**
* **リンク**: [https://arxiv.org/abs/2405.15793](https://arxiv.org/abs/2405.15793)
* **概要**: 人間向けの複雑なインターフェースをAIにそのまま使わせるのではなく、AI専用に設計された「ACI（Agent-Computer Interface）」を提供することで成功率が劇的に向上することを証明した論文。
* **アーキテクチャとの関係**: 「Tool層は対象システムをそのまま露出させるのではなく、AI向けに抽象化・整形して提供するプロキシである」という役割定義の絶対的な学術的根拠となる。

### III. インターフェースの標準化とデータ抽象化基盤

**8. Model Context Protocol (MCP) Specification**
* **リンク**: [https://modelcontextprotocol.io/](https://modelcontextprotocol.io/)
* **概要**: 2024年11月に発表され、2025年にかけて業界標準として定着した、AIエージェントとデータソース/ツールを標準化された手法で接続するためのオープンプロトコル。
* **アーキテクチャとの関係**: Tool層が個別のAPI連携によるスパゲッティ化を避け、「Tools（実行）」と「Resources（文脈）」を統合したエンタープライズ業務実行インフラとして独立していることの最大の技術的根拠。

**9. The Semantic Layer: The Interface for LLMs to the Enterprise (dbt Labs Docs)**
* **リンク**: [https://docs.getdbt.com/docs/use-dbt-semantic-layer/dbt-sl](https://docs.getdbt.com/docs/use-dbt-semantic-layer/dbt-sl)
* **概要**: 複雑な物理データベース構造をビジネス用語に抽象化して提供するデータ基盤のベストプラクティス。
* **アーキテクチャとの関係**: 第5.2項の根拠。Tool層が物理テーブルではなく「意味付けされたビュー」のみを提供するアーキテクチャの正当性を担保する。

### IV. エンタープライズ・セキュリティと権限統制

**10. NIST SP 800-207: Zero Trust Architecture**
* **リンク**: [https://csrc.nist.gov/publications/detail/sp/800-207/final](https://csrc.nist.gov/publications/detail/sp/800-207/final)
* **概要**: 米国国立標準技術研究所（NIST）が定めるゼロトラスト・アーキテクチャの標準定義。「ネットワークの境界ではなく、アイデンティティ（ユーザー権限）に基づくアクセス制御」を大原則とする。
* **アーキテクチャとの関係**: Tool層（MCPサーバー）がIdPを介してユーザー個人の認証トークンを中継し、バックエンドに権限判定を委ねるというセキュアな「User Context Forwarding」設計のコンプライアンス要件。
