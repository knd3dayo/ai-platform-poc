## 検証目的

本検証の主目的は、サブ課題 A-01-01「型選択基準の定義」について、PoC 環境に存在する実装をもとに、WF型 / SV型 / 自律型の使い分け基準を明文化できるかを確認することである。

## 検証で確認したいこと

### 1. 正常系

- WF型、SV型、自律型のそれぞれに対応する実装入口が存在すること。
- 代表的な要求を、予見性、承認頻度、探索性の観点でいずれかの型へ割り当てられること。
- WF型の PoC 実装が LangGraph workflow として成立していること。

### 2. 異常系

- 型選択基準が曖昧なために、同じ要求が複数の型にまたがって解釈される箇所がないか。
- 現行実装で、WF型 / SV型 / 自律型の入口選択が暗黙の運用に依存していないか。
- routing 設定が存在しても、それが A-01-01 の「型選択基準」そのものを代替していないこと。

### 3. 運用系

- どの基準でその型を選んだかを、設定・文書・ログのいずれかで追跡できること。
- 新しいユースケースを追加した際に、どの型へ載せるべきかを再利用可能な判断表として整理できること。

## 現時点の実装確認結果

### 1. WF型

- `ai-chat-util` には `run_workflow` / `run_workflow_durable` の CLI 入口があり、LangGraph ベースの workflow 実行が可能である。
- `run_mermaid_workflow_from_file` は Markdown ファイルを読み込み、`execute_workflow_markdown` を通じて LangGraph 実行へ接続している。
- Durable 実行と plan mode も存在し、承認待ちを含む WF 型の PoC 検証は可能である。

### 2. SV型

- `agent_chat` は supervisor 型の入口として存在し、route 決定後に tool agents を束ねて結果を返す構成になっている。
- `AgentClientUtil.create_workflow()` は supervisor 配下の sub agents を組み立てる実装を持つ。
- `coordinated_chat` が SV 型を選んだ場合も、最終的には supervisor 入口へ接続される。
- 既存の structured routing は、SV 型の内部で `coding_agent` / `general_tool_agent` / `deep_agent` を選ぶための仕組みであり、cross-type Coordinator と役割分担している。

### 3. 自律型

- `coding_agent` route と `deep_agent` route が存在し、探索的な調査や複数ステップ処理の委譲先として扱える。
- `${HOME}/source/repos/ai-platform-poc/infra/31-ai-chat-util-mcp/ai-chat-util-config.structured-routing.poc.yml` では `routing_mode: structured`、`enable_deep_agent: true`、`preferred_coding_route: deep_agent` が定義されている。
- `coordinated_chat` では、明示的な coding/deep 指示や `exploration_level=high` を使って自律型候補を上位判定できる。

### 4. Coordinator / cross-type 監査

- `run_coordinated_chat` は `predictability`、`approval_frequency`、`exploration_level`、`has_side_effects`、workflow 定義有無、明示的な coding/deep 指示を使って WF型 / SV型 / 自律型を決定する。
- 上位型選択ポリシーは `type_selection_mode`、`type_selection_workflow_on_high_predictability`、`type_selection_autonomous_on_high_exploration` などの `type_selection_*` 設定として外だしされている。
- 型選択時には `cross_type_route_decided` 監査イベントが出力され、policy snapshot、候補ルート、request hints、selected backend を記録できる。
- 候補スコア差が小さい場合や WF 型が適しそうだが workflow 定義が不足する場合は clarification を返す制御も実装されている。

## 不足している実装 / 今後の論点

1. generic request から workflow 定義へ自動マッピングする仕組みは未実装

- `coordinated_chat` による上位型選択自体は実装済みである。
- 一方で、WF型を選ぶには既定では `workflow_file_path` の指定が必要であり、「汎用要求だけを受けて適切な workflow 定義へ自動接続する」運用はまだ未実装である。

2. Coordinator は既定で disabled であり、導入時の設定反映が必要

- `type_selection_*` の設定項目は `app/ai-chat-util-config.yml` と README に追加されている。
- ただし既定値は `type_selection_mode: disabled` のため、PoC 環境で有効化する設定反映と運用ルール整備が必要である。

3. 実トラフィックでの end-to-end 検証は別途必要

- 今回の再検証では、Coordinator の単体テストとエントリポイントテストを実行し、cross-type 判定・監査・clarification の成立を確認した。
- 一方で、LiteLLM、MCP、workflow 定義、外部依存を含む end-to-end の実行確認はこの文書では未再実施であり、別サブ課題または PoC 実施時の確認が必要である。

4. workflow 資産の管理規約は今後の運用課題

- A-01-01 のポリシーと Coordinator は整備されたが、どのユースケースにどの workflow 定義を紐付けるかという資産管理ルールは別途必要である。

## A-01-01 としての暫定判定基準

| 型 | 適用すべき要求の特徴 | 現行 PoC での入口 |
| --- | --- | --- |
| WF型 | 手順が事前定義できる。分岐条件や承認ポイントを Mermaid / Markdown で固定できる。 | `run_workflow` / `run_workflow_durable` / `coordinated_chat` + `workflow_file_path` |
| SV型 | 単一手順では固定しにくいが、tool 選択、結果評価、追加照会、HITL の統制が必要。 | `agent_chat` / `coordinated_chat` |
| 自律型 | 解決手順が探索的で、ワークスペース調査、複数段の試行錯誤、長いコンテキスト処理が必要。 | `agent_chat` 内の `deep_agent` または `coding_agent` route / `coordinated_chat` |

この暫定判定基準は、PoC 文書上の説明に加えて、`coordinated_chat` の deterministic policy として実装されている。したがって A-01-01 は「型選択基準を実装付きで定義できた」段階まで進んだと判断できる。一方で、workflow 定義の自動割当や実トラフィックでの運用検証は今後の課題として残る。

### 5. 型選択基準の整理

上記 2 から 4 の結果を使い、次を表に落とす。

- 手順が固定できる要求は WF型
- 結果評価や HITL を伴う要求は SV型
- 探索的で複数段の調査が必要な要求は自律型

## 判定基準

| 観点 | 判定基準 |
| --- | --- |
| 構造成立性 | WF型、SV型、自律型に対応する実装入口が存在し、それぞれの責務差分を説明できる。 |
| 制御成立性 | 予見性、承認頻度、探索性の 3 観点で、代表要求を矛盾なく各型へ割り当てられる。 |
| 運用成立性 | 型選択基準を設定または文書として再利用可能な形で残せる。不足実装が明示され、次の実装課題へ接続できる。 |

## 残課題

- `type_selection_mode: deterministic` を PoC 設定へどう反映するか、既定無効のままにするかを運用判断する必要がある。
- WF型の workflow 定義ファイルを、ユースケース単位でどう管理し、どの要求に紐付けるかの運用設計が必要である。
- generic request から workflow 定義を自動発見・選択する仕組みを導入するかを判断する必要がある。
- end-to-end の PoC 実行で、実際の LLM/MCP 接続下でも型選択と監査が期待どおり動くことを別途確認する必要がある。
