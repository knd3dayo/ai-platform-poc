# T-02-01_ビジネス概念出力とメタデータ付与の検証

## 検証目的

本検証の主目的は、サブ課題 T-02-01「ビジネス概念での出力設計」および T-02-02「出典・制約メタデータ付与」について、support-desk-agent の実装をもとに、Tool 層がビジネスドメイン概念で構造化された出力を返し、根拠データと出典メタデータを付与できることを確認することである。

最終的には、T-02 の完了判定に必要な材料として、正常系、代表的な異常系、運用上の制約を明確にすることを目指す。

## 検証で確認したいこと

### 1. 正常系

- `WorkflowKind`（`specification_inquiry` / `incident_investigation` / `ambiguous_case`）でワークフローが分類され、分類に応じたラベルと計画ステップが返ること。
- `_DeepAgentSearchResponse` の各 `_DeepAgentSourceResult` に `evidence` フィールドが含まれ、根拠原文断片が保持されること。
- `knowledge_retrieval_adopted_sources` に採用した出典パスが記録されること。

### 2. 異常系

- 検索対象ドキュメントが存在しない場合、`status: "unavailable"` として返り、生データを捏造しないこと。
- Tool が生 API や生データに直接接触せず、MCP abstraction 経由でのみ外部データを取得すること。

### 3. 運用系

- `WORKFLOW_LABELS` の日本語ラベルと `build_plan_steps()` の内容がドメイン概念（仕様調査 / 障害調査 / 判定困難）と対応していること。
- `feature_bullets` は「機能一覧問い合わせ」の時のみ付与され、不要な項目が混入しないこと。

## 現時点の実装確認結果

### 1. ビジネス概念分類

- `WorkflowKind` は `"specification_inquiry"` / `"incident_investigation"` / `"ambiguous_case"` の 3 値 Literal 型として定義されている。
- `route_workflow()` がプロンプトのキーワードマッチで `WorkflowKind` を決定し、CaseState に記録する。
- `WORKFLOW_LABELS` で日本語表示名（`仕様調査ワークフロー` 等）にマッピングされる。

### 2. 構造化出力

- `build_plan_steps(workflow_kind)` は分類に応じた 3 ステップの計画をリストで返す。  
  例（`incident_investigation`）:
  1. 問い合わせ内容と workspace から障害兆候を把握する
  2. ログ解析とナレッジ探索を並列で実施する
  3. 原因仮説と暫定対応を統合して回答ドラフトを作成する
- `summarize_plan()` は分類ラベルを含む 1 文サマリーを返す。

### 3. 出典・証拠メタデータ

- `_DeepAgentSourceResult` は `source_name` / `status` / `summary` / `matched_paths` / `evidence` / `feature_bullets` を持つ。
- `evidence` には根拠原文断片を入れ、`raw_content` は `raw_backend` モード時のみ本文を保持する（通常は空文字）。
- `knowledge_retrieval_adopted_sources` には採用した出典パスが CaseState として蓄積され、最終レポートで参照される。
- `_collect_adopted_sources()` が workspace の `evidence/` / `extracted/` ディレクトリや ticket context ラベルも採用ソース候補として収集する。

## 前提条件

## 検証手順

### 1. 事前準備

```bash
cd ${HOME}/source/repos/support-desk-agent
uv sync
```

### 2. ワークフロー分類の動作確認

```bash
cd ${HOME}/source/repos/support-desk-agent
uv run python -c "
from support_desk_agent.workflow.router import route_workflow, build_plan_steps, summarize_plan
kind = route_workflow('ログに NullPointerException が出ており障害が発生しています')
print('kind:', kind)
print('plan_steps:', build_plan_steps(kind))
print('summary:', summarize_plan(kind))
"
```

期待結果:

- `kind: incident_investigation`
- `plan_steps` に 3 要素のリストが返る。

### 3. 構造化検索出力の確認

```bash
cd ${HOME}/source/repos/support-desk-agent
uv run python -m pytest tests/ -v -k "search_document or knowledge"
```

期待結果:

- `evidence` フィールドを含む `_DeepAgentSourceResult` オブジェクトが正しくシリアライズされる。
