# G-02-01_根拠データ取得の組み込みの検証

## 検証目的

本検証の主目的は、サブ課題 G-02-01「根拠データ取得の組み込み」について、support-desk-agent の実装をもとに、調査エージェントが回答生成の前に構造化されたナレッジドキュメントを取得し、その結果を CaseState に記録できることを確認することである。

最終的には、G-02 の完了判定に必要な材料として、正常系、代表的な異常系、運用上の制約を明確にすることを目指す。

## 検証で確認したいこと

### 1. 正常系

- `KnowledgeDocumentSource` に設定されたパスのドキュメントを `default_search_documents` ツールが取得できること。
- 取得結果が `knowledge_retrieval_results` として CaseState に記録されること。
- 採用した出典が `knowledge_retrieval_adopted_sources` に追記されること。
- `knowledge_retrieval_final_adopted_source` に最終採用ソースが記録されること。

### 2. 異常系

- ドキュメントソースが存在しない場合、`status: "unavailable"` を返し、根拠なき回答を生成しないこと。
- LLM が検索結果なしに事実を捏造しないよう、プロンプトに「事実を捏造しないこと」の制約が含まれていること。

### 3. 運用系

- `config.yml` の `agents.InvestigateAgent.document_sources` でナレッジソースを追加・変更できること。
- `GET /ui-config` の `knowledge_sources` フィールドで利用可能なソース一覧を API 経由で確認できること。

## 現時点の実装確認結果

### 1. ナレッジドキュメント取得

- `KnowledgeDocumentSource` は `name` / `description` / `path` を持ち、`DocumentSourceSettings` に複数ソースを定義できる。
- `_invoke_deepagents_search()` はソース一覧とクエリを受け取り、DeepAgents 互換エージェントが Filesystem tools で各ソースを検索する。
- 検索プロンプトに「事実を捏造しないこと」の制約が明示されており、hallucination を instruction レベルで抑制している。

### 2. 構造化取得結果

- `_DeepAgentSearchResponse.results` は `_DeepAgentSourceResult` のリストで返される。
- 各結果に `source_name` / `status` / `summary` / `matched_paths` / `evidence` が含まれる。
- `extraction_mode="relaxed"` では広めの関連語探索、`extraction_mode="raw_backend"` では生テキストも保持する。

### 3. CaseState への記録

- 取得結果は `knowledge_retrieval_results` に JSON シリアライズして保存される。
- 採用ソースのパスは `knowledge_retrieval_adopted_sources` にリストとして追記される。
- `knowledge_retrieval_final_adopted_source` に最終的に根拠として採用したソースが記録される。
- これらは `_collect_adopted_sources()` によって最終レポート生成時にも参照される。

### 4. UI-config 公開

- `GET /ui-config` の `knowledge_sources` フィールドに `name` / `path` のペアが返り、フロントエンドがソース一覧を表示できる。

## 前提条件

- `support-desk-agent` の依存が導入済みであること（`uv sync`）。
- `config.yml` の `agents.InvestigateAgent.document_sources` にナレッジソースが定義されていること。


- `knowledge_sources` に設定済みのナレッジソース名とパスが返ること。
