## 検証で確認したいこと

### 1. 正常系

- WF型実行中に approval や入力待ちが発生した場合、同一 trace_id で pause / resume できること。
- 上位の型選択が競合した場合、Coordinator が clarification を返して人間入力へ昇格できること。
- SV型配下で要承認ツールや情報不足が発生した場合、即時に誤実行せず HITL へ切り替えられること。

### 2. 異常系

- 型間エスカレーションの名前で、実際には単なる個別 route 内 pause/resume しかできていない箇所がないか。
- WF型から SV型、SV型から WF型のような cross-type の自動再ルーティングが未実装であるにもかかわらず、成立済みと誤認しないこと。
- 曖昧な要求で workflow 定義がない場合に、無理に誤った型へ進まず clarification を返せること。

### 3. 運用系

- pause / clarification / approval の判断を trace_id 単位で追跡できること。
- どの段階で人間入力へエスカレーションしたかを、CLI / API / 監査ログのいずれかで確認できること。
- 自動 reroute 未実装の部分が、次に A-01-03 として残る課題として明示されていること。

## 現時点の実装確認結果

### 1. Coordinator による型選択競合時の昇格

- `run_coordinated_chat` は、workflow と autonomous など複数候補のスコア差が小さい場合に `requires_clarification=true` とし、`ChatResponse.status="paused"` と `hitl.source="coordinator:routing"` を返す。
- これは、cross-type の自動確定が難しいケースを人間入力へ昇格させる実装として成立している。
- `cross_type_route_clarification_requested` 監査イベントにより、なぜ昇格したかを trace_id 単位で追跡できる。

### 2. WF型内の approval / resume

- `WorkflowChatClient` は plan mode で updated Markdown を返し、承認後に同一 trace_id で実行を再開できる。
- durable workflow では graph 実行中の pause に対して `resume_value` を使って再開できる。
- これは WF型内の HITL 昇格としては成立しているが、「WF型では扱いきれないため SV型へ切り替える」自動 reroute ではない。

### 3. SV型内の HITL / approval

- supervisor 系は要承認ツールや clarification を `paused` 応答で返し、CLI / API から再入力できる。
- したがって、SV型から人間入力への昇格は成立している。
- 一方で、SV型が途中で WF型へ降格したり、WF型へ処理を載せ替えたりする共通制御は確認できていない。

### 4. cross-type 自動 reroute の不足

- 現状の `coordinated_chat` は初回の上位型選択を行うが、型を選んだ後に実行途中で別の型へ自動的に切り替える機構は確認できていない。
- 具体的には、次のようなケースは未実装である。
  - WF型で処理を開始したが、途中で探索性が高まったため SV型または自律型へ切り替える。
  - 自律型で処理を開始したが、手順が定型化できると分かったため WF型へ固定化する。
  - SV型で処理を開始したが、workflow 定義が明示されたため WF型へ委譲し直す。

## A-01-03 としての暫定判定

| 観点 | 現状評価 |
| --- | --- |
| 人間入力への昇格 | 成立している |
| 同一型内の pause / resume | WF型、SV型ともに一部成立している |

## 残課題

- WF型 / SV型 / 自律型の実行途中で、別の型へ自動的に切り替える共通制御を実装するかを判断する必要がある。
- cross-type reroute を実装する場合、どのシグナルで切替えるか、trace_id と監査イベントをどう接続するかを設計する必要がある。
- 現状の A-01-03 は、人間入力への昇格と同一型内再開までは扱えるが、型間の自動降格・昇格は残課題である。
