## 検証で確認したいこと

### 1. 正常系

- coding-agent 実行タスクを API / MCP / CLI から起動できること。
- `execute`、`status`、`cancel` などの基本契約が成立していること。
- 自律型の成果物回収、レビュー待ち、進捗確認へ接続できること。

### 2. 異常系

- 許可されていない workspace やネットワーク経路を拒否できること。
- 長時間実行や異常終了時に status / cancel で制御できること。
- coding-agent 経路を通常の同期ツールのように扱わないこと。

### 3. 運用系

- task_id、trace_id、workspace_path を追跡できること。
- Docker / Proxy / LiteLLM Hook を含む多層防御を前提に運用できること。
- SV 型からの呼び出し契約と、自律型側の task 契約を分離して扱えること。

1. ai-chat-util 側の入口

3. 自律型としての成立要件

- task 指向の契約であること。
- status / cancel を備えた非同期ジョブ型であること。
- サンドボックスとレビュー待ちを前提に扱うこと。

結果:

- `11 passed in 5.52s`
- `execute` / `status` / `cancel` / `workspace_path` / `get_result` の MCP 公開契約が維持されていることを確認した。
- process backend 系の設定正規化も確認した。
- `coding-agent-util run` が config に応じて backend factory を使うこと、`prune` が Docker backend 以外では拒否されることを確認した。
- `TaskManager.cancel_task()` が `process` backend を正しく受理すること、および cancel 後の status 再評価で `cancelled` を保持する単体テストを追加した。

API 正常系実測:

```bash
cd ${HOME}/source/repos/ai-chat-util/app
AI_PLATFORM_TASK_BACKEND=process \
uv run coding-agent-api \
  --config /home/user/source/repos/ai-platform-poc/infra/31-ai-chat-util-mcp/ai-chat-util-config.coding-only.poc.yml \
  --host 127.0.0.1 -p 7111

curl -sS http://127.0.0.1:7111/healthz
curl -sS -X POST http://127.0.0.1:7111/execute \
  -H 'Content-Type: application/json' \
  -H 'X-Trace-Id: a0405-trace-001' \
  --data '{"prompt":"作業ディレクトリを確認し、ファイルは変更せず、現在の workspace_path と対象ディレクトリ名だけを短く報告してください。","workspace_path":"/home/user/source/repos/ai-platform-poc/tmp/a0405-workspace","timeout":120,"task_id":"a0405-task-001"}'

curl -sS 'http://127.0.0.1:7111/status/a0405-task-001?wait_seconds=8&tail=80'
curl -sS 'http://127.0.0.1:7111/get_result/a0405-task-001?wait_seconds=2&tail=120'
```

API 実行結果:

- `/healthz` は `{"status":"ok"}` を返した。
- `/execute` は `{"task_id":"a0405-task-001"}` を返した。
- 直後の `/status` では次を確認した。
  - `trace_id: a0405-trace-001`
  - `workspace_path: /home/user/source/repos/ai-platform-poc/tmp/a0405-workspace`
  - `status: running`
  - `sub_status: running-background`
  - `metadata.backend: process`
- 収束後の `/status` は `status=exited`、`sub_status=completed` を返し、artifact 一覧と stdout / stderr を保持していた。
- `/get_result` では次の stdout を取得できた。

```text
**作業ディレクトリ:** /home/user/source/repos/ai-platform-poc/tmp/a0405-workspace

**対象ディレクトリ名:** a0405-workspace
```

CLI 正常系実測:

```bash
cd ${HOME}/source/repos/ai-chat-util/app
uv run coding-agent-util \
  --config /home/user/source/repos/ai-platform-poc/infra/31-ai-chat-util-mcp/ai-chat-util-config.coding-only.poc.yml \
  run --no-wait --src /home/user/source/repos/ai-platform-poc \
  "作業ディレクトリを確認し、ファイルは変更せず、現在の workspace_path と対象ディレクトリ名だけを短く報告してください。"
```

CLI 実行結果:

- backend factory 利用へ修正後、CLI 直叩きでも process backend で task を起動できた。
- 実際に `status f7961091-e4ed-4cf2-989c-4b20d8c67783` で確認可能な task_id が返り、以前の `docker-compose.yml` 不在エラーは解消した。
- したがって、PoC の coding-agent 経路は API / CLI の双方で process backend を前提に起動成立を確認した。

cancel の観測:

- 修正前は 2 つの不具合があった。
  - `TaskManager.cancel_task()` が `backend == "subprocess"` のみを受理し、実際の `metadata.backend: process` を拾えていなかった。
  - CLI `cancel` が TaskManager の戻り値を捨て、固定で「キャンセルしました」と表示していた。
- これらは ai-chat-util 側で修正し、単体テストへ追加した。
- live 実測では、待機指示付き task に対して `status=running-background` まで到達したケースと、agent が待機を短絡して即時完了したケースが混在した。
- そのため、prompt 依存の実エージェント挙動まで含めた `running -> cancelled` の安定した再現証跡は未取得である。
- ただし cancel 契約そのものは API / MCP / CLI 実装と単体テストの両面で成立させた。

| 項目 | 結果 | 補足 |
| --- | --- | --- |
| 正常系 | 一部確認済み | API 経路では `execute` で task_id を返し、`status` と `get_result` で workspace_path、trace_id、成果物、stdout を取得できた。CLI 経路も backend factory 修正後は process backend で task 起動に成功した。 |
| 異常系 | 一部確認済み | `cancel_task()` の backend 判定不備と CLI cancel の false positive 表示を修正した。実エージェント prompt まで含めた live 停止再現は、task が即時完了するケースがあり未安定。 |

## 残課題

- prompt 依存ではない、十分に長い task を確実に生成できる cancel 検証シナリオを別途用意する必要がある。
- PR レビューや成果物承認をどの UI / BFF へ返すかは別課題である。
- 本文書では呼び出し基盤の成立性を扱い、個別コーディングエージェントの能力比較までは扱わない。
