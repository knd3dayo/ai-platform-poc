# O-04-03_セッション予算と再帰制御の検証

## 検証目的

本検証の主目的は、サブ課題 O-04-03「セッション予算と再帰制御」について、support-desk-agent の実装をもとに、調査ループ回数・ドラフトレビュー回数・LangGraph 再帰深度のそれぞれに上限が設定されており、無限実行に陥らないことを確認することである。

最終的には、O-04 の完了判定に必要な材料として、正常系、代表的な異常系、運用上の制約を明確にすることを目指す。

## 検証で確認したいこと

### 1. 正常系

- `SampleSupervisorAgent` の調査ループが `MAX_INVESTIGATION_FOLLOWUP_LOOPS`（= 1）で打ち切られること。
- `execute_draft_review()` が `config.constraints.agents.SuperVisorAgent.draft_review_max_loops` を参照して `draft_review_max_loops` を CaseState にセットすること。
- `GET /runtime-constraints` から制約値の一覧を参照できること。

### 2. 異常系

- `max_investigation_loops` を超えた追加調査要求が来ても、ループを継続せず break すること。
- `langgraph_recursion_limit` を超えるノード遷移が発生した場合、LangGraph が `GraphRecursionError` を送出すること。

### 3. 運用系

- `config.yml` の `constraints.agents.SuperVisorAgent.max_investigation_loops` でループ上限を変更できること。
- `GET /runtime-constraints` により現在適用中の制約値を API 経由で確認できること。

## 対象構成

| 論点 | 実装箇所 | 現状評価 |
| --- | --- | --- |
| 調査ループ上限 | `agents/sample/sample_supervisor_agent.py` の `MAX_INVESTIGATION_FOLLOWUP_LOOPS` と `execute_investigation()` の `while True` ループ | 実装あり |
| レビューループ上限 | `config/models.py` の `ConstraintsConfig.agents.SuperVisorAgent.draft_review_max_loops` と `execute_draft_review()` | 実装あり |
| LangGraph 再帰深度上限 | `util/langchain/agent_factory.py` の `recursion_limit=9_999` | 実装あり（既定値は高め） |
| 制約値の API 公開 | `interfaces/api.py` の `GET /runtime-constraints` | 実装あり |

## 現時点の実装確認結果

### 1. 調査ループ上限

- `SampleSupervisorAgent.MAX_INVESTIGATION_FOLLOWUP_LOOPS = 1` によりクラス定数として上限を固定している。
- `execute_investigation()` の `while True` ループでは `followup_loops >= self.MAX_INVESTIGATION_FOLLOWUP_LOOPS` の条件で `break` し、最大 2 回（初回 + followup 1 回）で調査を完了する。
- 評価スコアが `RESULT_PASS_SCORE`（= 80）以上であれば、上限に達する前にも早期終了する。

### 2. レビューループ上限

- `ConstraintsConfig.agents.SuperVisorAgent` に `draft_review_max_loops: int = 1` が定義されている。
- `execute_draft_review()` がこの値を CaseState の `draft_review_max_loops` にセットすることで、ワークフロー上のループ制御に引き渡す。

### 3. LangGraph 再帰制御

- `AgentFactory` の `graph.compile()` 時に `recursion_limit=9_999` を設定し、LangGraph デフォルト（25）を超える深いエージェント呼び出しに対応する。
- 上限値はハードコードであり、現時点では config での変更は不可。Tool call 数上限は別途未実装。

### 4. 制約値の API 公開

- `GET /runtime-constraints` は `describe_control_catalog()` の `runtime_constraints` キーから制約ポリシー一覧を返す。
- `RuntimeConstraintManager` に登録された Global / Role-Specific ポリシーが対象となる。

## 前提条件

- `support-desk-agent` の依存が導入済みであること（`uv sync`）。
- `samples/support-desk-agent/config-sample.yml` が配置済みであること。

### 1. 事前準備

```bash
cd ${HOME}/source/repos/support-desk-agent
uv sync
```

### 2. ループ上限の確認（ユニットテスト）

```bash
cd ${HOME}/source/repos/support-desk-agent
uv run python -m pytest tests/test_sample_investigate_agent.py -v -k "investigation"
```

### 3. runtime-constraints API の確認

```bash
# API 起動後に実行
curl -s http://localhost:8010/runtime-constraints | python3 -m json.tool
```

期待結果:

- `constraints.max_context_chars`、`constraints.max_investigation_loops` などの制約値が JSON 配列で返る。
