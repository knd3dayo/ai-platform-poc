# AIエージェントの業務適用を見据えた生成AIエンタープライズアーキテクチャ

## 1. はじめに

企業の業務には、ERP や SaaS、RPA によって長年システム化されてきた領域がある一方で、依然として人手に頼らざるを得ない業務が残っています。特に、調査、判断、起案、例外対応、複数システムをまたぐ登録や確認といった非定型業務は、従来の IT では費用対効果や統制の観点から自動化が難しい領域でした。

生成AIの登場により、この前提は大きく変わりました。LLM は単なる文章生成の道具ではなく、非定型情報の理解、根拠の収集、選択肢の比較、回答案の起案、ツールを用いた外部システム操作まで担えるようになっています。もちろん、承認や最終責任までを AI に委ねられるわけではありません。しかし、人間が最終判断を行う前提を維持しながら、その手前の探索、整理、起案、登録準備をシステム化することは現実的な選択肢になりました。

本書は、こうした変化を踏まえ、なぜ今までシステム化されていない業務が残っていたのか、なぜ生成AIによってそれらがシステム化可能になったのか、そしてそれを業務システムとして成立させるためにどのようなアーキテクチャ、課題整理、検証が必要かを簡潔に整理したものです。

## 2. なぜ今までシステム化されていない業務があったのか

従来の IT システムで取り残されてきた業務の背景には、大きく 4 つの壁がありました。

### 2.1 技術の壁

従来システムは、表形式や固定スキーマを前提とした構造化データの処理を得意としてきました。一方、実業務ではメール本文、PDF、ログ、議事録、問い合わせ文、社内ナレッジのように、形式も粒度もばらばらな非定型データを扱う場面が多くあります。人間は文脈を補って読めますが、従来のルールベースシステムはこの翻訳を苦手としていました。

### 2.2 投資の壁

業務の例外条件が多く、顧客や案件ごとに少しずつ処理が変わる領域では、If-Then の作り込みがすぐに高コスト化します。仕様化に手間がかかり、保守時の修正も連鎖的に膨らむため、業務上必要でも「システム化するほどではない」と判断され、長い間、人手運用のまま残されてきました。

### 2.3 統制の壁

対外送信、顧客回答、申請承認、更新系処理のように、誤ると影響が大きい業務では、単純な自動化だけでは責任の所在を明確にできません。最終的な Go/No-Go の判断は人間が持つべきであり、そこを機械に全面委任する設計は企業統制と相性が悪いという問題がありました。

### 2.4 暗黙知の壁

現場の業務には、ベテランの経験、文脈理解、過去事例の記憶、社内事情の読み替えといった暗黙知が深く入り込んでいます。これは単純な手順書に落とし込めず、そもそも要件定義が困難です。そのため、業務に詳しい人が都度判断しながら回すしかない状態が固定化されてきました。

以上の 4 つの壁により、企業には「業務として重要だが、従来 IT ではシステム化しきれない仕事」が大量に残り続けてきました。

## 3. 生成AIにより、なぜそれらの業務がシステム化可能になったのか

生成AIは、これら 4 つの壁を一足飛びに消す魔法ではありません。しかし、従来のシステムが苦手だった部分を担当させることで、業務をシステム化可能な形へ変換できるようになりました。

第一に、LLM は非定型データを読み、要点を抽出し、分類し、構造化する能力を持ちます。これにより、人間が翻訳機として介在していた工程の多くを機械化できます。これは技術の壁を崩す力です。

第二に、自然言語で振る舞いを定義できるため、例外処理が多い業務でも、すべてを厳密な分岐で作り込まずに一定の対応力を持たせられます。探索、比較、要約、起案のような工程を低コストで実装できるようになり、投資の壁を下げられます。

第三に、生成AIは「決定者」ではなく「起案者」「副操縦士」として組み込むことができます。人間が承認や最終判断を担い、AI はその前段で必要な調査、根拠整理、ドラフト作成、登録準備を行う構成をとることで、統制の壁を越えながら自動化の範囲を広げられます。

第四に、暗黙知を完全代替するのではなく、その手前の情報収集、過去事例検索、論点整理、候補案の提示を AI に担わせることで、人間が本当に必要な判断に集中できるようになります。暗黙知の壁は消えませんが、AI を挟むことで人間の判断負荷を大きく下げ、結果としてシステム化可能な範囲を広げられます。

つまり、生成AIは「人間の仕事をすべてなくす技術」ではなく、「これまで人間が担っていた探索、理解、起案、整理を肩代わりすることで、非定型業務をシステムの処理対象へ変える技術」と捉えるべきです。

## 4. 生成AI、AIエージェント、ツール、ガバナンスの進化の歴史

現在の生成AIシステムは、最初から完成された形で存在していたわけではありません。能力の進化とともに新しい課題が現れ、それに対応する形でアーキテクチャも進化してきました。

### 4.1 第1期: 単一 LLM の時代

初期の生成AIは、テキストを入力しテキストを返す単体の LLM が中心でした。この段階でも、文章生成、要約、分類、ドラフト作成といった価値は大きく、非定型業務の一部を支援できることが明らかになりました。

一方で、プロンプトインジェクション、機密情報送信、ハルシネーションといった、従来の WAF や IAM では十分に扱えない脅威も同時に顕在化しました。この時点で、自然言語入出力を監視する AI ガバナンスの必要性が生まれています。

### 4.2 第2期: AIエージェントとツール利用の時代

LLM が外部システムや API、ファイル、検索機能を使って行動する AI エージェントへ進化すると、AI は単なる回答生成装置ではなく、業務を進める主体になり始めました。これにより、情報取得、比較、チケット起票、更新準備など、これまで人手でつないでいた工程をまとめて処理できる可能性が高まりました。

その反面、生の API や DB を直接触らせると、引数の取り違え、誤更新、不要なデータ参照、ハルシネーションによる誤操作が起きやすくなりました。この反省から、AI が外界へ触れる経路を標準化し、思考と実行を分離する Tool 層の考え方が強く求められるようになりました。

### 4.3 第3期: ハイブリッドな Application 層の時代

エージェントが複数ステップの思考と行動を自律的に回すようになると、微小な誤りが連鎖的に増幅するカスケード障害や、長時間実行、状態喪失、承認待ちによるタイムアウトが実務上の問題になりました。ここで重要になったのが、「すべてを自律型で動かさない」という設計思想です。

定型工程はワークフロー型で固定し、合議や判断が必要な難所はスーパーバイザー型で制御し、探索的な処理だけを自律型ワーカーへ委譲する。こうしたハイブリッドな Application 層の考え方が定着し、あわせて Checkpointer、Pause/Resume、非同期 Human-in-the-Loop などのハーネスが不可欠になりました。

### 4.4 第4期: ガバナンスの高度化

AI がより広い権限を持ち、複数エージェントが連携するようになると、脅威も高度化します。プロンプトインジェクション、ハルシネーション、間接インジェクション、情報漏洩、カスケード障害、エージェント間の権限昇格や汚染伝播といった問題は、個別アプリだけで対処できません。

そのため、入力・出力 Guardrails、グラウンディング、サイテーション、LLM-as-a-Judge、Risk-Adaptive HITL、キルスイッチ、Observability といった共通統制を担う AI ガバナンス層が必要になりました。能力が増えるほどガバナンスも独立レイヤーとして強化されるというのが、この進化史の本質です。

## 5. 生成AIシステムアーキテクチャの3つの層

以上の歴史を踏まえると、生成AIを業務へ適用するシステムは、少なくとも 3 つの層に分けて考える必要があります。重要なのは、思考、実行、統制を混ぜないことです。

### 5.1 Application 層

Application 層は AI の脳に相当し、業務目標をどのように分解し、どこまでを固定フローで実行し、どこからを AI に委譲するかを決める層です。ここでは、予見可能な処理を WF 型で、合議や評価を伴う処理を SV 型で、探索的な処理を自律型で扱います。

また、高リスク業務では人間の承認を織り込んだ非同期 HITL が不可欠です。AI が処理途中で承認待ちになる場合、状態を永続化して安全に待機し、承認後に再開できる設計を取らなければ業務運用に耐えません。

### 5.2 Tool 層

Tool 層は AI の手足であり、社内システム、SaaS、ファイル、検索、データ基盤との接続を標準化された形で提供する層です。ここでは、AI に個別 API の叩き方を覚えさせるのではなく、MCP のような標準インターフェースでツールをカプセル化し、再利用性と置換容易性を確保します。

Tool 層はステートレスであることが重要です。人間の承認待ちや長時間の状態管理を持たず、即座に結果または Job ID を返す設計とします。また、実行ユーザーの権限をバックエンドまで継承し、セマンティックレイヤーを介して意味付けされたデータだけを AI に渡すことで、安全性と品質を担保します。

### 5.3 AIガバナンス層

AIガバナンス層は、業務ロジックとは独立して、全社共通の統制を担う層です。入力・出力 Guardrails、モデル利用統制、グラウンディング、サイテーション、Tool 利用上限制御、共通評価、監査ログ、キルスイッチなどをここで集中的に扱います。

この層の役割は、個別アプリが自前で安全性を抱え込まないようにすることです。AI の自然言語処理やエージェント的振る舞いに伴うリスクは、アプリごとに都度実装していては品質も統制水準も揃いません。だからこそ、意味論的な統制を共通レイヤーとして独立させる必要があります。

### 5.4 3層分離の意味

Application 層は不確実な思考を扱い、Tool 層は確実な実行を担い、AIガバナンス層は全体を横断して意味論的な統制をかける。この分離により、変更の種類ごとに責務を切り離し、再利用性、説明責任、安全性、運用性を両立できます。

## 6. 実現に向けた技術的な課題

3 層アーキテクチャが妥当であっても、実装上は複数の技術課題を解かなければなりません。重要なのは、個別アプリのローカル実装に閉じず、できるだけ基盤側で透過的に解決することです。

### 6.1 認証と認可

AI エージェントが誰の代理で処理しているのかを全レイヤーで一貫して扱う必要があります。アプリ認可、機能認可、データ認可を分離し、システム用キーとユーザー個人トークンを適切に使い分けることが必要です。

### 6.2 AIガバナンスの実装

Guardrails や機密情報検知、送信前遮断、事後評価をどの地点で実装するかを明確にしなければなりません。生成結果だけでなく、ツール呼び出しやモデル利用経路まで含めた統制設計が必要です。

### 6.3 非同期化と Human-in-the-Loop

長時間処理や承認待ちを同期通信のまま扱うと、すぐにタイムアウトや状態喪失が発生します。イベント駆動、Webhook、Checkpointer、Pause/Resume を前提とした非同期アーキテクチャが必要です。

### 6.4 E2E トレースと Observability

生成AIシステムでは、最終回答だけ見ても問題の原因を追えません。どのプロンプト、どの根拠、どのツール、どの判断で結果に至ったのかを再構成できるよう、trace_id を軸に全層を串刺しにして観測する必要があります。

### 6.5 データの意味付け

AI に生のテーブルや複雑な API をそのまま渡すと、ハルシネーションや誤操作が起きやすくなります。スキーマの説明、セマンティックレイヤー、カスタムインストラクションを組み合わせ、AI が理解できる意味単位へ変換する必要があります。

### 6.6 自律型ワーカーの隔離

探索的な調査やコード実行を伴う自律型ワーカーは、高い能力と引き換えに逸脱リスクも高くなります。そのため、サンドボックス、ネットワーク制御、外部送信制限、短命コンテナなどによる物理的な封じ込めが必要です。

### 6.7 Write 系ツールの安全化

更新、送信、起票、登録のような不可逆操作は、Read 系ツールと分離し、Dry Run、承認、最終ゲートを設ける必要があります。AI にいきなり本番更新を許すのではなく、計画と実行を分離した設計が前提です。

## 7. 技術知見を得るための検証

上記の課題は机上で解けるものではなく、実機検証を通じて構成の有効性を確かめる必要があります。本 PoC では、代表的に次のような検証を進めています。

### 7.1 セキュアなコーディングエージェント利用の検証

[docs/11_技術課題検証/A-03-01_Docker_Proxy環境によるセキュアなコーディングエージェント利用の検証.md](../11_技術課題検証/A-03-01_Docker_Proxy環境によるセキュアなコーディングエージェント利用の検証.md) では、Docker による作業領域の限定、Squid Proxy による外向き通信の限定、LiteLLM Proxy のフックによる機密情報送信の遮断を組み合わせ、AI エージェントを安全側に倒して利用開始できるかを検証しています。これは、自律型ワーカーの隔離、AI ガバナンス層の入力ガード、多層防御の実証に対応します。

### 7.2 コーディングエージェントの MCP サーバー化検証

[docs/11_技術課題検証/T-01-01_コーディングエージェントのMCPサーバー化検証.md](../11_技術課題検証/T-01-01_コーディングエージェントのMCPサーバー化検証.md) では、スーパーバイザーからコーディングエージェントを MCP サーバーとして利用し、通常ツールと複数ステップ処理向けツールを分離しながら委譲できるかを確認しています。これは、Application 層と Tool 層の責務分離、および標準化されたツール利用の有効性を検証するものです。

### 7.3 スーパーバイザーのツール選択と結果判断の検証

[docs/11_技術課題検証/A-01-02_スーパーバイザーのツール選択とMCP結果判断の検証.md](../11_技術課題検証/A-01-02_スーパーバイザーのツール選択とMCP結果判断の検証.md) では、スーパーバイザーが利用可能なツール一覧から適切な委譲先を選び、結果が十分かどうかを判定し、不足時には追加照会または HITL に進めるかを検証しています。これは、SV 型の評価ユニット、結果判断、証跡性、HITL 遷移の要件に対応します。

これらの検証は、単一機能の動作確認ではなく、3 層アーキテクチャを現実の業務システムへ落とす際に必要な設計原則が実装可能かどうかを確かめるためのものです。

## 8. シナリオベースの検証

技術検証だけでは、業務適用時の価値や責務分界は見えません。そのため、業務シナリオを用いた検証も必要です。本 PoC では、カスタマーサポート業務を代表例として、As-Is と To-Be を比較しながら検証を進めています。

### 8.1 As-Is: 従来のカスタマーサポート

顧客から障害問い合わせと大容量ログが届くと、担当者はチケットを確認し、内部調査チケットを手動で起票し、ログを目視で調べ、過去事例や社内ナレッジを検索し、原因仮説と回答案を時間をかけて作成していました。この流れは、技術の壁、投資の壁、暗黙知の壁、統制の壁が同時に表れる典型例です。

### 8.2 To-Be: 生成AIを組み込んだ支援フロー

問い合わせ受付後、WF 型の処理が即座に起動し、内容のトリアージ、PII のマスキング、内部チケット起票を実施します。次に、SV 型がログ解析やナレッジ探索を配下のワーカーへ委譲し、根拠を集めながら原因仮説と回答ドラフトを作成します。最後に、人間が内容を確認し、承認後にのみ対外送信やチケット更新が実行されます。

この構成では、初動の速さと調査支援は AI が担い、最終判断と責任は人間に残ります。つまり、業務品質と統制を両立した形でシステム化を実現できます。

### 8.3 どの層がどの役割を担うか

このシナリオでは、Application 層が全体の進行管理、調査委譲、回答ドラフト作成、承認待ちの状態管理を担います。Tool 層は Zendesk、Redmine、ナレッジベース、ログ解析環境との接続を提供します。AI ガバナンス層は PII マスキング、根拠性チェック、出力統制、最終送信前の安全確認を担います。

業務シナリオに落とし込むことで、3 層アーキテクチャが抽象論ではなく、どの工程をどの仕組みで支えるかという具体像として理解できるようになります。

## 9. まとめ

企業に未システム化業務が残っていた理由は、従来 IT では非定型データ、例外処理、統制、暗黙知を十分に扱えなかったためです。生成AIは、それらを完全に置換するのではなく、探索、理解、要約、起案、構造化を肩代わりすることで、これまで人手でしか回らなかった業務をシステムの処理対象へ変えつつあります。

そのために必要なのが、Application 層、Tool 層、AI ガバナンス層からなる 3 層アーキテクチャです。これは単なる設計上の好みではなく、LLM、AI エージェント、ツール利用、ガバナンスの進化の結果として生まれた必然的な構造です。

ただし、業務適用には認証・認可、非同期 HITL、Observability、意味付け、サンドボックス、Write 系安全化といった技術課題があり、それらは実機検証とシナリオ検証を通じて段階的に詰めていく必要があります。

生成AIの企業適用は、単なるモデル導入では完結しません。業務、基盤、統制を一体で設計し、人間の責任を残したまま自動化可能な範囲を広げていくことが、これからのエンタープライズアーキテクチャの中核になります。
