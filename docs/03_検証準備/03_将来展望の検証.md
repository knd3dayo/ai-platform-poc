# 将来展望の検証

## 0. この文書の位置づけ

本書は、[docs/01_アーキテクチャ検討/11_【別紙】将来展望：セマンティックレイヤーを活用した動的ツール生成と自己進化型AIエージェントへの接続.md](/home/user/source/repos/ai-platform-poc/docs/01_アーキテクチャ検討/11_【別紙】将来展望：セマンティックレイヤーを活用した動的ツール生成と自己進化型AIエージェントへの接続.md) をもとに、将来的な動的ツール生成と自己進化型AIエージェントに関する検証ケースを定義する文書である。

ここで扱う内容は、現行アーキテクチャの必須要件をただちに本番導入するための実装計画ではなく、将来的な拡張可能性を見極めるための探索的PoCである。そのため、既存の Application層 / Tool層 / AIガバナンス層の責務を崩さず、段階的に「どこまで安全に自動生成を許容できるか」を確認することを主眼に置く。

---

## 1. 検証の狙い

将来展望の検証では、次の4点を確認する。

1. セマンティックレイヤーやデータカタログのメタデータを、ローカルファイルや既存コードの代替となる実用的なコンテキストとして使えるか。
2. AIがそのメタデータから、単発のSQLだけでなく再利用可能なプログラムやツールを安全に生成できるか。
3. 生成物をMCPツールとしてカプセル化した場合でも、既存の統制原則（ステートレス実行、権限継承、Dry Run、監査）を維持できるか。
4. 生成されたツール群を前提に、上位のAIエージェント定義まで自動生成した場合でも、暴走を抑えながら再利用可能な単位として運用できるか。

---

## 2. 検証の前提

### 2.1 前提とするアーキテクチャ原則

本検証では、将来志向のテーマであっても、以下の原則は固定とする。

* Tool層の実行はステートレスとし、長時間待機や承認待ちは Application層で扱う。
* 実行時の認証・認可は実行ユーザー文脈を継承し、固定権限の裏口を作らない。
* 生成コードは隔離環境でDry RunまたはRead-Only相当の検証を経てから実行する。
* 生成されたSQL、プログラム、MCP定義、エージェント定義はすべて監査可能な成果物として保存する。
* 生テーブルや生APIを直接AIへ露出せず、セマンティックレイヤーまたはメタデータAPI経由の情報のみを主要コンテキストとする。

### 2.2 PoCで用意する最低限の入力資産

各検証ケースでは、少なくとも以下を準備する。

* 業務用語で意味付けされたデータセット定義
* 指標定義、ディメンション定義、利用制約を含むメタデータ
* 正答確認用の期待結果セット、または人手で妥当性を確認できる比較用クエリ
* 実行ログ、生成コード、評価結果を紐付ける `trace_id`
* 失敗時の遮断条件とレビュー手順

製品はDenodo、dbt Semantic Layer、Unity Catalog等を想定するが、PoC初期段階では同等のメタデータ構造を模した簡易実装または静的サンプルでもよい。ただし、物理スキーマを直接プロンプトへ貼り付けるだけの擬似検証は対象外とする。

## 3. 共通の評価観点

全検証ケースで共通して、以下の観点を評価する。

| 評価観点 | 確認内容 |
| --- | --- |
| 正確性 | 生成物が期待する業務定義、指標定義、抽出条件を満たしているか。 |
| 根拠性 | どのメタデータ、どの制約、どの出典に基づいて生成したかを追跡できるか。 |
| 安全性 | 権限逸脱、禁止テーブル参照、危険な外部通信、破壊的操作を抑止できるか。 |
| 再現性 | 同じ入力条件で同等の成果物を再生成し、失敗時の原因を分析できるか。 |
| 再利用性 | 一度生成した成果物を別タスクでも流用できるか、またはキャッシュ対象にできるか。 |
| 統制可能性 | 既存のAIガバナンス層、監査ログ、承認フローへ接続できるか。 |

## 4. 検証ケース一覧

| 検証ケース | 段階 | 主題 | 成果物 | 次段階へ進む判断 |
| --- | --- | --- | --- | --- |
| JC-01 | 第1段階 | メタデータからの自動SQL生成 | SQL、根拠メタデータ、評価結果 | SQL生成が安定し、禁止経路を踏まずに期待精度へ到達できること |
| JC-02 | 第2段階 | メタデータからのプログラム生成 | Python等の実行コード、実行ログ、出力データ | 単発SQLより複雑な加工処理でも安全に生成・実行できること |
| JC-03 | 第3段階 | 生成コードの自動MCPツール化 | MCPスキーマ、description、ツール実装、Dry Run結果 | 生成物を標準インターフェースへ包み、既存統制へ接続できること |
| JC-04 | 第4段階 | 動的ツールを使う専用AIエージェント生成 | エージェント定義、オーケストレーション、評価ログ | ツール生成から利用までを一連で自律化しても停止条件を保てること |

## 5. 検証ケース詳細

### 5.1 JC-01: 整備済みメタデータからの自動SQL生成

#### 目的

セマンティックレイヤーやデータカタログの情報だけを主要な手掛かりとして、AIが物理テーブルを直接探索せずに妥当なSQLを生成できるかを確認する。

#### 想定シナリオ

* ユーザー要求例: 「直近3か月の主要製品カテゴリ別売上と粗利率を、地域別に比較したい」
* 利用コンテキスト: 指標定義、ディメンション定義、利用可能なGold層データセット、鮮度情報、アクセス制約
* 禁止事項: 生テーブル名の総当たり、未承認ビュー参照、メタデータにない推測カラムの使用

#### 入力

* セマンティックレイヤー上の指標一覧
* データカタログ上の承認済みデータセット一覧
* 利用者権限に応じたアクセス可能範囲
* 検証用の業務質問セット

#### 実施手順

1. AIへ業務質問と、メタデータAPIから取得した意味付け済み情報のみを渡す。
2. AIに対し、実行前に参照予定の指標、ディメンション、フィルタ条件、出典メタデータを宣言させる。
3. 生成SQLをDry Runし、構文妥当性、参照先、返却件数、権限違反の有無を確認する。
4. 正答クエリまたは既存BI結果と比較し、数値誤差や定義逸脱を評価する。
5. 失敗時は、どのメタデータ不足または解釈ミスが原因かを分類する。

#### 主な確認観点

* SQLが物理名ではなく、意味付け済み概念に基づいて構成されているか。
* 指標定義の解釈違いが発生した場合、出典メタデータを根拠に差分説明できるか。
* 禁止ビューや未承認データセットへのアクセスをAIガバナンス層またはTool層で遮断できるか。
* ローカルファイル検索や既存コード参照なしでも、一定の正答率へ到達できるか。

#### 完了条件

* 代表的な質問セットに対し、期待値と照合可能なSQLを継続的に生成できる。
* 失敗ケースの大半が「モデルの偶発的な誤り」ではなく、「不足メタデータの追加」で改善可能な分類に整理できる。
* 生データ直結の近道を使わずに、セマンティックレイヤー中心の経路で成立する。

---

### 5.2 JC-02: 整備済みメタデータからのプログラム生成

#### 目的

AIがメタデータからSQL以上の処理単位を理解し、抽出、加工、可視化準備、他システム連携を含むプログラムを動的生成できるかを確認する。

#### 想定シナリオ

* ユーザー要求例: 「売上低下カテゴリを抽出し、前年差分と在庫回転率を加味した分析結果をCSVで出力したい」
* 利用コンテキスト: 指標定義、利用可能列、出力形式、CSVの列定義、実行環境の制約
* 禁止事項: 任意パス書き込み、外部ネットワークへの持ち出し、未承認ライブラリの導入

#### 入力

* JC-01で利用したメタデータ一式
* 許可済みランタイムとライブラリ一覧
* 出力ファイル仕様、期待フォーマット
* テストデータと期待出力例

#### 実施手順

1. AIに、必要な中間処理の計画を自然言語または疑似コードで宣言させる。
2. 計画に従ってPython等のコードを生成させる。
3. 隔離環境で静的検査、構文検査、依存関係検査を実施する。
4. Dry Run可能な処理はDry Runし、本実行ではテストデータだけを扱う。
5. 出力内容、処理時間、再実行性、禁止操作の有無を確認する。

#### 主な確認観点

* セマンティックレイヤー上の定義を処理ロジックへ一貫して反映できるか。
* SQL生成よりも複雑な手続き型処理で、想定外の副作用を起こさないか。
* エラー時に、どの処理段階で何が不足していたかを分解できるか。
* 生成コードを将来の再利用候補として保存できる品質か。

#### 完了条件

* 代表ユースケースで、期待する出力ファイルまたは集計結果を再現できる。
* サンドボックスとポリシーにより、危険なファイル操作や外部通信を遮断できる。
* 同種タスクに対して、コード生成結果をテンプレートまたはキャッシュ候補として扱える。

---

### 5.3 JC-03: 第1・第2段階の結果からの自動MCPツール生成

#### 目的

動的生成されたSQLやプログラムを単発の成果物に終わらせず、descriptionと入出力スキーマを付与したMCPツールとして自動カプセル化できるかを確認する。

#### 想定シナリオ

* 生成済みプログラム例: 「地域別売上比較レポート生成」「在庫回転率異常検知」
* 期待する状態: 上記処理がMCPツールとして登録され、他のエージェントから再利用できる
* 禁止事項: 権限要件が曖昧なツール公開、説明不足のブラックボックス化、Write系操作の無審査公開

#### 入力

* JC-01 / JC-02で評価済みのSQLまたはプログラム
* ツールのdescription生成ルール
* 入出力JSON Schemaのテンプレート
* Read / Write分類、Dry Run要件、必要権限の定義

#### 実施手順

1. AIに、生成済みコードの用途、前提条件、制約、想定入力を要約させる。
2. その要約からMCPツールの `name`、`description`、入出力スキーマを生成させる。
3. ツールをエフェメラルMCPサーバーまたは一時登録モジュールとして配備する。
4. ツール一覧取得、スキーマ検証、Read / Write分類、Dry Run実行を確認する。
5. 別のエージェントにそのツールを呼び出させ、descriptionだけで正しく利用できるかを評価する。

#### 主な確認観点

* 生成されたdescriptionが、元コードの責務と制約を過不足なく表現しているか。
* スキーマ不備や権限不足のツールが公開前に弾かれるか。
* 動的生成ツールであっても、既存のMCPベース運用と監査方式に収まるか。
* 単発コードよりも、再利用可能なツール資産として価値が増しているか。

#### 完了条件

* 複数の生成済み成果物を、破綻なくMCPツールとして登録・呼び出しできる。
* descriptionとスキーマだけで、別のAIが正しく利用できる水準に達している。
* Read / Write分離、Dry Run、監査ログ連携を既存ルールのまま適用できる。

---

### 5.4 JC-04: 第3段階の結果からの自動AIエージェント生成

#### 目的

動的に生成・登録されたMCPツール群を前提に、特定タスク向けの専用AIエージェント定義まで自動生成し、タスク完遂までの自律化可能性を確認する。

#### 想定シナリオ

* タスク例: 「営業会議向け週次分析レポートを作成し、根拠付きサマリを返す」
* 期待する状態: メタAIが必要ツールを選び、プロンプト、手順、停止条件を持つ専用エージェントを生成する
* 禁止事項: 停止条件なしの無限再試行、未承認ツールの自動昇格、説明不能な意思決定フロー

#### 入力

* JC-03までで生成済みのMCPツール群
* エージェント生成テンプレート
* 許可された計画長、利用予算、最大反復回数、HITL条件
* ベンチマーク用の業務タスクセット

#### 実施手順

1. メタAIに、与えられたタスクに必要なツール候補を選定させる。
2. 選定結果から、専用エージェントの役割、プロンプト、オーケストレーション手順、停止条件を生成させる。
3. 生成エージェントを隔離環境で実行し、どのツールをどの順に使ったかを記録する。
4. 結果品質だけでなく、不要なツール呼び出し、無駄な再試行、予算逸脱の有無を評価する。
5. 高リスク経路ではHITLへエスカレーションできるかを確認する。

#### 主な確認観点

* 「ツールを生成するAI」と「ツールを使うAI」の責務分離が保たれているか。
* エージェント自動生成後も、既存のApplication層の停止条件や監査方式へ接続できるか。
* 新規タスクに対して、既存ツールの組み合わせで足りない部分だけを追加生成する挙動になっているか。
* 自己進化を許しても、統制不能な自己増殖に陥らないか。

#### 完了条件

* 代表タスク群で、専用エージェントの自動生成から実行完了までを一連で再現できる。
* 予算、停止条件、HITL条件、監査ログを付与した状態で実行できる。
* 人手レビューにより、再利用可能なエージェント定義として保存する価値があると判断できる。

## 6. 段階移行の判断基準

探索的PoCであっても、各段階を無制限に進めるのではなく、以下の判断基準で次段階への進行可否を決める。

| 現段階 | 次段階へ進む条件 | 進めない場合の見直し点 |
| --- | --- | --- |
| JC-01 -> JC-02 | メタデータだけでSQL品質が一定水準に達し、失敗原因が主にメタデータ不足へ整理できる | セマンティックレイヤーの粒度、指標定義、権限制約の記述不足を補う |
| JC-02 -> JC-03 | 生成プログラムが安全に実行でき、再利用候補として保存する価値がある | ランタイム制約、コードレビュー手順、出力仕様を明確化する |
| JC-03 -> JC-04 | 動的生成ツールがMCP標準へ収まり、別エージェントから再利用できる | description品質、Schema妥当性、権限分類を見直す |
| JC-04 -> 将来拡張 | 自動生成エージェントが停止条件と監査条件を守って動作する | 役割分離、予算管理、HITL条件、評価ベンチマークを強化する |

---

## 7. 本検証で特に確認すべきリスク

* メタデータが不十分な状態で「AIが賢ければ補完できる」と見なしてしまい、誤った成功判定を出すリスク
* 生成コードの妥当性だけに注目し、descriptionや権限制約の不備を見逃して危険なツールを公開するリスク
* 単発の成功例をもって「自己進化が実用化できた」と誤認し、再現性や監査可能性を軽視するリスク
* 動的生成の検証であることを理由に、既存のガバナンス原則を一時的に外してしまうリスク

したがって、本検証は「どこまで自動生成できるか」の競争ではなく、「既存の統制原則を壊さずに、どこまで自動生成を取り込めるか」を評価するものとして運用する。
