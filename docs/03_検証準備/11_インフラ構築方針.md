# インフラ構築方針

## 目的

本方針書は、PoC における生成AI基盤インフラをどのような単位で構築し、どのネットワーク境界と共通ミドルウェアを前提に整備するかを整理するための文書である。

本来は個別検証に先立って定義すべき文書だが、本版は `infra/` 配下の実装と `11_技術課題検証` で進めた手順確認結果を踏まえて後付けで整理したものである。

個別ミドルウェアの詳細な検証結果や制約、運用上の補足は `21_検証結果` に集約し、本書では「どういう前提で整備するか」に絞って記載する。

## 対象範囲

本書の対象は、PoC の生成AI基盤を構成する以下のインフラ要素とする。

* 共通ネットワークと外向き Proxy
* 共通 PostgreSQL
* 共通 LiteLLM Proxy
* AIガバナンス層のガードレール実行基盤
* そのほか `infra/` 配下に配置する周辺ミドルウェア群

アプリケーション層の業務ロジック、シナリオ固有の設定、個別ツールの詳細な権限制御は本書の対象外とする。

## 基本方針

### 1. ミドルウェアはディレクトリ単位の Docker Compose で分割構築する

* 各ミドルウェアは `infra/<番号>-<名前>/` 単位で compose を分ける。
* 起動・停止・再作成はミドルウェア単位で実施できるようにし、障害切り分けと再現性を確保する。
* compose をまたぐ依存関係は `depends_on` ではなく、起動順序と検証手順で管理する。

### 2. 共通ネットワークを基準に各サービスを接続する

* 共通内部ネットワークとして `ai_platform_internal` を利用する。
* 外向き通信の出口制御用ネットワークとして `ai_platform_egress` を利用する。
* ミドルウェア間通信は原則として `ai_platform_internal` 上のサービス名解決で行う。

### 3. 外向き HTTP/HTTPS 通信は Squid を経由させる

* 外部向きの HTTP/HTTPS 通信は `infra/00-network` の Squid を経由させる方針とする。
* `ai_platform_internal` を `internal: true` とし、直接外部へ出ないことを基本前提とする。
* 必要なサービスには `HTTP_PROXY` / `HTTPS_PROXY` / `NO_PROXY` を設定し、Docker 内からは `http://squid:3128` を参照する。

### 4. 共通ミドルウェアを先に整備し、その上に後続コンポーネントを載せる

* ネットワーク、Proxy、PostgreSQL、LiteLLM のような横断利用コンポーネントを先に整備する。
* 後続のミドルウェアは、これらの共通基盤を前提として構成する。
* 起動順は「共通ネットワーク → 共通DB → 共通LLMゲートウェイ → ガバナンス/周辺基盤 → 上位アプリ」の順を基本とする。

### 5. 設定値は役割ごとに分離して管理する

* 非秘匿の構成値は compose / `config.yml` / README などに明示し、構成として追跡可能にする。
* APIキーや Secret は `.env` または実環境変数で扱い、YAML へ直書きしない。
* 各ミドルウェアの `.env` は「そのサービスが上流へ接続するための認証情報」を保持するものとして扱う。

## 採用するインフラ構成

### 1. 共通ネットワーク / Proxy

`infra/00-network/` で次を提供する。

* `ai_platform_internal`
* `ai_platform_egress`
* `squid`

この構成により、内部サービスは `ai_platform_internal` 上で相互接続しつつ、外向き通信が必要な場合のみ Squid を経由する前提をとる。

### 2. 共通 PostgreSQL

`infra/01-postgresql/` で共有 PostgreSQL を提供する。

* Langfuse、Dify、LiteLLM、Zitadel など、PoC 内の複数コンポーネントが共通 DB として利用する。
* DB コンテナを各ミドルウェア配下に重複配置せず、共通サービスとして起動する。
* データ永続化はホスト側ディレクトリへの bind mount で扱う。

### 3. 共通 LiteLLM Proxy

`infra/02-litellm/` で LiteLLM Proxy を提供する。

* 外部 LLM プロバイダへの接続は LiteLLM に集約する。
* 上位サービスは OpenAI 互換 API として LiteLLM を利用し、モデル利用経路を統一する。
* API キー隠蔽、接続先集約、コールバック連携の基盤として位置付ける。
* クライアントや上位ミドルウェアから参照する `model` 名は、実モデル名ではなく LiteLLM 上の alias 名を基本とする。
* alias 名は業務役割や環境ごとの差し替えに耐えられる名称とし、実モデルの切替は LiteLLM の設定変更で吸収する。

### 4. AIガバナンス層のガードレール基盤

`infra/03-nemo-guardrails/` で NeMo Guardrails を提供する。

* 生成AIアーキテクチャのガードレール機能としては NeMo Guardrails を利用する方針とする。
* NeMo Guardrails は前段の意味論的ガードレール実行基盤として配置し、上流の推論先には LiteLLM を利用する。
* NeMo Guardrails から LiteLLM を呼び出す際も、実モデル名ではなく LiteLLM の alias 名を参照する構成を基本とする。
* 詳細な採用妥当性、運用制約、代替案との比較は `21_検証結果` 側で整理する。

### 5. 周辺ミドルウェア

PoC では、以下を用途別の周辺ミドルウェアとして整備する。

* `infra/11-redis/`: Event Bus および Langfuse 補助用途の共有 Redis 基盤
* `infra/12-langfuse/`: 観測・トレース基盤
* `infra/21-samba/`: 自律型エージェントのワークスペース用共有ファイルアクセス基盤
* `infra/22-dify/`: WF型アプリ実行基盤
* `infra/23-denodo/`: 外部データ連携・SDK系資材
* `infra/91-zitadel/`: IdP / 認証基盤

各コンポーネントの採用理由や起動順の詳細は、個別手順と検証結果文書で補完する。

## 起動順序の基本方針

PoC の基本起動順は次のとおりとする。

1. `infra/00-network/`
2. `infra/01-postgresql/`
3. `infra/02-litellm/`
4. `infra/03-nemo-guardrails/`
5. Redis / Langfuse / Dify / Zitadel / Samba / Denodo などの周辺基盤

理由は以下のとおりである。

* 後続サービスの多くが共通ネットワークを前提とする。
* PostgreSQL は複数コンポーネントの接続先になる。
* LiteLLM は LLM 利用経路の共通ゲートウェイになる。
* NeMo Guardrails はガードレール実行時に LiteLLM を上流として利用する。

## ネットワークおよび通信方針

### 1. East-West 通信

* ミドルウェア間通信は `ai_platform_internal` 上で行う。
* 接続先は原則としてサービス名で参照し、ホスト公開ポートへの依存を避ける。

### 2. North-South 通信

* ホストから利用者や検証者がアクセスする UI / API は `ports` で必要最小限のみ公開する。
* ホスト公開は「利用のための入口」であり、サービス間接続の主経路には使わない。

### 3. Egress 通信

* 外向き HTTP/HTTPS は Squid を通すことを基本とする。
* `NO_PROXY` には `localhost`、`127.0.0.1`、および Docker 内部サービス名を列挙し、内部通信をプロキシ迂回させる。
* Squid 側の許可先や許可ポートは、必要最小限の allowlist で管理する。

## 設定・認証情報の扱い

### 1. 非秘匿設定

* ポート番号、接続先 URL、LiteLLM 上の alias 名、設定ディレクトリ、ネットワーク名などは compose や `config.yml` に定義する。
* 構成差分は Git で追跡できる状態を維持する。
* 実モデル名やプロバイダ個別設定は LiteLLM の内部設定に閉じ込め、クライアントや Guardrails 設定に直接露出させない。

### 2. 秘匿設定

* API キー、Client Secret、DB 認証情報などは `.env` または環境変数で定義する。
* LiteLLM の `LITELLM_MASTER_KEY`、NeMo Guardrails の `OPENAI_API_KEY` のように、上流サービスへ接続するための認証情報はサービスごとの `.env` に保持する。
* クライアントが利用する認証キーと、サービス間接続で利用する認証キーは同一とは限らないため、役割を分けて管理する。

## 現時点の整備優先順位

検証準備段階では、以下の順で整備・確認を進める。

1. 共通ネットワーク / Proxy
2. 共通 PostgreSQL
3. 共通 LiteLLM Proxy
4. ガードレール基盤
5. 監視・認証・業務向け周辺基盤

この順序により、上位コンポーネントが依存する共通基盤の成立性を先に確認できるようにする。

## 本書と検証結果文書の関係

* 本書は「どう整備するか」という方針を扱う。
* 個別の成立性、異常系、運用上の制約、採用是非の判断材料は `11_技術課題検証` と `21_検証結果` に記載する。
* 特に NeMo Guardrails に関する詳細な評価や制約整理は、本書ではなく `21_検証結果` 側で扱う。
