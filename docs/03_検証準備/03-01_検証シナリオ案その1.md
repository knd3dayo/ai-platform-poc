# AI事業者ガイドライン（第1.2版）に準拠した生成AIガバナンス基盤構築の検証

## 1. 開発テーマ
**AI事業者ガイドライン（第1.2版）に準拠した生成AIガバナンス基盤構築の検証**

## 2. 背景・目的
生成AIの業務適用はシステムと連携して自律的にタスクをこなす「AIエージェント」へと移行しつつある。これに伴い、総務省・経済産業省は2026年3月に「AI事業者ガイドライン 第1.2版」を公表し、意思決定の透明性（トレーサビリティ）や、外部システム操作時の人間による介入（Human-in-the-Loop: HITL）の重要性を新たに明記した。

本研究開発では、アプリケーションロジックとは独立した「生成AIガバナンス層」をインフラ基盤として構築する。AIエージェントの暴走や不適切利用をシステム的に統制し、安全かつ迅速にAI業務適用を進めるための標準アーキテクチャを検証・確立する。

## 3. ガイドライン第1.2版に基づく各主体の責務と本検証のスコープ

| 主体 | ガイドライン（第1.2版）が求める主な責務 | 本プロジェクトでの対応スコープ |
| :--- | :--- | :--- |
| **共通の指針** | トレーサビリティの確保と説明責任 | 内部相関ID（`trace_id`）による、入力からモデル選択、Tool実行、事後評価までの一気通貫の追跡基盤構築。 |
| **AI提供者** | サービスの安全管理、エージェントの行動範囲の制限、システム権限の最小化 | **【インフラ層の主眼】** プロキシによる利用モデル・Tool呼び出しの上限制御。Dockerサンドボックスによるデータアクセス範囲とシステムコマンドの制限。 |
| **AI利用者** | 自律アクションに対するHITL（人間による確認）の実施 | **【アプリ層の疎通確認】** プロキシの遮断シグナルを受け取り、処理を一時停止・再開するHITLハーネスのサンプル実装（LangGraph等を利用）。 |

## 4. 技術的アプローチ：第1.2版の要求に対する実装方式

本検証では、インフラエンジニアの主導領域である「統制・隔離・追跡」の基盤構築に注力し、アプリケーション側のHITL機構は疎通確認用のサンプル実装とする。

* **① 入出力Guardrailsとエージェント間統制（NeMo Guardrails）**
    * プロンプトインジェクションの防御、PIIのマスキング。エージェント間のHandoff時における機密情報のサニタイズ。

* **② Tool利用制御と予算統制（LiteLLM Proxy：検知とインターセプト）**
    * アプリからLLMへの直接通信を禁止。仮想キー単位でのTool利用制限と予算監視を行い、ポリシー抵触時や重要Tool実行時はLLMへのリクエストをインターセプト（例外返却）し、上位のアプリ層へ制御を戻す。

* **③ サンドボックスによる実行権限の最小化（Docker環境）**
    * AIエージェントの実処理（コード実行等）を隔離されたDockerコンテナ内で行い、ホストシステムへの影響を物理的・論理的に遮断する。

* **④ 完全なトレーサビリティと事後評価（Langfuse / Ragas）**
    * `trace_id` を伝搬契約として守り、Langfuseに全ログ（LLM呼出、Tool実行、HITL待機状況）を集約・可視化する。

* **⑤ HITLハーネスのサンプル実装（LangGraph / LangChain等の活用）**
    * インフラ（Proxy）からのインターセプト信号を受け取った際、処理を一時停止（Suspend）し、人間の承認後に再開（Resume）する仕組みはアプリ層の責務となる。
    * 本検証では、インフラ側の統制機構が正しく機能することを証明するため、**LangGraph（またはLangChain DeepAgents等）を用いたサンプル的なHITLハーネス実装**を行い、アーキテクチャ全体の疎通確認をスコープとする。

## 5. システム構成案（アーキテクチャ）
北境界を3段（API Gateway, BFF, AIガバナンス層）で責務分離し、インフラとアプリが協調するアーキテクチャを採用する。

* **BFF / App層（HITLハーネス）**:
    * `trace_id` の確定。LangGraph等を用いたサンプルアプリとして、Proxyからの「HITL要求」時に状態を保持し、再開する基本フローを実装。

* **AIガバナンス層（インフラ中核部）**:
    * **LiteLLM Proxy**: モデル集約、Tool上限制御、予算監視。HITL要件の検知とシグナル発出。
    * **NeMo Guardrails**: 意味論的な入出力検査。
    * **Langfuse + Ragas**: トレース集約と非同期の事後評価。

* **Tool層（サンドボックス環境）**:
    * **Docker等**: Toolの実処理をサンドボックス内で行う隔離環境。

## 6. 今後のPoCロードマップ
* **Phase 1 (Foundation)**: LiteLLM Proxyによる集約と、Langfuse + `trace_id` によるトレーサビリティ基盤の確立。

* **Phase 2 (Guardrails & Sandboxing)**: NeMo Guardrailsの導入、およびDockerを用いたTool実行環境の隔離とアクセス制限の検証。

* **Phase 3 (HITL Integration)**: **LangGraph等を用いたサンプルハーネス**とLiteLLM Proxyを結合し、インフラ主導でのHITL（検知→中断→再開）の技術的な疎通を証明する。
