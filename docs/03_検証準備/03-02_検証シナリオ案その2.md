# セマンティックレイヤー/メタデータを活用した生成AI回答のグラウンディング・エージェントおよびハーネス機構の検証

## 1. 開発テーマ
**セマンティックレイヤー/メタデータを活用した生成AI回答のグラウンディング・エージェントおよびハーネス機構の検証**
～エンタープライズ水準の信頼性を担保する、根拠づけとサイテーション（出典提示）自動化基盤の実現～

## 2. 背景・目的
生成AIを社内業務や顧客向けサービスに適用する際、LLM（大規模言語モデル）特有のハルシネーション（もっともらしい嘘）が重大なビジネスリスクとなる。生成AIの回答が業務上「正しい」と判断されるためには、単に自然な文章であるだけでなく、**「回答の根拠は何か（グラウンディング）」**と**「その情報はどこから来たか（サイテーション/出典提示）」**がシステム的に保証されていなければならない。

本研究開発では、企業のデータ資産に意味付けを行う「セマンティックレイヤー/メタデータ」を活用し、回答生成AIの出力を事実確認に特化した「グラウンディング・エージェント」がチェックし、そのプロセスを強制する「ハーネス機構」の実現可否を検証する。

## 3. 課題と解決アプローチ
生成AIのエンタープライズ適用において、最大のリスクはハルシネーション（幻覚）である。
生成AIは、回答を生成するために本当に必要な「ドメイン固有の情報（社内ルール、製品の独自仕様、機密データなど）」が入力プロンプトに含まれていない場合でも、「わかりません」と答えず、**事前学習された一般的な知識を用いて、もっともらしい嘘（ハルシネーション）を返してしまう**という特性を持っている。

この課題に対し、本検証では**「ドメイン固有知識の宝庫である社内DB（データ層）」**と、そのデータ構造や意味・出所を説明する**「セマンティックレイヤー / メタデータ」**を活用し、以下の技術的アプローチで解決を図る。

* **データ層とセマンティックレイヤーの統合:**
    単なるテキスト検索（従来のRAG）ではなく、社内DBに蓄積された事実データに対して、「これはどの部門のどんなデータか」を示すメタデータと意味的な繋がり（セマンティックレイヤー）を付与し、機械的に解釈・追跡可能な状態にする。
* **「一般的な知識による嘘」と「根拠ある事実」のチェック機構（グラウンディング・エージェント）:**
    生成AIから出力された回答が、「LLMの一般的な知識によって補完された嘘（推測）」なのか、それとも「社内DB等に格納された事実データに裏付けられた根拠ある情報」なのかを、メタデータを照合キーとしてシステム的に突き合わせ、真偽を判定する専用のAIエージェントを構築する。
* **出力統制とオーケストレーション（ハーネス機構）:**
    回答をユーザーへ返す直前に必ず上記のエージェントによるチェックを行わせる「関所（ハーネス）」をインフラ層・アプリ層として実装する。社内データに基づく明確な根拠とサイテーション（出典提示）がない限りユーザーには回答を返さない仕組みを構築する。

## 4. 検証スコープ：グラウンディング・エージェントとハーネスの構築
本検証では、以下の機能を備えたアーキテクチャを構築し、その有効性とパフォーマンス（UXへの影響）を評価する。

1. **メタデータ統合型パイプラインの構築:**
    セマンティックレイヤー/データカタログ機能を備えた製品（Snowflake、Databricks、Denodo、dbtなど）が提供するMCP（Model Context Protocol）を用いて、データ基盤とLLMを標準化された手順で安全に連携させる。これにより、構造化されたメタデータ（データの出所、定義、品質スコアなど）をコンテキスト内に強制的に組み込む。

2. **グラウンディング・エージェントの構築:**
    回答生成モデルとは別に、事実確認と評価に特化した「グラウンディング用AIエージェント」を作成する。このエージェントは以下の要件を検証する。
    * **Faithfulness（忠実性）:** 回答が、提供されたコンテキスト（検索結果）のみから生成されているか。
    * **Citation Accuracy（出典の正確性）:** メタデータとして渡した情報（ファイル名、ページ数等）が正確に引用されているか。

3. **ハーネス機構（強制チェックとフォールバック）の実装:**
    ユーザーに回答を返す際には、**必ずグラウンディング・エージェントを経由させるインターセプト機構（ハーネス）**を実装する。
    検証により「根拠なし」と判定された場合、誤情報を出力せず、「根拠が見つかりませんでした」という定型文への置換、または検索条件を緩和しての自動再試行（リトライ）を行う安全なフローを確立する。
    * **【検証ポイント】:** 同期チェックによるレイテンシ（遅延）増大を防ぐため、グラウンディング・エージェントに用いる軽量かつ高速なJudgeモデル（評価特化型LLM）の選定と最適化も併せて検証する。

## 5. システム構成案（アーキテクチャ）



本システムは、生成処理と検証処理を分離し、それをハーネスが統制するアーキテクチャとする。

* **① Semantic Layer & Data Store (MCP経由):**
    データカタログ / セマンティックレイヤー製品。社内データに「意味」と「出所（メタデータ）」を紐付けて管理し、MCPを介してエージェントにセキュアに提供する。
* **② Generation Agent (回答生成):**
    ユーザーの質問に対し、①から取得したコンテキストとメタデータを元に、回答のドラフトと出典タグ（例：`[Doc1_Page3]`）を生成する。
* **③ Harness Mechanism (統制・ルーティング層):**
    ②の出力を受け取り、そのままユーザーに返さず、強制的に④へルーティングするオーケストレーター。
* **④ Grounding Agent (検証・今回の主眼):**
    高速なJudgeモデルを用いて、②の「ドラフト回答」と「①の元データ/メタデータ」を比較照合する。
    * *検証OKの場合:* 出典リンクをリッチテキスト化してハーネス経由でユーザーへ返却。
    * *検証NGの場合:* ハーネスにNGシグナルを返し、安全な回答へフォールバック（再試行または定型文返却）させる。

## 6. 期待される成果
* **「責任あるAI」基盤の確立と監査対応:**
    ユーザーが「なぜその回答になったのか」を即座に元ドキュメントを辿って確認できるだけでなく、グラウンディング・エージェントの検証結果がログとして残るため、コンプライアンス監査に耐えうる証跡確保が可能になる。
* **ハルシネーションリスクの極小化:**
    インフラ的に回避不可能なハーネス機構を設けることで、企業ブランドや業務プロセスを毀損する誤情報の出力を未然に防ぐ。
* **技術評価指標の獲得:**
    MCPを介したデータアクセス手法の確立、および「生成」と「検証」のマルチエージェント構成に伴う処理遅延（レイテンシ）の実測値を把握し、本番適用に向けたサイジングとアーキテクチャ設計の基礎データとする。
